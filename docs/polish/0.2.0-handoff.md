# Uprava 0.2.0 — portable handoff

Updated: 2026-07-10

Branch: `0.2.0`

Implementation baseline: `2e4565e`

Release state: implementation in progress; not yet `0.2.0-rc.1`

## Current point

The branch is a stable, checked implementation baseline with significant
quality-foundation work already integrated. It is not closeable using only the
four residual notes from the earlier private handoff: several original audit
steps remain partially implemented.

The authoritative remaining route is
[`0.2.0-completion-from-current.md`](../tmp-plans/0.2.0-completion-from-current.md).
It replaces the earlier worker/reviewer/manager execution protocol with one
sequential implementation stream. Small coherent changes and milestone gates
remain required, but subagent handoffs and private execution-unit tracking are
not.

The user owns the final live E2E and product review. Codex must finish the
implementation, automated clean-state checks and immutable RC handoff first.
After the user review, Codex fixes recorded defects and cuts final `0.2.0`.

## What is implemented

- Architecture, compatibility, separate state-slot and CI-lane decisions for
  `0.2.0`.
- Numbered Core SQLite migrations with checksums and incompatible-state guard.
- Core aggregate transaction and legal-transition improvements.
- Durable Core event-publication outbox with retry, replay and idempotency.
- Projection-completion boundary coupled to publication enqueue.
- Durable messages for `ProviderMessageCompleted`, `RuntimeError`,
  `ApprovalRequested` and `ApprovalResolved` coupled to the completion
  transaction and protected by `source_event_id` uniqueness.
- Durable Core command-dispatch outbox with transactional enqueue, backfill,
  at-least-once retry accounting and terminal cleanup.
- Project/Placement convergence improvements and addressed workspace command
  waiters with durable result fallback.
- Argon2id password records and transactional enrollment audit.
- Protocol `v2` coordinated across Core, Node and Web.
- Versioned Node SQLite state with normalized registration, command cache,
  event outbox, runtime metadata, transcript, resume-reference and placement
  sequence tables.
- Node owner-boundary mutation APIs, normalized reopen hydration, bounded
  command cache/outbox and reconnect replay.
- Command outcomes, event batches, ACK state and enrollment transitions
  persisted before external delivery at the completed owner boundaries.
- Persisted Node counters for reconnects, dropped events and heartbeat
  failures.
- Terminal outbound-route rebinding after Core control reconnect.
- SQLite online backup and read-only restore verification, deterministic
  temporary backup/restore rehearsal and guarded rollback rehearsal.
- Static non-root Core/Node/Web image policy checks.
- Deterministic Playwright golden tests isolated from host auth state and a
  passing real Codex smoke path.

The detailed commit and residual ledger is in
[`0.2.0-implementation-ledger.md`](0.2.0-implementation-ledger.md).

## What remains

The ordered completion plan owns the detailed work. Its major open boundaries
are:

1. Finish single-transaction Core projection for every event kind.
2. Add object authorization, legal transition validation, size limits and a
   bounded generation-aware Core connection registry for Node frames.
3. Finish canonical identity routing and durable asynchronous long commands.
4. Make Node SQLite state fully message-owned with complete normalized CRUD,
   retention and joined shutdown.
5. Separate Node transport, dispatch and provider execution with bounded
   priority queues, cancellation and output limits.
6. Complete daemon-owned PTY replay, gap, reconciliation and child-reap
   semantics.
7. Turn protocol `v2` into generated schemas, Web types, validators and drift
   checks; remove shared presentation contracts.
8. Add Web draft ownership/conflict preservation, split the workbench and
   lazy-load Monaco/xterm under a bundle budget.
9. Complete public-ingress quotas, bounded non-blocking logging, metrics,
   correlation and optional OTLP behavior.
10. Finish Core, Node and Web module boundaries and dependency/docs hygiene.
11. Build and execute production images, then pass full clean-instance state,
    backup/restore and forced rollback rehearsals.
12. Assemble an immutable automated RC, hand it to the user for live E2E, fix
    evidence-backed defects and cut final `0.2.0`.

## Accepted constraints and known caveats

- P0-3 remains accepted risk: Codex runs with approvals/sandbox restrictions
  disabled under the trusted controlled-deployment boundary. Do not change or
  claim this posture as fixed without a scope decision.
- There is no in-place `0.1.x` data migration. Never point a `0.1.8` binary at
  `0.2.0` state or overwrite the retained `0.1.8` state/configuration.
- Current package/release metadata remains `0.1.8`; protocol and state slots
  are already the coordinated `0.2.0`/v2 boundary. Version metadata moves to
  `0.2.0-rc.1` only after every implementation milestone passes.
- Event and command delivery are intentionally at-least-once; handlers must
  remain idempotent.
- Node's ephemeral localhost enrollment fixture is sandbox-sensitive and can
  also return HTTP 503 in the full suite. It has passed both full and isolated
  escalated runs, but the fixture remains a quality-gate reliability issue.
- `make c` currently rewrites one formatting-only expression in
  `apps/web/src/app/App.test.tsx`. Restore that known unrelated diff after the
  gate until the underlying formatting baseline is deliberately reconciled.
- Do not interpret static non-root image checks as runtime proof.
- Do not substitute automated golden/smoke tests for the user's final live
  E2E review.

## Resume on another machine

1. Clone/fetch the repository and switch to branch `0.2.0`.
2. Read the documents in the order listed in
   [`docs/polish/README.md`](README.md).
3. Confirm `git status --short --branch` and inspect any user changes before
   editing.
4. Run `make help`, `make init` if dependencies/hooks are not installed, then
   `make l`.
5. Run `make c` to establish the new machine baseline. If Docker is available,
   also establish the production image/runtime gate when that milestone is
   reached.
6. Preserve separate `0.1.8` and `0.2.0` Core/Node state roots. Do not copy
   ignored local databases, credentials or logs from the old machine.
7. Start with Milestone 1 of the completion plan. Work sequentially; add
   focused regression tests, run `make l` at each slice and `make c` at the
   milestone gate.
8. Update this handoff, the implementation ledger and RC checklist at
   meaningful milestone boundaries.

Local service commands and environment variables are documented in the
[development runbook](../en/runbooks/v01-local-dev.md). Routine operations go
through `make`.
