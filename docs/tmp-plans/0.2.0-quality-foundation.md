# Uprava 0.2.0 Quality Foundation

Date: 2026-07-10

Status: active

Implementation branch: 0.2.0

Source baseline: 0.1.8

Target baseline: 0.2.0

## Purpose

This plan turns the 2026-07-09 audit into an ordered implementation program for
the 0.2.0 quality baseline.

The release is not a new product-feature wave. Its purpose is to make the
already deployed MVP a reliable foundation for the next feature-queue items:

- Core must enforce ownership, transactions and legal state transitions;
- Node must have one owner for durable state and long-lived resources;
- control, provider and terminal paths must remain responsive under load and
  reconnects;
- Core, Node and Web must share checked contracts instead of handwritten
  mirrors;
- Web must preserve user work and load heavy workbench surfaces on demand;
- delivery must be reproducible, least-privilege and recoverable;
- the resulting 0.2.0 candidate must be testable from a clean local
  installation through the complete user workflow.

## Fixed Scope Decisions

### Accepted unrestricted Codex posture

Audit finding P0-3 is intentionally not remediated in 0.2.0.

The current Codex adapter continues to launch with approvals and sandboxing
disabled. This is an explicit controlled-deployment decision backed by the
dedicated Unix user, filesystem permissions, Node workspace allow-list and a
server where destructive failure is acceptable.

Consequences:

- do not change Codex launch flags in this plan;
- do not add a partial approval flow and call it hardened;
- continue to name the effective deployment profile controlled development;
- do not claim team, cloud or hostile-workload isolation;
- Step 0 must add a precise feature-queue follow-up for provider-native
  persistent execution policy if the current runtime entries do not yet name
  it explicitly. Its exit criteria are safe-by-default sandboxing, an explicit
  unsafe-mode switch, real approval handling and visible effective policy.

All other provider-process reliability work remains in scope: cancellation,
ownership, bounded output, traceability and separation from transport state.

### Breaking changes are allowed

Compatibility with 0.1.x APIs, Web contracts, Core SQLite data and Node local
state is not a release constraint.

The implementation may replace schemas and identifiers when that produces a
cleaner foundation. It must not silently destroy or reinterpret old state.
Instead it must:

- fail startup with an actionable incompatible-state error;
- document backup, reset and re-enrollment steps;
- make a clean 0.2.0 installation deterministic;
- update Core, Node and Web as one protocol release;
- exercise the reset and reconfiguration path in the final release rehearsal.

An in-place 0.1.8 to 0.2.0 data migration is explicitly out of scope.

The first 0.2.0 run uses separate versioned Core and Node state roots. It must
not overwrite the retained 0.1.8 database, Node JSON state or their matching
configuration. Rollback atomically selects the old binaries, configuration and
state together. Work created only in the new 0.2.0 state is lost after such a
rollback; that limitation must be shown before activation.

Release-candidate builds use unique 0.2.0-rc.N package versions and
Git-SHA-based release ids. The final 0.2.0 version is assigned only after the
last release candidate passes the complete clean-state suite.

### Test cadence

Deep end-to-end testing is intentionally deferred until the bulk
implementation has landed on the 0.2.0 integration branch.

During implementation every slice still receives:

- focused regression tests for the failure modes it changes;
- unit and component tests around new boundaries;
- targeted integration tests when a transaction, socket, process or reconnect
  invariant cannot be proved at unit level;
- make l while iterating;
- make c at slice handoff.

Starting individual services or running a small smoke test is allowed when it
shortens feedback. The full Core, Web, Node, real Codex and manual workflow
matrix runs only in the release-candidate phase, followed by a dedicated polish
loop.

## Architecture Decisions For 0.2.0

These decisions must be promoted into synchronized design documentation before
or with the first behavior-changing commit.

### Project, Placement and Workspace identity

- Project is a Core-owned logical aggregate. Its identity does not depend on a
  Node or local path.
- ProjectPlacement is the physical binding between a Node and one canonical
  local workspace path.
- The database enforces uniqueness of node_id plus canonical_workspace_path.
- Workspace is the user-facing workbench surface over a Placement in 0.2.0,
  not a second persisted entity with another identifier.
- One Project may own Placements on multiple Nodes.
- Node canonicalizes and validates local paths and reports local facts, but
  does not mint Project or Placement identifiers.
- Heartbeat discovery converges on one unbound Placement for the physical
  Node/path pair. Explicit binding attaches that Placement to a selected or
  newly created Project.
- No automatic cross-node Project grouping is inferred from a path. A trusted
  repository identity or an explicit rebind use case can provide that later.
- The canonical Core resource route remains /placements/:id, while the
  canonical user-facing Web surface is /workspaces/:placement_id.
- The duplicate Web /placements/:id route and the second Workspace domain
  identity/reference are removed during the breaking protocol change. The Web
  route is explicitly documented as a view over Placement.

### Persistence authority

- Numbered SQLx migrations with checksums replace inline create-if-missing
  migrations.
- Unknown persisted enum/state values are corruption or compatibility errors,
  never silent fallbacks to an initial state.
- Each duplicated index or snapshot has one documented authority and rebuild
  rule.
- Normalized capability rows are authoritative; a second mutable capabilities
  JSON copy is removed.
- Event envelopes remain immutable evidence, while normalized searchable
  columns and projections are derived within the same transaction.
- Session/runtime links have one authoritative relation instead of reciprocal
  mutable copies.
- Enrollment claim, Project/Placement binding, session creation, turn
  submission and event ingestion each execute as one unit of work.
- In-memory notifications are emitted only after database commit.

### Node state and supervision

- A daemon-level NodeSupervisor owns heartbeat snapshots, registration,
  command deduplication, runtime metadata, outbox state and shutdown.
- A single state-store actor is the only writer to Node durable state.
- The 0.2.0 Node store uses SQLite tables for identity, command cache, outbox,
  runtime metadata, transcripts and resume references instead of rewriting one
  growing JSON document.
- The old Node JSON state is not imported. The reset runbook requires
  re-enrollment.
- Completed commands, stopped runtimes, transcripts and acknowledged outbox
  entries receive explicit count, age and byte retention policies.
- RuntimeSupervisor owns provider processes, cancellation, transcript and
  resume state.
- TerminalSupervisor owns PTY children independently of any one control
  connection.

### Shared contracts

- Rust types in uprava-protocol are the source of truth.
- Rust schema roots generate tracked JSON Schema, TypeScript types and runtime
  validators for Web-facing HTTP, SSE and terminal payloads.
- The default implementation path is Rust schema generation followed by
  generated TypeScript and Ajv validators.
- The generator covers only Web-facing roots; Node control-only contracts do
  not enter the browser bundle.
- Built-in commands and events use tagged typed payloads. Extensibility remains
  an explicit Extension variant rather than arbitrary JSON for known kinds.
- Ingress validation proves scope, target and identifier relationships before
  persistence or dispatch.
- Generated files and canonical fixtures are checked for drift in make c and
  CI.

### Reliable command and stream semantics

- Short bounded reads, writes and diff snapshots may remain synchronous.
- Long workspace run/check operations return an accepted durable command
  resource and expose progress/terminal state through lookup and stream
  updates.
- Synchronous distributed waiters use per-command oneshots with a durable
  result lookup fallback.
- Terminal clients use per-terminal streams with sequence, bounded replay and
  explicit gap/reload frames.
- Bounded queues have an explicit overload policy. No reliable path may
  silently ignore lag.
- Provider and terminal truncation produces typed truncation or gap evidence.

### Projection authority

- Core owns legal domain transitions and remote action capability predicates.
- Web consumes an ActionCapability with allowed state and reason code; it does
  not fabricate domain summaries or independently authorize remote actions.
- Node keeps only the operational state required to own local work.
- Browser incremental projection either proves equivalence with canonical Rust
  fixtures or reloads the durable Core projection after a gap or validation
  error.
- The current request-time ArtifactTree is renamed to a session evidence/trace
  projection and receives deterministic identifiers derived from source
  messages/events. A persisted Artifact aggregate remains future product work.

## Work And Commit Strategy

The existing 0.2.0 branch is the integration branch.

- Land ordered, reviewable commits on 0.2.0.
- Use temporary topic branches only for slices that are truly independent
  after their input contract is stable.
- Do not merge any 0.2.0 topic branch directly to main.
- Do not mix a semantic behavior change with a large mechanical file move.
- Add or update a regression test in the same commit as the behavior fix.
- Keep version metadata at 0.1.8 during bulk implementation.
- Enter unique 0.2.0-rc.N metadata only when all implementation slices are
  integrated and make c passes.
- Record late E2E defects as separate fix commits so their cause and regression
  test remain visible.
- Treat all wire changes before Step 7 as coordinated, unreleased protocol-v2
  work. Core, Node and fixtures change together; Step 7 freezes and generates
  the final v2 contract rather than introducing another compatibility layer.
- Every Step must land as its own series of atomic commits and end with a
  numbered checkpoint. The checkpoint records the Step status, focused
  evidence and a passing make c; it may be the final implementation commit or a
  small plan/evidence commit, but never an empty marker.
- Do not begin the next Step before its predecessor checkpoint passes.
- Synchronize main only at clean Step boundaries, resolve drift there and
  rerun the checkpoint gate instead of postponing all integration drift until
  the release-candidate phase.

Suggested commit subjects below define required review boundaries. A large
Step may use several commits, but must not be silently combined with the next
Step.

### Step 1. Establish the Core schema, domain rules and units of work

Goal: make durable Core state internally consistent before changing concurrent
runtime behavior.

Work:

1. Introduce focused domain, application and persistence seams needed by the
   fixes; do not perform the complete mechanical split yet.
2. Replace inline migrations with a fresh numbered 0.2.0 schema and migration
   runner.
3. Implement the configured versioned Core state slot/path and reject a
   mismatched schema without touching the retained 0.1.8 state.
4. Add database constraints for Placement uniqueness, turn ordering,
   one-to-one session/runtime ownership and other invariants currently held
   only in procedural code.
5. Implement pure legal command transition and domain-state transition rules.
6. Make enrollment claim a transactional compare-and-set.
7. Make Project/Placement binding, session creation and turn submission
   transactional units of work.
8. Allocate turn indexes and session projection cursors inside their
   transactions with atomic update-returning operations, not max-plus-one
   reads.
9. Insert an event, advance the inbox/cursor and apply all runtime, session,
   turn, approval and message projections in one transaction.
10. Publish SSE/in-memory notifications only after successful commit.
11. Treat a duplicate event as accepted only when its committed inbox record
    proves that projections completed; otherwise replay idempotently.
12. Remove or document duplicate persistent representations and reject unknown
    state strings.

Focused verification:

- fault injection after event insert leaves neither event nor partial
  projection;
- replay after injected failure produces exactly one complete projection;
- two concurrent enrollment claims yield one Node;
- session/turn creation cannot leave partial aggregates;
- concurrent turn indexes remain unique and monotonic;
- unknown stored state fails visibly;
- migrations succeed from an empty database and fail clearly on incompatible
  0.1.x state.

Concurrency tests use a file-backed WAL pool with multiple connections so they
exercise real SQLite contention rather than the current single-connection test
path.

Completion gate:

- every acknowledged event has all required projections;
- aggregate creation is atomic;
- persistence authority and rebuild rules are test-backed.

Suggested commits:

- refactor(core): establish domain and persistence seams
- refactor(core): replace inline schema with numbered migrations
- fix(core): make aggregate writes and event projection atomic

Audit coverage: P0-2, P1-5, P1-10, P2-1, P2-9, P2-10 and P3.

### Step 2. Authorize Node frames and stabilize Core connection ownership

Goal: an authenticated Node may affect only its own objects, and an old
connection may never delete a newer channel.

Work:

1. Create an authenticated NodeContext for every accepted control connection.
2. Route all inbound ACK, result, event, terminal and control-error frames
   through one ownership validator before state mutation.
3. Verify command ownership and legal transitions for ACK/results.
4. Verify Node, runtime, session, Placement, command and actor relationships
   for events.
5. Verify terminal ownership through Placement and Node.
6. Reject non-terminal result states and inconsistent envelope identities.
7. Validate typed result echoes against the original command target,
   Placement/path and terminal identity.
8. Add count, string-length, nesting and total-byte limits to control frames
   and event batches.
9. Replace the raw control channel map with a generation-aware
   ConnectionRegistry.
10. Register a connection only after valid Hello, supersede the previous
    generation explicitly and remove an entry only when the closing generation
    still owns it.
11. Re-check active generation before every mutating frame from an old socket.
12. Make Core outbound control queues bounded and observable.
13. Record rejected forgery/transition/limit events without leaking secrets.

Focused verification:

- a Node cannot ACK, complete or mutate another Node command;
- cross-node runtime/session/event forgery is rejected;
- terminal output cannot be injected into another Placement;
- illegal command transitions and conflicting duplicate results are rejected
  without changing state;
- pre-Hello and superseded-generation frames are rejected;
- oversized batches and frames fail predictably;
- overlapping connect/disconnect keeps the newest channel registered;
- queue saturation follows the documented overload path.

Completion gate:

- object authorization is enforced at the transport boundary;
- stale connections cannot remove or mutate active routing state;
- control memory use is bounded.

Suggested commits:

- fix(core): authorize inbound node control frames
- fix(core): make control connections generation aware and bounded

Audit coverage: P0-1, P1-2 and the Core half of P1-4.

### Step 3. Normalize identity and replace lossy request/response broadcasts

Goal: one physical workspace has one Placement, and reliable operations have
addressed delivery paths.

Work:

1. Implement Project and Placement application services using the identity
   model fixed above.
2. Make heartbeat discovery and explicit binding converge on the same
   Node/canonical-path Placement.
3. Allow one Project to bind Placements on multiple Nodes.
4. Remove random Project creation from Placement validation.
5. Return conflicts instead of creating duplicate physical Placements.
6. Preserve explicit delete/rebind semantics without letting heartbeat
   resurrect a removed binding unexpectedly.
7. Remove the duplicate Workspace API/reference identity and the duplicate Web
   Placement route; keep /placements/:id as the Core resource and
   /workspaces/:placement_id as the one workbench route.
8. Replace the global command-result broadcast waiter with per-command
   oneshots plus durable result lookup on timeout/registration races.
9. Register the waiter before durable enqueue and clean it through a guard on
   timeout/cancellation.
10. Define synchronous operations versus accepted asynchronous command
    resources; move long run/check operations to the async path.
11. Persist async command progress, terminal outcome, typed result and
    cancellation/expiry state.
12. Add a bounded TerminalHub keyed by terminal id. Sequence and replay
    semantics are completed in Step 6.

Focused verification:

- heartbeat then explicit bind produces one Placement and stable id;
- explicit bind then heartbeat does not duplicate it;
- canonical aliases converge;
- one Project owns Placements on two Nodes;
- simultaneous binding of one Node/path yields one winner;
- delete tombstone and explicit rebind follow the documented lifecycle;
- synchronous result lookup survives waiter lag;
- async workspace command returns immediately and later exposes its durable
  result;
- unrelated result floods do not affect another waiter;
- dropped waiter registry entries do not leak.

Completion gate:

- identity creation is deterministic and database-enforced;
- no reliable command path depends on a lossy global broadcast;
- long workspace work no longer holds an HTTP request for its full lifetime.

Suggested commits:

- refactor(core): normalize project placement identity
- feat(workspace): introduce durable asynchronous command resources
- fix(core): replace global result broadcasts with addressed waiters

Audit coverage: P1-9, P1-10, P2-10 and part of P2-12.

### Step 4. Introduce the single-owner Node state store and supervisor

Goal: remove competing in-memory state copies and make Node durability
transactional, bounded and independently testable.

Work:

1. Add the daemon-level NodeSupervisor and state-store message API.
2. Replace the JSON document with a private-permission Node SQLite store.
3. Implement the configured versioned Node state slot/path without modifying
   the retained 0.1.8 JSON state.
4. Separate tables/APIs for registration identity, command cache, outbox,
   runtime metadata, transcripts and provider resume references.
5. Route heartbeat snapshots, control updates, deduplication and registration
   clearing through the one owner.
6. Persist command ACK/result payloads and outbox sequence updates
   transactionally before external delivery where durability is required.
7. Make stale control generations unable to restore cleared credentials.
8. Move database/file blocking work off Tokio worker threads where needed.
9. Add bounded batching/coalescing without weakening durable-before-send
   guarantees.
10. Add configurable retention and compaction for completed commands, stopped
   runtimes, transcripts and acknowledged outbox entries.
11. Establish supervisor registry/cancellation shells for later runtime and
    terminal owners. At this Step, NodeSupervisor cancels and joins the control
    and state-store tasks it actually owns before clearing registration.
12. Remove state-file polling as inter-task communication.

Focused verification:

- concurrent heartbeat/control updates remain monotonic;
- delayed old revisions cannot overwrite new state;
- auth rejection cannot resurrect old registration;
- restart replays the same command result without repeating side effects;
- outbox sequence never rolls back;
- retention stays inside configured count/byte bounds;
- slow persistence does not block unrelated Tokio workers;
- abrupt shutdown leaves a readable transactional store;
- incompatible old JSON state produces re-enrollment guidance rather than
  implicit import.

Completion gate:

- Node has one authoritative durable-state writer;
- lifecycle supervisors own resources and persist only through that writer;
- runtime/outbox/credential updates cannot overwrite each other;
- stored state growth has explicit bounds.

Suggested commits:

- refactor(node): add supervisor-owned sqlite state
- fix(node): make command and outbox durability transactional
- feat(node): compact retained operational state

Audit coverage: P1-8, P2-3 and the state ownership portion of P2-11.

### Step 5. Decouple transport, execution and provider output

Goal: long-running work must not block the Node control plane or exhaust
memory.

Work:

1. Split the control socket into transport-only reader and writer tasks.
2. Introduce bounded priority-aware queues so ping/pong, ACK and cancellation
   traffic cannot be starved by provider output.
3. Route commands through a bounded dispatcher owned by NodeSupervisor.
4. Return an explicit retryable busy/NACK outcome when the dispatcher is full;
   never silently drop or block the socket reader.
5. Serialize operations per runtime or Placement where order matters.
6. Permit bounded concurrency between independent runtimes/workspaces.
7. Keep active provider work in a tracked task so the runtime worker can still
   select interrupt/stop/cancellation.
8. Track every worker with a cancellation token and join handle.
9. Give RuntimeSupervisor a narrow ProviderExecutionContext and daemon-owned
   typed outcome/event sink.
10. Keep the Codex adapter concrete; do not introduce a speculative provider
    trait hierarchy.
11. Separate provider process success from control-channel delivery success.
12. Bound line length, raw JSON, stdout/stderr bytes, activity count,
    transcript bytes and live outbox retention while the process is running.
13. Parse resume/approval/session metadata incrementally before raw-output
    truncation can discard it.
14. Batch events and propagate backpressure.
15. Emit typed truncation/gap events with dropped counts when evidence is
    capped.
16. Keep the current unrestricted Codex flags unchanged.

Focused verification:

- a slow Codex turn does not delay ping/pong, event ACK or terminal input;
- independent runtimes make progress within the configured concurrency limit;
- same-runtime commands preserve order;
- interrupt/stop cancels and joins the active child;
- a full dispatcher returns the documented retryable outcome;
- a disconnected control socket does not turn local provider success into
  provider failure;
- an oversized single line, infinite stderr and activity stream remain inside
  memory/state bounds;
- outbox limits apply during the run, not only after exit;
- overflow creates observable truncation/gap evidence.

Completion gate:

- control liveness is independent of command duration;
- all execution and output queues are bounded;
- process and task ownership is explicit.

Suggested commits:

- refactor(node): separate control transport from execution
- refactor(node): move provider lifecycle into runtime supervisor
- fix(node): bound provider streams queues and outbox

Audit coverage: P1-1, P1-4, P2-3 and P2-11.

### Step 6. Make PTY lifetime and reconnect semantics daemon-owned

Goal: a temporary Core connection must not own a long-lived local terminal.

Work:

1. Move WorkspaceTerminalManager into daemon-level TerminalSupervisor.
2. Connect the supervisor to the current outbound router rather than one
   captured WebSocket sender.
3. Define terminal states for open, attached, detached, closing, exited,
   failed and lost-after-daemon-restart.
4. Keep bounded sequenced replay per terminal, capped by both frame count and
   bytes.
5. Reattach after Core/control reconnect from the last acknowledged sequence.
6. Emit an explicit gap/reload state when replay is unavailable.
7. Reconcile Node terminal snapshots with Core projections after reconnect or
   Core restart.
8. On close, cancellation or daemon shutdown, terminate the process group and
   reap the child within a bounded timeout, escalating if necessary.
9. Make Web client disconnect detach rather than accidentally destroy the
   Node terminal unless explicit close policy says otherwise.
10. Complete per-terminal Core routing without a global lossy broadcast.

Focused verification:

- a terminal remains usable across overlapping control reconnects;
- the old Core connection cannot remove the new terminal route;
- a Node snapshot restores Core terminal routing after Core restart;
- output resumes from the correct sequence or produces a visible gap;
- Node restart reconciles old Core terminals to lost/closed;
- explicit close and daemon shutdown leave no shell/process-group child;
- terminal flood cannot starve control traffic;
- in-flight independent commands continue during terminal reconnect.

Completion gate:

- reconnect does not orphan an inaccessible PTY;
- every terminal child has one supervisor, cancellation path and reap proof;
- output loss is explicit.

Suggested commits:

- refactor(terminal): move pty ownership to daemon supervisor
- fix(terminal): add reconnect replay gaps and reconciliation
- test(terminal): cover child cleanup and overlapping reconnect

Audit coverage: P1-2, P1-3 and the terminal half of P1-9.

### Step 13. Polish from end-to-end evidence and cut the release

Goal: use the integrated system rather than the audit alone to finish 0.2.0.

For each defect cluster:

1. Record reproduction, expected behavior and affected invariant.
2. Add the narrowest regression test that reproduces it.
3. Fix it in a separate commit.
4. Run the affected focused checks.
5. Increment rc.N before producing the next candidate artifact.
6. Repeat the relevant E2E segment.

After the defect queue is empty:

1. Repeat the complete release-candidate suite from freshly reset state using
   the final 0.2.0-rc.N.
2. Bump package/version API metadata to final 0.2.0 and build one immutable
   Git-SHA-addressed final candidate.
3. Repeat the complete clean-state suite against that final build. If it
   fails, discard the unshipped final artifact, return to the next rc.N, fix
   the defect and repeat; never publish two different 0.2.0 artifacts.
4. Run make c one final time.
5. Update README and synchronized EN/RU versioning, releases, feature queue,
   architecture, runbooks, deployment and observability documents.
6. Mark audit findings implemented, accepted risk or deferred with an explicit
   reason. P0-3 is accepted risk/deferred, not implemented or fixed.
7. Mark feature-queue audit follow-up work complete. Mark daily-use hardening
   complete only if its remaining visual/operational criteria were actually
   delivered; otherwise retain a clearly narrowed queue item.
8. Add the final 0.2.0 release-ledger row as current.
9. Set this plan to implemented and archive/remove it after durable decisions
   are present in canonical docs.
10. Review the full diff and commit history, then merge 0.2.0 into main.

Suggested commits:

- fix(rc): one commit per verified defect cluster
- docs(0.2.0): synchronize the quality baseline
- chore(release): cut 0.2.0

Final release gate:

- a clean clone can build and start 0.2.0 using the documented path;
- the full local E2E workflow passes for the final RC and again for the single
  immutable final 0.2.0 build;
- CI is green;

## Explicit Non-Goals

- changing the unrestricted Codex launch posture;
- production multi-user RBAC or hostile-node isolation beyond authenticated
  object ownership;

- a persisted visual Artifact aggregate;
- task-based sandbox runtime;
- a broad visual redesign unrelated to the audited reliability/performance
  findings;
- in-place import of 0.1.x Core or Node state;
- indefinite compatibility aliases for protocol v1 routes/types.

## Overall Completion Criteria

0.2.0 is complete only when:

- all findings in the coverage matrix are implemented or carry the explicit
  accepted-risk disposition;
- Core transactions and ownership tests prove the critical invariants;
- Node control, provider and terminal paths remain responsive and bounded;
- reconnect cannot select stale routing state or orphan PTY children;
- generated contracts and runtime validation prevent silent Rust/Web drift;
- user drafts survive navigation, refetch and conflicts;
- the initial Web graph excludes Monaco/xterm and respects its budget;
- local auth, public ingress, logs and metrics have bounded operational cost;
- images, CI, backup, restore and rollback pass their release checks;
- the complete clean-state E2E and polish loop succeeds;
- version metadata and synchronized documentation describe 0.2.0 accurately.
