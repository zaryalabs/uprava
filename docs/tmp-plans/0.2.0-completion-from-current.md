## Purpose

This is the execution plan from the current implementation to the complete
`0.2.0` release. It does not replace the audit findings, architecture
decisions, non-goals or release criteria in the quality-foundation plan. It
reorders only the work that is still open and removes the previous
worker/reviewer/manager orchestration overhead.

The implementation runs sequentially in one Codex task on the `0.2.0` branch.
The user owns the final live end-to-end review. Codex prepares a complete,
automatically verified release candidate, hands it off, then resumes only for
defects found by that review and the final release cut.

## Fixed scope and working rules

- Preserve every fixed scope decision and explicit non-goal from the
  quality-foundation plan.
- Keep the unrestricted Codex execution posture as the explicit accepted risk
  P0-3. Do not silently represent it as fixed.
- Do not add in-place `0.1.x` state migration. Keep `0.1.8` and `0.2.0` state,
  configuration and rollback targets isolated.
- Implement in one continuous stream. Do not require subagents, independent
  reviewers, execution-unit journals or manager handoffs.
- Keep changes in small coherent slices even though the work is sequential.
  Each slice must be understandable and revertible on its own.
- Preserve unrelated user changes. Do not use destructive Git operations.
- Use focused tests while iterating, `make l` at each slice boundary and
  `make c` at every milestone gate below.
- Update tests in the same slice as behavior. Update synchronized EN/RU docs
  whenever a durable behavior, contract or operational procedure changes.
- Do not enter `0.2.0-rc.1` while any implementation milestone is open.
- Do not run or claim the final user-owned live E2E. Deterministic automated
  E2E, smoke, clean-state and failure-injection checks remain implementation
  gates and are not substitutes for the user's review.

### Milestone 1 — Finish Core event atomicity

Status: completed on 2026-07-10 (uncommitted milestone baseline after

4. Keep in-memory SSE publication strictly after commit.

6. Remove obsolete standalone write helpers after all callers move to the

Verification:

- injected failure at each projection stage leaves no partial durable state;
- duplicate and restart replay produces exactly one projection/message;

Evidence:

### Milestone 5 — Separate control transport from runtime execution

Status: completed on 2026-07-10 (uncommitted milestone baseline after
`2e4565e`).

Goal: slow or noisy provider work cannot block control traffic or exhaust

5. Introduce `RuntimeSupervisor` with cancellation tokens and joined child

6. Separate provider-process outcome from delivery outcome and keep successful
   local results durable across disconnection.
7. Bound line length, stdout/stderr, raw JSON, activity count, transcript and
   live outbox bytes while execution is active.
8. Parse resume, approval and session metadata incrementally before evidence
   truncation.
9. Emit typed truncation/gap evidence with dropped counts and batch provider
   activity under backpressure.

Verification:

- slow/flooding fake provider with concurrent heartbeat, ACK and terminal
  input;
- same-runtime ordering and independent-runtime concurrency tests;
- dispatcher saturation/NACK test;
- interrupt, stop and shutdown child-reap tests;
- disconnect-after-local-success replay test;
- memory/output/outbox bound tests;
- Node/protocol tests, `make l`, then `make c`.

Gate: P1-1 and P1-4 are closed; every provider task and child has one owner and
bounded resources.

Evidence:

- control websocket writes now go through a bounded writer queue; saturated
  writes fail the connection instead of blocking the reader indefinitely;

## User-owned live E2E gate

The user performs the manual workflow and end-to-end review from the original
quality-foundation Step 12. Codex does not replace it with mocks, golden tests
or a second automated smoke run.

The result is one of:

- **pass** — proceed to Milestone 13;
- **defects found** — record reproduction and expected behavior, then fix each
  coherent defect cluster sequentially with a regression test, focused checks
  and a new immutable `rc.N`;
- **scope decision required** — stop and ask the user before changing accepted
  risks, non-goals or architecture.

## Milestone 13 — Polish evidence and cut final 0.2.0

Goal: turn the user-approved final RC into the one immutable final release.

Work:

1. Close every user-reported defect cluster with a regression test and repeat
   the affected E2E segment through the user-owned gate.
2. Repeat the complete automated clean-state suite on the final RC.
3. Bump all package/version API metadata to final `0.2.0` and build one
   Git-SHA-addressed final artifact.
4. Repeat the complete automated clean-state suite against that final build.
   If it fails, discard it, return to a new `rc.N` and never publish two
   different `0.2.0` artifacts.
5. Synchronize README and EN/RU releases, versioning, feature queue,
   architecture, runbooks, deployment and observability documents.
6. Record every audit finding's final disposition and keep P0-3 visible as
   accepted risk/deferred.
7. Mark the quality-foundation and this completion plan implemented, update
   the temporary-plan index and archive/remove superseded temporary evidence
   after durable decisions are in canonical docs.
8. Run final `make c`, review the full diff/history and leave the branch clean
   and mergeable to `main`.

Gate: the final release criteria from `0.2.0-quality-foundation.md` are all
satisfied and the current release ledger is `0.2.0`.

## Overall completion criteria

This plan is complete only when:

- all original finding dispositions and quality-foundation completion criteria
  are satisfied;
- no partial Core projection or unauthorized/stale Node control mutation is
  possible under covered failure scenarios;
- Node state, provider work and PTYs each have one explicit owner, bounded
  queues/storage and joined shutdown;
- protocol v2 Web boundaries are generated and runtime-validated;
- drafts survive navigation/refetch/conflict and heavy workbench runtimes are
  absent from the initial Web graph;
- public ingress, logs, telemetry, images, backup, restore and rollback meet
  their stated bounded/reproducible gates;
- all automated checks pass from clean state and the user's separate live E2E
  has passed;
- final version metadata and synchronized documentation describe the shipped
  `0.2.0` accurately.
