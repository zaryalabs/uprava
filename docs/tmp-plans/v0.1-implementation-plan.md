## Назначение

Этот документ превращает уже спроектированный продуктовый срез V01 в порядок
реализации для версии `0.1`.

`V01` - это scope продукта: **Distributed Agent Control Panel**. `0.1` - это
первая implementation release, в которой этот scope работает end-to-end в
trusted local/single-user или controlled development deployment.

План временный. Когда в ходе реализации появляются долговечные решения, их нужно
переносить в канонические документы. Этот файл не должен становиться постоянным
источником архитектуры.

## Release Contract 0.1

Версия `0.1` готова, когда пользователь может:

1. Запустить Core Backend и Web Control Panel локально.
2. Запустить как минимум один Node Daemon и enroll его в Core.
3. Увидеть в Web Control Panel node status, heartbeat age, capabilities,
   degraded state and active sessions.
4. Зарегистрировать или открыть project/workspace binding на выбранной node.
5. Запустить Codex-backed persistent interactive session для этого workspace.
6. Отправлять chat turns через Core-routed commands.
7. Получать ordered provider/runtime events в session view.
8. Detach и reattach к той же session.
9. Interrupt, stop and resume там, где это поддерживает provider adapter.
10. Перезагрузить UI и восстановить nodes, workspace bindings, sessions,
    messages, runtime state and recent event history из Core persistence.
11. Видеть stale/offline/error/resource warning states как UI state, а не как
    текст, спрятанный в chat.
12. Видеть явное trusted/local/non-production warning там, где UI открывает
    node pairing или remote control.

## Явно не входит в 0.1

В `0.1` не реализуем:

- production security hardening;
- multi-user auth, team RBAC, secrets management, hardened credentials;
- полноценный Tool Registry или Plugin Registry;
- MCP integration platform;
- Project Workspace Inspector, file tree, editor, terminal, diff/check surface;
- task-based sandbox runs или durable workflow engine;
- multi-agent orchestration;
- cloud node pool;
- full provider-neutral parity beyond the Codex-first adapter;
- Next.js как обязательный app runtime;
- Tauri-specific domain logic.

## Рабочие implementation decisions

Эти решения достаточны, чтобы начать реализацию. Если реализация покажет, что
решение нужно поменять, обновляем этот план и переносим итоговую позицию в
канонические документы.

| Область | Решение для 0.1 |
| --- | --- |
| Backend | Rust, Tokio, Axum |
| Node Daemon | Rust, Tokio |
| Persistence | SQLite first |
| Web app | React 19, TypeScript, Vite SPA |
| Styling | Tailwind CSS v4, shadcn/ui conventions, lucide-react |
| Client state | TanStack Query plus local and URL state |
| UI snapshots | HTTP JSON APIs |
| Browser live events | SSE для session/event streams, если WebSocket не окажется явно проще |
| Core-to-Node control | outbound WebSocket/control channel from Node to Core |
| Core-to-Node presence | HTTP heartbeat from Node to Core, default около 5 секунд |
| First provider | Codex через минимальный Provider Adapter boundary |
| First deployment profile | trusted local/single-user or controlled development |
| Durable truth | Core persistence and event log, not browser state or provider process memory |

## Critical Open Decisions

Эти решения нужно закрыть до соответствующей стадии реализации.

| Решение | Нужно до | Default, если не закрыто |
| --- | --- | --- |
| SQLite access layer: SQLx, rusqlite или другой crate | Core persistence stage | SQLx, если compile-time workflow не мешает; иначе rusqlite для быстрого bootstrap |
| Codex runtime protocol: live process IO, provider resume command или adapter wrapper | Codex adapter stage | сначала fake provider; real Codex adapter остается единственным provider-specific module |
| UI live transport: SSE или WebSocket | Web session stream stage | SSE для browser events, WebSocket только для Node control |
| Workspace registration: explicit path или root scanning | Node workspace stage | explicit registration first |
| Same workspace active runtime: warning или hard block | coordination preflight stage | warning with explicit override in `0.1` |

## Architecture Slices

### Core Backend

Core - client-facing control plane.

Core владеет:

- node enrollment and registry;
- heartbeat projection and node status;
- project and workspace placement metadata;
- session threads, turns, messages, runtime projections;
- command intents and idempotency records;
- ordered event log and per-session projections;
- warning acknowledgements;
- API and live event streams for Web Control Panel;
- static delivery of the built web app, если это практично для `0.1`.

Core не должен:

- напрямую выполнять local files, shell или provider processes;
- требовать от browser прямого подключения к каждой node;
- считать provider process state долговечной truth.

### Node Daemon

Node Daemon - data plane на конкретной машине.

Node владеет:

- local daemon config and development credential;
- outbound registration, heartbeat and control channel;
- workspace path validation;
- minimal capability and health snapshot;
- minimal git/resource snapshot;
- provider process lifecycle;
- command execution and deduplication by `command_id`;
- provider event normalization into Cortex events;
- runtime lifetime policy and local process handles.

Node не должен:

- владеть global session history;
- обходить Core для client-visible commands;
- прятать важные provider actions только в raw terminal output.

### Web Control Panel

Web Control Panel - первый client. Он должен ощущаться как control panel, а не
как full IDE.

UI владеет:

- app shell and `Nodes -> Projects/Workspaces -> Sessions` tree;
- node pairing and diagnostics surfaces;
- project/workspace binding flow;
- session/chat surface;
- message, runtime event, status and approval rendering;
- lifecycle controls: start, attach, detach, interrupt, stop, resume;
- visible warning badges and trusted deployment messaging.

UI не должен:

- подключаться напрямую к Node Daemons;
- изобретать durable state, которого нет в Core;
- показывать future workspace/editor/terminal surfaces так, будто они уже
  входят в `0.1`.

## Data Model Baseline

Первая схема должна быть небольшой, но явной. Имена могут поменяться во время
имплементации, но границы ответственности должны сохраниться.

### Core Tables

- `nodes`: `node_id`, display name, status, enrollment state, last heartbeat,
  version, created/updated timestamps.
- `node_capabilities`: node id, capability key, version/value JSON.
- `node_enrollments`: short-lived pairing request, pairing code hash, status,
  expiry, claimed node id.
- `projects`: тонкая logical project identity, display name, optional repo id.
- `project_placements`: node-local workspace binding candidate with node id,
  project id, workspace path, git snapshot, last validation.
- `session_threads`: durable user work thread, project placement, run mode,
  runtime strategy, work contract, status, provider resume reference.
- `runtime_sessions`: live-or-recoverable runtime projection, provider,
  lifecycle state, timestamps, last runtime step, exit reason.
- `turns`: user turn sequence, status, linked messages and blocked request.
- `messages`: user, assistant, system, runtime and approval-visible messages.
- `commands`: command id, target node/runtime/session, command kind, status,
  dedupe key, payload, created/completed timestamps.
- `events`: event id, command id, runtime session id, session thread id, seq,
  kind, payload, happened timestamp.
- `approvals`: approval or user-input request, status, payload, response.
- `warning_acknowledgements`: actor, warning kind, affected resources, command
  id, timestamp.

### Node Local State

- daemon installation id and config;
- Core URL and development node credential;
- registered workspace roots or explicit workspace paths;
- provider process handles and local runtime metadata;
- recent command dedupe cache, persisted where needed for reconnect safety;
- provider-native session id or resume cursor, если доступен.

## Event And Command Contract

### Command Envelope

Каждая Core-to-Node command должна включать:

```text
command_id
kind
target_node_id
session_thread_id optional
runtime_session_id optional
project_placement_id optional
issued_at
payload
```

Initial command kinds:

- `StartRuntime`
- `ResumeRuntime`
- `SendTurn`
- `ResolveApproval`
- `InterruptRuntime`
- `StopRuntime`
- `ValidateWorkspace`
- `RefreshResourceSnapshot`

Правила:

- Core записывает command intent до dispatch.
- Node deduplicates by `command_id`.
- Commands never hidden inside heartbeat payloads.
- Если control channel недоступен, Core записывает pending dispatch и просит
  Node открыть control channel через heartbeat response.

### Event Envelope

Каждый Node-to-Core runtime event должен включать:

```text
event_id
command_id optional
runtime_session_id
session_thread_id
seq
kind
happened_at
payload
```

Initial event kinds:

- `runtime.starting`
- `runtime.ready`
- `runtime.running`
- `runtime.blocked`
- `runtime.expired`
- `runtime.resuming`
- `runtime.stopped`
- `runtime.error`
- `turn.started`
- `turn.completed`
- `turn.interrupted`
- `provider.output.delta`
- `provider.message.completed`
- `approval.requested`
- `approval.resolved`
- `coordination.warning_acknowledged`
- `workspace.validated`
- `resource.snapshot.updated`

Правила:

- `seq` monotonic per `runtime_session_id`.
- Core deduplicates by `event_id`.
- Core может detect gaps by `seq`.
- UI projection строится per `session_thread_id`.
- Global ordering across all nodes не нужен для `0.1`.

## Milestones

### M0. Repo And Tooling Scaffold

Цель: превратить documentation-only repository в buildable multi-app workspace,
пока еще без продуктового поведения.

Tasks:

- Создать Rust workspace scaffold.
- Добавить initial crates для shared protocol/domain, Core server, Node daemon
  и client helpers, если они нужны.
- Создать `apps/web` with Vite, React, TypeScript, Tailwind CSS v4 and baseline
  shadcn/ui conventions.
- Обновить `Makefile` только там, где нужно запускать реальные checks при
  наличии manifests.
- Добавить formatting, lint and test defaults для Rust and web.
- Добавить минимальные README run instructions для local development.

Acceptance:

- `make help` показывает runnable targets.
- `make c` проходит в fresh checkout после установки dependencies.
- Rust and web scaffolds build without placeholder failures.

### M1. Protocol And Domain Foundation

Цель: определить общий язык до того, как Core, Node and UI разъедутся.

Tasks:

- Определить typed ids для node, project, placement, session thread, runtime
  session, turn, message, command and event.
- Определить lifecycle enums для node, workspace placement, session thread,
  runtime session, turn, command, approval and warning severity.
- Определить command and event envelopes.
- Определить API DTOs для snapshots: nodes, inventory tree, project placements,
  session thread, session messages, runtime state, event stream resume cursor.
- Добавить serialization tests для representative commands/events.
- Добавить conversion boundaries between persistence rows and API/domain types.

Acceptance:

- Core and Node depend on the same protocol/domain crate.
- Command/event examples round-trip through JSON.
- Domain types encode `V01` invariants instead of a generic chat transcript.

### M2. Core Server Skeleton And Persistence

Цель: сделать Core durable local control plane.

Tasks:

- Реализовать Axum server with config loading, tracing, health endpoint and API
  version endpoint.
- Добавить SQLite migrations для baseline tables.
- Добавить repository layer для nodes, enrollments, placements, sessions,
  commands, events, messages and approvals.
- Реализовать event append with dedupe by `event_id`.
- Реализовать per-runtime sequence validation and gap marking.
- Реализовать status projections from persisted state.
- Добавить local development profile and visible API flag for non-production
  mode.

Acceptance:

- Core стартует с пустой SQLite database и применяет migrations.
- Core restart сохраняет nodes, placements, sessions, messages, commands and
  events.
- Unit/integration tests покрывают event append, command recording and
  projection reconstruction.

### M3. Node Enrollment And Heartbeat

Цель: сделать Node видимой в Core без inbound access to Node.

Tasks:

- Реализовать Node daemon CLI/config with Core URL, display name, workspace roots
  or explicit paths and local config file.
- Реализовать development enrollment request and pairing code flow.
- Реализовать Core endpoints для creating, approving, revoking and listing node
  enrollments.
- Хранить node credential locally on Node and credential metadata in Core.
- Реализовать heartbeat from Node to Core with health, version, capabilities and
  last known workspace summary.
- Реализовать stale/offline projection based on missed heartbeat.
- Добавить basic node diagnostics endpoint and UI-ready snapshot.

Acceptance:

- User can start Core, start Node, approve pairing and see the node online.
- Stopping Node moves it through stale/offline states without deleting it.
- Revoked node cannot continue heartbeat/control operations.
- Heartbeat remains presence/config/channel negotiation, not command transport.

### M4. Workspace Placement And Resource Signals

Цель: дать пользователю bind work к concrete node-local workspace.

Tasks:

- Реализовать explicit workspace registration or validation request.
- Добавить `Project` and `ProjectPlacement` APIs.
- Node validates path existence, writability and allowed root boundary.
- Node gathers minimal git snapshot: repo URL/id, branch, commit, dirty, ahead,
  behind, untracked count where available.
- Core computes resource badges: offline, stale, missing workspace, read-only
  workspace, dirty, same branch active, same workspace active, behind remote,
  missing capability.
- Реализовать warning acknowledgement event.

Acceptance:

- UI/API can show `Nodes -> Projects/Workspaces`.
- Starting work on missing or unwritable workspace hard-blocks.
- Dirty or same-branch-active workspace warns and allows explicit override.
- Warning override is persisted as an event.

### M5. Core-To-Node Control Channel

Цель: перейти от presence к routed commands and ordered events.

Tasks:

- Реализовать outbound WebSocket/control channel opened by Node.
- Добавить heartbeat response flag for `open_control_channel`.
- Реализовать Core command dispatch over active channel.
- Реализовать pending dispatch behavior when channel is temporarily closed.
- Реализовать Node command deduplication by `command_id`.
- Реализовать Node-to-Core event streaming over the control channel.
- Persist all command intents and event results in Core.
- Добавить tests for duplicate command handling and reconnect behavior.

Acceptance:

- Core can request a channel through heartbeat and dispatch a command when Node
  connects.
- Duplicate command delivery does not duplicate turns or runtime actions.
- Ordered events update Core projection and browser stream.
- Historical session state remains readable when Node disconnects.

### M6. Runtime Lifecycle With Fake Provider

Цель: доказать full lifecycle до того, как product correctness станет зависеть
от неопределенности Codex adapter.

Tasks:

- Определить minimal Provider Adapter trait для launch, resume, send turn,
  approval response, interrupt, stop and event stream.
- Реализовать fake provider adapter that streams deterministic output,
  approvals, errors and completion events.
- Реализовать `StartRuntime`, `SendTurn`, `ResolveApproval`,
  `InterruptRuntime`, `StopRuntime` and `ResumeRuntime` through fake adapter.
- Реализовать runtime states: `starting`, `ready`, `running`, `blocked`,
  `expired`, `resuming`, `stopped`, `error`, `stale`.
- Реализовать `last_runtime_step_at` and 24h no-steps expiry behavior with a
  configurable test interval.
- Persist provider resume reference shape even if fake provider uses dummy data.

Acceptance:

- Full session lifecycle works end to end with fake provider.
- UI/API can show running, blocked, interrupted, stopped, expired and resuming.
- Reloading Core/UI reconstructs session state from persistence.
- Expired runtime can be resumed or shown as degraded when resume is impossible.

### M7. Codex Provider Adapter

Цель: заменить fake provider первым production adapter без утечки
Codex-specific details в Core or UI.

Tasks:

- Провести короткий protocol spike for Codex launch, streaming, resume,
  interrupt, stop, approvals and provider session identity.
- Реализовать Codex adapter behind the Provider Adapter boundary.
- Normalize Codex output into `provider.output.delta` and completed assistant
  messages.
- Normalize provider approval/user-input requests into Cortex approval events.
- Capture provider session id or resume cursor when available.
- Реализовать safe error mapping for provider startup, missing binary,
  unsupported resume, interrupted process and provider crash.
- Keep fake provider available for tests and local demos without Codex.

Acceptance:

- User can start a Codex-backed session in a registered workspace.
- User can send multiple turns into the same live runtime while it is active.
- Provider output streams through Core and appears in the Web Control Panel.
- Stop and interrupt work for supported Codex process behavior.
- Resume works where provider support exists; otherwise UI shows degraded resume
  without losing thread history.

### M8. Web Control Panel

Цель: поставить видимую product surface версии `0.1`.

Tasks:

- Build application shell with left inventory tree and main activity surface.
- Implement node list, add/pair node flow, revoke action, diagnostics, heartbeat
  age and status badges.
- Implement project/workspace placement form and validation state.
- Implement session list and session route.
- Implement session header with node, project/workspace, provider, runtime
  status and warning badges.
- Implement chat message composer and stream rendering.
- Render user messages, assistant text, system/runtime events, warnings, errors
  and approval cards.
- Implement lifecycle controls: start, attach, detach, interrupt, stop, resume.
- Subscribe to session event stream and update TanStack Query cache.
- Add trusted/local/non-production warning in node pairing and remote-control
  surfaces.
- Keep reserved layout space for future workspace/artifact/trace panels without
  implementing those panels.

Acceptance:

- User can complete the full `0.1` release contract from the browser.
- UI reload reconstructs state from Core snapshots and resumes event streams.
- Stale/offline/error states are visible in tree and session header.
- Chat surface remains usable without pretending to be a full IDE.

### M9. End-To-End Hardening

Цель: сделать первую release достаточно надежной для actual local development
use.

Tasks:

- Add end-to-end smoke script or documented manual runbook for Core + Node + Web.
- Test single local node, two local nodes, stale node, offline node, Core
  restart, Node restart, UI reload, duplicate command, provider crash and
  degraded resume.
- Add structured logs around enrollment, heartbeat, command dispatch, runtime
  lifecycle and event append.
- Add user-safe error messages and raw debug fields where appropriate.
- Check database migrations from empty database and from previous dev snapshots.
- Document known limitations and recovery steps.
- Run `make c` before handoff.

Acceptance:

- The documented golden path works from a clean checkout.
- Failure scenarios do not corrupt persisted session history.
- All non-goals remain clearly outside the shipped surface.
- `make c` passes or has documented no-op behavior for unscaffolded stacks.

## Рекомендуемый порядок сборки

1. M0 repo/tooling scaffold.
2. M1 protocol/domain foundation.
3. M2 Core persistence and server skeleton.
4. M3 enrollment/heartbeat.
5. M4 workspace placement/resource signals.
6. M5 control channel.
7. M6 fake provider lifecycle.
8. M8 Web Control Panel in parallel with M6 after APIs stabilize.
9. M7 Codex provider adapter.
10. M9 hardening and release gate.

Fake provider должен появиться до Codex adapter. Это снижает integration risk и
дает Core/UI стабильный contract for tests.

## Parallel Work Tracks

После M1 можно вести параллельно:

- Core persistence/API implementation.
- Node daemon heartbeat/control implementation.
- Web shell and read-only snapshots.
- Provider adapter spike.
- Test harness and local runbook.

Не стоит параллелить API shape changes после M5 без явного contract update:
иначе Core, Node and Web начнут расходиться.

## Quality Gates

Каждый milestone должен оставлять repository in runnable state.

Обязательно перед handoff `0.1`:

- `make c`
- Rust formatting and clippy with warnings denied where Rust workspace exists.
- Rust unit/integration tests for domain, persistence, command/event logic and
  provider adapter behavior.
- Web lint, typecheck, build and component/unit tests where web app exists.
- Manual or automated golden-path E2E for Core + Node + Web + fake provider.
- Manual or automated golden-path E2E for Core + Node + Web + Codex provider,
  если Codex доступен в environment.

## Release Gate Checklist

`0.1` можно tag только когда:

- Release contract satisfied end to end.
- Trusted/local/non-production profile visible and documented.
- Persistent Core state survives restart.
- Node stale/offline behavior visible and does not destroy history.
- Session history, messages, runtime events and provider resume reference stored
  in Core.
- Command idempotency prevents duplicate turns during retry/reconnect.
- Event ordering is monotonic per runtime session.
- Codex-specific behavior contained inside provider adapter.
- Web Control Panel does not depend on direct Node access.
- Known limitations documented.
- `make c` passes.

## Main Risks And Mitigations

| Risk | Mitigation |
| --- | --- |
| Codex persistent protocol weaker than desired runtime model | Build fake provider first, keep degraded resume explicit, isolate Codex specifics behind Provider Adapter |
| Core/Node/Web contracts drift during implementation | Land shared protocol crate before feature work and require JSON round-trip tests |
| Web UI grows into future workbench scope | Keep `0.1` to control panel, chat, lifecycle, status and reserved layout only |
| Security expectations become ambiguous | Ship only trusted/local/controlled-development profile with visible warning and no production-security claim |
| Event log becomes raw text dump | Normalize minimum runtime/message/approval/error events from the start |
| Reconnect/retry creates duplicate turns | Make `command_id` mandatory and test Node command dedupe |
| Stale/offline state loses user trust | Keep historical sessions readable and show last known state, heartbeat age and blocked commands clearly |

## Post-0.1 Follow-Up Candidates

После shipping `0.1` эти направления должны вернуться в feature queue:

- security baseline;
- runtime/session hardening;
- workspace shell and reference model;
- read-only Project Workspace Inspector;
- workspace intervention layer;
- causality and trace UX;
- git/review basics;
- Tool Registry v1.
