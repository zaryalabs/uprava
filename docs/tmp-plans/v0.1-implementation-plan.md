## Purpose

This document turns the already designed V01 product slice into the
implementation sequence for version `0.1`.

`V01` is the product scope: **Distributed Agent Control Panel**. `0.1` is the
first implementation release where this scope works end to end in a trusted
local/single-user or controlled development deployment.

This plan is temporary. When implementation produces durable decisions, those
decisions must be promoted into canonical documentation. This file must not
become the permanent source of architecture.

## Release Contract 0.1

Version `0.1` is ready when a user can:

1. Start Core Backend and Web Control Panel locally.
2. Start at least one Node Daemon and enroll it in Core.
3. See node status, heartbeat age, capabilities,
   degraded state and active sessions.
4. Register or open a project/workspace binding on the selected node.
5. Start a Codex-backed persistent interactive session for that workspace.
6. Send chat turns through Core-routed commands.
7. Receive ordered provider/runtime events in the session view.
8. Detach from and reattach to the same session.
9. Interrupt, stop and resume where the provider adapter supports it.
10. Reload the UI and reconstruct nodes, workspace bindings, sessions,
    messages, runtime state and recent event history from Core persistence.
11. See stale/offline/error/resource warning states as UI state, not as text
    hidden in chat.
12. See explicit trusted/local/non-production warning where the UI exposes node
    pairing or remote control.

## Explicitly Out Of Scope For 0.1

In `0.1`, we do not implement:

- production security hardening;
- multi-user auth, team RBAC, secrets management, hardened credentials;
- full Tool Registry or Plugin Registry;
- MCP integration platform;
- Project Workspace Inspector, file tree, editor, terminal, diff/check surface;
- full visual artifact system or artifact gallery beyond lightweight
  session-local `artifact.tree` scaffolding;
- task-based sandbox runs or durable workflow engine;
- multi-agent orchestration;
- cloud node pool;
- external plugin ecosystem or arbitrary plugin JS in the main React tree;
- full provider-neutral parity beyond the Codex-first adapter;
- Next.js as a required app runtime;
- Tauri-specific domain logic.

## Architecture Slices

### Core Backend

Core - client-facing control plane.

Core owns:

- node enrollment and registry;
- heartbeat projection and node status;
- project and workspace placement metadata;
- session threads, turns, messages, runtime projections;
- command intents and idempotency records;
- ordered event log and per-session projections;
- warning acknowledgements;
- API and live event streams for Web Control Panel;
- static delivery of the built web app, if practical for `0.1`.

Core must not:

- execute local files, shell or provider processes directly;
- require the browser to connect directly to every node;
- treat provider process state as durable truth.

### Node Daemon

Node Daemon is the data plane on a concrete machine.

Node owns:

- local daemon config and development credential;
- outbound registration, heartbeat and control channel;
- workspace path validation;
- minimal capability and health snapshot;
- minimal git/resource snapshot;
- provider process lifecycle;
- command execution and deduplication by `command_id`;
- provider event normalization into Cortex events;
- runtime lifetime policy and local process handles.

Node must not:

- own global session history;
- bypass Core for client-visible commands;
- hide important provider actions only in raw terminal output.

### Web Control Panel

Web Control Panel is the first client. It should feel like a control panel, not
a full IDE.

UI owns:

- app shell and `Nodes -> Projects/Workspaces -> Sessions` tree;
- node pairing and diagnostics surfaces;
- project/workspace binding flow;
- session/chat surface;
- message, runtime event, status and approval rendering;
- lifecycle controls: start, attach, detach, interrupt, stop, resume;
- visible warning badges and trusted deployment messaging.

UI must not:

- connect directly to Node Daemons;
- invent durable state that does not exist in Core;
- show future workspace/editor/terminal surfaces as if they already
  belong to `0.1`.

## Data Model Baseline

The first schema should be small but explicit. Names can change during
implementation, but responsibility boundaries must remain.

### Core Tables

- `nodes`: `node_id`, display name, enrollment state, presence state, optional
  sleep/power hint, last heartbeat, version, created/updated timestamps.
- `node_capabilities`: node id, capability key, version/value JSON.
  This is an actual-state snapshot, not the future Tool Registry source of
  truth.
- `node_enrollments`: short-lived pairing request, pairing code hash, status,
  expiry, claimed node id.
- `projects`: thin logical project identity, display name, optional repo id.
- `project_placements`: node-local workspace binding candidate with node id,
  project id, workspace path, git snapshot, last validation.
- `actors`: minimal local actor identities for `local_user`, `system`,
  `node`, `provider` and future `agent`/`external_integration` attribution.
- `session_threads`: durable user work thread, project placement, run mode,
  runtime strategy, work contract, status, provider resume reference.
- `runtime_sessions`: live-or-recoverable runtime projection, provider,
  lifecycle state, timestamps, last runtime step, exit reason.
- `turns`: user turn sequence, status, linked messages and blocked request.
- `messages`: user, assistant, system, runtime and approval-visible messages.
- `commands`: command id, actor ref, target node/runtime/session, command kind,
  status, dedupe key, source/cause refs, payload, created/completed timestamps.
- `events`: event id, command id, actor ref, scope ref, optional node id,
  optional runtime session id, optional session thread id, optional turn id,
  seq, kind, source/evidence/cause/result refs, payload, happened timestamp.
- `approvals`: approval or user-input request, status, payload, response.
- `warning_acknowledgements`: actor, warning kind, affected resources, command
  id, timestamp.

### Node Local State

- daemon installation id and config;
- Core URL and development node credential;
- registered workspace roots or explicit workspace paths;
- provider process handles and local runtime metadata;
- recent command dedupe cache, persisted where needed for reconnect safety;
- provider-native session id or resume cursor, if available.

## Event And Command Contract

### Command Envelope

Every Core-to-Node command must include:

```text
command_id
kind
target_node_id
actor_ref
session_thread_id optional
runtime_session_id optional
project_placement_id optional
source_refs optional
cause_refs optional
issued_at
payload
```

Initial command kinds:

- `StartRuntime`
- `ResumeRuntime`
- `SendTurn`
- `ResolveApproval`
- `InterruptRuntime`
- `StopRuntime`
- `ValidateWorkspace`
- `RefreshResourceSnapshot`

Rules:

- Core records command intent before dispatch.
- `actor_ref` in `0.1` can be one of `local_user`, `system`, `node`,
  `provider` or `unknown`; the shape must leave room for future `agent` and
  `external_integration` actors.
- Node deduplicates by `command_id`.
- Commands never hidden inside heartbeat payloads.
- If the control channel is unavailable, Core records pending dispatch and asks
  Node to open a control channel through the heartbeat response.

### Event Envelope

Every Core/Node/provider/system event must include:

```text
event_id
command_id optional
actor_ref
scope_ref
node_id optional
runtime_session_id optional
session_thread_id optional
turn_id optional
seq
kind
happened_at
source_refs optional
evidence_refs optional
cause_refs optional
result_refs optional
payload
```

Initial event kinds:

- `runtime.starting`
- `runtime.ready`
- `runtime.running`
- `runtime.blocked`
- `runtime.expired`
- `runtime.resuming`
- `runtime.stopped`
- `runtime.error`
- `turn.started`
- `turn.completed`
- `turn.interrupted`
- `provider.output.delta`
- `provider.message.completed`
- `approval.requested`
- `approval.resolved`
- `coordination.warning_acknowledged`
- `workspace.validated`
- `resource.snapshot.updated`

Rules:

- For `0.1` runtime events use `scope_ref = runtime_session_id`; later
  workspace, tool, artifact, workflow and integration events can use their own
  scope refs without changing the envelope.
- `seq` monotonic per `scope_ref`; for runtime streams this means monotonic per
  `runtime_session_id`.
- Core deduplicates by `event_id`.
- Core can detect gaps by `seq`.
- UI projection is built per `session_thread_id`.
- Events carry actor/source/cause refs when known, and explicitly omit them when
  unknown instead of inventing causality.
- Global ordering across all nodes is not needed for `0.1`.

## Distributed Runtime Implementation Blueprint

This section intentionally duplicates part of the design docs in implementation
form. During development, this is the baseline contract for Core, Node and Web;
the design docs are used to verify direction.

### Main Runtime Flow

Golden path:

```text
Web command
-> Core API validates actor, placement, node and resource signals
-> Core records command intent
-> Core dispatches command to Node over control channel
-> Node acknowledges command receipt
-> Node executes through Provider Adapter
-> Node emits ordered events
-> Core persists events and updates projections
-> Web receives SSE events and refreshes affected query caches
```

Important invariants:

- browser never connects directly to Node;
- heartbeat negotiates channel state, but never carries runtime commands;
- command state is durable in Core before dispatch;
- event state is durable in Core before UI treats it as truth;
- Node may buffer events during short Core/channel outages, but Core remains the
  durable product history;
- command retry must not create duplicate turns, approvals or runtime actions;
- provider-specific details do not escape the Provider Adapter boundary.

### State Machines

State enums should be encoded in the shared protocol/domain crate and tested as
transitions where practical.

Node enrollment:

```text
unregistered -> pending_user_approval -> registered
pending_user_approval -> expired
pending_user_approval -> rejected
registered -> revoked
```

Daemon presence:

```text
reachable -> stale -> offline
stale -> reachable
offline -> reachable
reachable -> revoked
stale|offline -> revoked
```

Node activity is not the same state machine as presence. A registered daemon
with fresh heartbeats and no active runtime is the normal baseline:

```text
reachable_idle     = reachable + active_runtime_count == 0
reachable_active   = reachable + active_runtime_count > 0
reachable_degraded = reachable + health/capability/resource degradation
```

Sleep/hibernation should be represented as a hint or reason, not as the only
source of truth:

```text
sleep_hint = unknown | awake | suspending | sleeping | woke
```

Reason: when a laptop sleeps, the daemon usually cannot heartbeat. Core can
infer `stale` or `offline`, but it can call the state `sleeping` confidently
only if Node reported a pre-sleep hint, a managed devbox/cloud provider reported
hibernation, or the daemon reports a wake reason on the next heartbeat. UI can
render `possibly sleeping` for stale personal nodes, but command routing should
still use reachability.

Control channel:

```text
closed -> requested -> connecting -> open
open -> closing -> closed
open -> lost -> requested
```

Command:

```text
recorded -> pending_dispatch -> dispatched -> acknowledged
acknowledged -> completed
acknowledged -> failed
recorded|pending_dispatch -> blocked
dispatched|acknowledged -> expired
```

Runtime session:

```text
starting -> ready -> running
running -> blocked
blocked -> running
running|blocked -> stopping -> stopped
running|blocked -> interrupted -> ready
running|ready|blocked -> stale
stale -> resuming -> running
running|ready|blocked|resuming -> error
running|ready|blocked|stale -> expired
expired -> resuming -> running
```

Session thread:

```text
created -> active
active -> detached
detached -> active
active|detached -> stopped
active|detached -> degraded
```

Turn:

```text
created -> dispatched -> running
running -> blocked_on_approval
blocked_on_approval -> running
running -> completed
running -> interrupted
created|dispatched|running -> failed
```

Hard blocks:

- node revoked or offline;
- workspace missing or not writable for write-capable runtime;
- required capability missing;
- command targets a stale runtime without a resume/reconnect path.

Warnings with persisted override:

- stale node;
- dirty workspace;
- same workspace active;
- same normalized repo/branch active elsewhere;
- branch behind upstream;
- optional capability missing.

### Control Channel Frames

The Node control channel should use small versioned JSON frames first. Binary
or compressed frames can come later only if needed.

Common frame fields:

```text
frame_id
protocol_version
kind
sent_at
payload
```

Core to Node frame kinds:

- `control.hello_ack`
- `command.dispatch`
- `command.cancel`
- `event_batch.ack`
- `config.update`
- `control.ping`

Node to Core frame kinds:

- `control.hello`
- `command.ack`
- `command.result`
- `event_batch`
- `node.diagnostic`
- `control.pong`
- `control.error`

Rules:

- Node sends `control.hello` with `node_id`, daemon version, protocol version,
  active runtime ids and last known event seq per runtime.
- Core rejects incompatible protocol versions before accepting commands.
- `command.dispatch` includes the full command envelope and expected target
  ids.
- Node sends `command.ack` after dedupe check and before provider execution.
- Node sends `command.result` for command-level completion/failure; runtime
  output still flows as events.
- Node sends `event_batch` in runtime seq order.
- Core persists accepted events and replies with `event_batch.ack`.
- Node keeps a bounded local outbox of unacked events until ack, reconnect or
  retention expiry.
- If local outbox retention is exceeded, Node emits the next possible
  `runtime.error` or `runtime.stale` event and Core marks the stream gap.

### Dispatch And Reconnect Rules

Core command dispatch:

1. validate placement, node state and resource warnings;
2. create `command_id`;
3. persist `commands` row as `recorded`;
4. write warning acknowledgement events if user overrode warnings;
5. mark command `pending_dispatch`;
6. if channel open, send `command.dispatch`;
7. if channel closed, set `channel_needed` for node and wait for heartbeat;
8. on `command.ack`, mark `acknowledged`;
9. on terminal command result or terminal runtime event, mark `completed` or
   `failed`.

Node reconnect:

1. Node heartbeats with `control_channel_open = false`;
2. Core heartbeat response says `open_control_channel = true` if work is
   pending or active;
3. Node opens outbound control channel;
4. Node sends active runtime ids, command dedupe cursor and last emitted seqs;
5. Core sends any pending commands whose targets are still valid;
6. Node retransmits unacked event batches;
7. Core deduplicates events by `event_id` and validates per-runtime `seq`;
8. UI receives normal events or a projection reload signal.

Duplicate handling:

- same `command_id` on Node returns previous command status/result where
  possible;
- same `event_id` in Core is ignored after verifying compatible payload;
- same `seq` with different `event_id` is a stream conflict and marks runtime
  degraded;
- gap in `seq` marks runtime/session projection as `degraded` until snapshot
  repair or explicit resume.

### Projection Rules

Core should maintain projections as read models, not as separate truth.

Minimum projections:

- inventory tree: nodes, placements, active sessions and warning badges;
- node detail: heartbeat age, capabilities, diagnostics, active runtime summary;
- placement detail: workspace validation, git/resource snapshot, start
  eligibility;
- session detail: session thread, runtime state, messages, approvals, warnings,
  recent events;
- runtime summary: lifecycle state, provider, last step, resume support,
  degraded reason.
- artifact tree: session-local index of turns, messages, approvals, warnings,
  runtime events, resource snapshots and artifact placeholders.
- agent projection: compact agent-readable session state and resume context
  rebuilt from Core persistence.

Messages are derived from events plus stored user turns:

- user turn creates a user message immediately after Core accepts command;
- `provider.output.delta` updates an in-progress assistant message;
- `provider.message.completed` finalizes assistant message content;
- `approval.requested` creates an approval-visible message/card;
- `runtime.*`, `workspace.*` and `resource.*` events are rendered as compact
  timeline/status blocks, not as assistant text.

### Node Implementation Modules

Node Daemon should stay split by responsibility:

```text
config          load/save daemon config and node credential
enrollment      request code, claim credential, revoke local state
heartbeat       periodic status and channel negotiation
control         websocket lifecycle, frame send/receive, reconnect
workspaces      explicit path validation and resource snapshots
commands        command dedupe and dispatch to runtime manager
runtimes        runtime process handles and lifecycle state
providers       Provider Adapter trait, fake provider, Codex provider
event_outbox    bounded local event buffering and ack handling
diagnostics     health/resource summaries and safe debug output
```

Node local persistence can be small and file-based in `0.1`, but it should have
a clear owner and schema. It must store at least daemon config, node credential,
recent command dedupe records and unacked event outbox records needed for
reconnect safety.

### Distributed Test Matrix

The implementation should add integration tests or smoke scenarios for:

- enrollment approval and claim;
- heartbeat reachable idle -> stale -> offline -> reachable idle;
- reachable idle -> reachable active -> reachable idle as runtimes start/stop;
- sleep hint reported before daemon sleep and cleared/reconciled on wake;
- control channel requested from heartbeat response;
- command pending while channel is closed, then dispatched after reconnect;
- duplicate command delivery does not duplicate turn execution;
- duplicate event delivery does not duplicate messages;
- event seq gap marks projection degraded;
- Core restart while Node/runtime is active;
- Node restart with active or recoverable runtime metadata;
- stale/offline node blocks new turns but keeps history readable;
- warning override creates `coordination.warning_acknowledged`;
- provider crash maps to `runtime.error` and user-safe UI state.

ship external plugins or generated dynamic UI, but the first implementation
must be structured around surfaces, blocks, references, commands and context so
future work does not require reshaping the app.

### Internal Block Contract

Session timeline and future surfaces should render through an internal block
registry, not through one hardcoded transcript component.

Minimal TypeScript shape:

```ts
type CortexRef =
  | { kind: "node"; nodeId: string }
  | { kind: "placement"; placementId: string }
  | { kind: "session"; sessionThreadId: string }
  | { kind: "runtime"; runtimeSessionId: string }
  | { kind: "turn"; turnId: string }
  | { kind: "message"; messageId: string }
  | { kind: "block"; blockId: string }
  | { kind: "artifact"; artifactId: string }
  | { kind: "event"; eventId: string; scopeRef: CortexRef; seq: number }
  | { kind: "command"; commandId: string }
  | { kind: "approval"; approvalId: string }
  | { kind: "warning"; warningKind: string; commandId?: string }
  | { kind: "tool-call"; toolCallId: string }
  | { kind: "file"; placementId: string; path: string; version?: string }
  | {
      kind: "file-range";
      placementId: string;
      path: string;
      range: TextRange;
      version?: string;
    }
  | { kind: "terminal"; terminalId: string; placementId: string }
  | { kind: "terminal-command"; terminalCommandId: string; terminalId?: string }
  | { kind: "terminal-output-range"; terminalCommandId: string; range: TextRange }
  | { kind: "diff-hunk"; diffId: string; hunkId: string }
  | { kind: "check-result"; checkRunId: string; failureId?: string }
  | { kind: "external-entity"; integrationKind: string; externalId: string }
  | { kind: "unknown"; refType: string; locator: Record<string, unknown> };

type TextRange = {
  startLine?: number;
  endLine?: number;
  startOffset?: number;
  endOffset?: number;
};

type UiBlock = {
  blockId: string;
  type: string;
  schemaVersion: number;
  surfaceId: string;
  primaryRef: CortexRef;
  parentRef?: CortexRef;
  children?: UiBlock[];
  sourceRefs?: CortexRef[];
  evidenceRefs?: CortexRef[];
  causeRefs?: CortexRef[];
  relatedRefs?: CortexRef[];
  traceRefs?: CortexRef[];
  data: unknown;
  actions?: string[];
  fallbackText?: string;
};
```

Resolution rules:

- `0.1` must resolve node, placement, session, runtime, turn, message, block,
  event, command, approval, warning and artifact-placeholder refs.
- Workspace, terminal, diff, check, tool-call and external-entity refs are
  reserved contract shapes in `0.1`; resolving them may return
  `not_implemented`, `not_available`, `permission_denied` or safe metadata until
  the corresponding feature queue slice ships.
- Refs are Core-resolvable and permission-scoped. They are not DOM ids.
- Important visible objects should carry source/evidence/cause refs when known
  and preserve an explicit missing/unknown state when not known.
- Fallback rendering must preserve copyable refs even when the rich target
  surface is unavailable.

Initial core block types:

- `core.inventory-tree`;
- `core.artifact-tree`;
- `core.agent-chat`;
- `core.user-message`;
- `core.assistant-message`;
- `core.provider-output-stream`;
- `core.runtime-event`;
- `core.warning`;
- `core.error`;
- `core.approval-request`;
- `core.workspace-validation`;
- `core.resource-snapshot`;
- `core.agent-projection-summary`;
- `core.unknown`.

Renderer rules:

- renderer receives typed block data and current context;
- renderer may show actions, but it does not execute privileged behavior
  directly;
- actions go through the command registry;
- unknown block types render `core.unknown` with safe metadata and raw payload
  only if safe;
- raw provider text is treated as content, not as executable HTML;
- markdown rendering is allowed for assistant text, sanitized, and isolated
  from command execution.

Universal block rules:

- every visible block must have `blockId`, `type`, `schemaVersion`, `surfaceId`
  and `primaryRef`;
- block types are resolved through a registry, not `switch` statements spread
  across feature components;
- a renderer declares allowed surfaces, supported schema versions, actions and
  fallback behavior;
- nested blocks are allowed only where the surface expects them, such as
  `artifact.tree` and future reports;
- block descriptors can be stored or reconstructed from Core projections;
- adding a new internal block type should require registering a descriptor,
  renderer and tests, not editing the whole shell;
- external plugin-provided renderers remain out of scope for `0.1`, but the
  internal API should resemble a future contribution point.

Minimal renderer registration shape:

```ts

  getNavigableObjects?: (block: UiBlock) => NavigableObject[];
};
```

### Artifact Tree And Agent Projection

`0.1` should not build the full visual artifact system, but it should avoid a
chat-only model by shipping two small read models.

#### Artifact Tree

The artifact tree is a lightweight session-local index, not a full artifact
gallery. It groups important visible objects so the session can scale beyond a
linear transcript.

Initial artifact tree nodes:

- session root;
- turns;
- assistant messages;
- approval requests;
- warnings and warning acknowledgements;
- runtime lifecycle/error events;
- workspace validation results;
- resource snapshots;
- provider resume/degraded-resume records;
- future artifact placeholders when Core receives an unknown structured
  artifact/event.

Rules:

- every tree node has a `CortexRef`;
- tree nodes may carry `sourceRefs`, `evidenceRefs`, `causeRefs` and
  `relatedRefs` using the same `CortexRef` model as timeline blocks;
- selecting a node opens the inspector stack;
- tree nodes can point back to timeline blocks and source events;
- missing source/cause must be represented as unknown or unavailable, not filled
  with agent-authored guesses;
- unknown artifact-like payloads are indexed as metadata plus safe fallback,
  not rendered as executable UI;
- this does not include file tree, editor, terminal, full diff view or artifact
  gallery.

#### Agent Projection

Agent projection is the minimal agent-readable read model of current session
state. It is not agent memory and not a hidden prompt transcript. It is a
structured snapshot Core can show in diagnostics, pass to a provider adapter as
resume context, and later expose to agents/tools as context.

Minimum shape:

```text
agent_projection
  session_thread_id
  project_placement
  runtime_summary
  current_turn optional
  pending_approvals
  active_warnings
  recent_turn_summaries
  recent_message_refs
  artifact_tree_summary
  available_block_types
  available_commands
  visible_refs
  source_cause_summary
  resume_context
  generated_at
```

Rules:

- projection is rebuilt from Core persistence and event log;
- projection is safe to show in UI diagnostics;
- projection excludes credentials, raw environment, secrets and unredacted
  provider payloads;
- provider adapter may use projection for degraded resume or explicit resume
  context;
- projection should say what is unknown or stale instead of inventing state;
- projection should distinguish human, system, node/provider and future agent
  authored state when that attribution exists;
- future agent-facing capability discovery should extend this shape rather than
  adding a parallel context model.

## Cross-Cutting Implementation Baselines

These details are not separate product features, but they must be defined before
active implementation. Otherwise Core, Node and Web will start making local
decisions that later have to be reconciled.

### Trusted Development Security Baseline

`0.1` is not a production security release, but the trusted/dev profile must
still have explicit boundaries.

Minimum:

- Core default bind address is loopback unless user explicitly enables remote
  access.
- Node Daemon stores development credential outside repository/workspace paths.
- pairing code is short-lived and never reused as node credential.
- node credential never appears in URL, query string, UI copy, logs or event
  payloads.
- Core stores only node credential hash or equivalent verification material.
- revoke immediately rejects heartbeat and control channel attempts from that
  node.
- local HTTP/WS is allowed only for trusted loopback/dev profile; remote profile
  must show explicit non-production warning until security baseline ships.
- browser API should reject unknown origins in remote/dev-cross-origin cases
  where practical.
- logs redact bearer tokens, pairing codes, environment variables and provider
  secrets.
- UI labels all pairing and remote-control surfaces as trusted/local or
  controlled-development, not production-secure.

Explicitly not in `0.1`:

- team auth/RBAC;
- hardened browser login/session model;
- keychain-backed credential storage;
- credential rotation UX;
- mTLS/request signing;
- secrets management.

### API Error And Compatibility Contract

All Core APIs should return a consistent error envelope:

```text
error_code
message
details optional
retryable
correlation_id
```

Rules:

- `message` is user-safe.
- `details` is structured and safe to show in diagnostics when appropriate.
- raw provider/node errors are captured only in redacted debug fields/logs.
- every request gets a correlation id and it flows into commands/events where
  possible.
- command APIs return `command_id` when a durable command was recorded, even if
  later dispatch is pending or blocked.
- API and control frames include protocol/API version.
- breaking DTO changes require updating shared examples and Web protocol
  fixtures in the same change.

Initial error code families:

- `validation.*`
- `node.*`
- `placement.*`
- `runtime.*`
- `provider.*`
- `command.*`
- `event_stream.*`
- `auth_dev.*`
- `internal.*`

### Persistence And Migration Policy

SQLite is the durable truth for Core in `0.1`; migrations must be boring and
repeatable.

Rules:

- migrations run automatically on Core startup in local/dev profile;
- schema version is visible in diagnostics;
- empty database bootstraps without manual steps;
- migrations are append-only after a release tag;
- tests cover empty database and at least one previous dev snapshot where
  practical;
- repository layer owns all SQL access;
- projections can be rebuilt from durable rows/events;
- database path is explicit in config and logs at startup;
- deleting/resetting dev database is a deliberate documented action, not a
  hidden side effect.

Suggested SQLite pragmas for local/dev:

- `foreign_keys = ON`;
- WAL mode where practical;
- busy timeout configured;
- timestamps stored consistently as UTC.

V01 needs enough observability to debug distributed behavior without turning UI
into raw logs.

Structured log fields:

- `correlation_id`;
- `command_id`;
- `event_id`;
- `node_id`;
- `session_thread_id`;
- `runtime_session_id`;
- `project_placement_id`;
- `provider`;
- `control_channel_id`.

Core spans/events:

- enrollment created/approved/claimed/revoked;
- heartbeat accepted/rejected;
- control channel open/close/lost;
- command recorded/dispatched/acknowledged/completed/failed/blocked;
- event batch received/accepted/rejected/acked;
- projection gap/degraded state;
- runtime lifecycle transition;
- API error with safe code.

Node spans/events:

- config loaded;
- enrollment request/claim;
- heartbeat send/result;
- control channel connect/reconnect;
- command dedupe hit/miss;
- provider launch/resume/interrupt/stop;
- event outbox enqueue/ack/retry/drop;
- workspace validation/resource snapshot.

Diagnostics endpoints should return safe summaries, not secrets or raw

### Provider Process Supervision

Provider Adapter implementation must define process behavior, not only method
names.

Minimum:

- launch command and cwd are explicit and logged in redacted form;
- provider process has lifecycle owner in Node runtime manager;
- stdout/stderr/protocol stream parsing is separated from event normalization;
- interrupt and stop have timeout-based escalation;
- process exit maps to `runtime.stopped`, `runtime.error` or
  `turn.interrupted` based on current state;
- provider startup failure maps to user-safe `provider.start_failed`;
- missing binary maps to `provider.missing_binary`;
- unsupported resume maps to degraded resume, not lost history;
- backpressure is bounded for provider output and event outbox;
- raw provider payload retention is explicit and redacted.

## Recommended Build Order

1. M0 repo/tooling scaffold.
2. M1 protocol/domain foundation.
3. M2 Core persistence and server skeleton.
4. M3 enrollment/heartbeat.
5. M4 workspace placement/resource signals.
6. M5 control channel.
7. M6 fake provider lifecycle.
8. M8 Web Control Panel read-only shell can start after M2; command/lifecycle
   parts should run in parallel with M6 after APIs stabilize.
9. M7 Codex provider adapter.
10. M9 hardening and release gate.

Fake provider should land before the Codex adapter. This reduces integration
risk and gives Core/UI a stable contract for tests.

After M1, these tracks can run in parallel:

- Core persistence/API implementation.

## Release Gate Checklist

`0.1` can be tagged only when:

- Release contract satisfied end to end.
- Trusted/local/non-production profile visible and documented.
- Trusted/dev security baseline is implemented: loopback default, credential
  redaction, pairing-code expiry and revoke enforcement.
- Persistent Core state survives restart.
- SQLite migrations are repeatable from empty database and documented previous
  dev snapshots.
- Node idle/active/degraded/stale/offline behavior visible and does not destroy
  history.
- Session history, messages, runtime events and provider resume reference stored
  in Core.
- Command idempotency prevents duplicate turns during retry/reconnect.
- Event ordering is monotonic per runtime session.
- Event envelope has actor refs, scope refs and optional source/evidence/cause
  refs; unknown causality is explicit rather than fabricated.
- Control channel reconnect retries unacked events without duplicating UI
  projection.
- Codex-specific behavior contained inside provider adapter.
- Web Control Panel does not depend on direct Node access.
- Web Control Panel uses internal surfaces, blocks, references and command
  registry for the session UI.
- `CortexRef` supports V01 refs plus reserved future workspace, terminal, diff,
  check, tool-call and external-entity refs with explicit unavailable states.
- Web Control Panel has lightweight artifact tree and agent projection surfaces
  without shipping the full artifact system.
- Artifact tree and timeline blocks preserve source/evidence/cause refs and
  safe fallbacks where available.
- Runtime events, warnings, approvals and errors are visible as structured UI
  state outside assistant prose.
- User-safe API error envelope and correlation ids are used across Core/Web and
  relevant Core/Node paths.
- Structured logs cover enrollment, heartbeat, control channel, command,
  event batch and runtime lifecycle paths.
- Known limitations documented.
- `make c` passes.

## Main Risks And Mitigations

| Risk | Mitigation |
| --- | --- |
| Codex persistent protocol weaker than desired runtime model | Build fake provider first, keep degraded resume explicit, isolate Codex specifics behind Provider Adapter |
| Core/Node/Web contracts drift during implementation | Land shared protocol crate before feature work and require JSON round-trip tests |
| Web UI grows into future workbench scope | Keep `0.1` to control panel, chat, lifecycle, status and reserved layout only |
| UI modularity becomes over-engineered before plugins exist | Implement internal registries only for V01 blocks, refs and commands; no external plugin runtime |
| Artifact tree expands into full artifact product too early | Ship only session-local artifact/ref index and safe placeholders in 0.1 |
| Security expectations become ambiguous | Ship only trusted/local/controlled-development profile with visible warning and no production-security claim |
| Trusted/dev path leaks credentials through logs or diagnostics | Redact pairing codes, tokens, env and provider-sensitive values from the first implementation |
| Event log becomes raw text dump | Normalize minimum runtime/message/approval/error events from the start |
| Reconnect/retry creates duplicate turns | Make `command_id` mandatory and test Node command dedupe |
| Channel reconnect loses live provider output | Add bounded Node event outbox and Core event batch acknowledgements |
| Error handling fragments across Core, Node and Web | Use one error envelope, stable code families and correlation ids |
| Config and local state become implicit | Document defaults, paths and reset flow in the runbook before release handoff |
| Idle nodes look broken or inactive work looks like daemon failure | Split node reachability, activity summary and runtime lifecycle in protocol and UI |
| Stale/offline state loses user trust | Keep historical sessions readable and show last known state, heartbeat age, sleep hint when available and blocked commands clearly |
| Future workspace/trace/plugin slices force protocol rewrites | Ship actor refs, scope refs, reserved CortexRef kinds and source/evidence/cause link fields in the first protocol/domain crate |
