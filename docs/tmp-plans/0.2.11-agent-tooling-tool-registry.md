# План реализации 0.2.11: Agent Tooling and Tool Registry v1

Статус: `epic-1-complete-stage-0-external-gate-blocked`

Целевая версия: `0.2.11`

Канонические направления:

- [Feature Queue, пункт 11](../product/feature-queue.md#11-agent-tooling-and-tool-registry-v1);
- [A-007 Agent Tooling, Tool Registry and MCP Strategy](../systems/areas/007-plugins-tool-registry-and-mcp-strategy.md);
- [A-009 Human-Agent Dual Interface](../systems/areas/009-human-agent-dual-interface.md);
- [общая архитектура](../systems/architecture.md#agent-tooling-and-tool-registry).

## Цель среза

Дать agent session единый MCP-first доступ к Uprava-native capabilities и
первой реальной внешней integration, не загружая полный tool catalog в model
context.

Core владеет registry, policy, effective availability, routing and trace. Node
владеет actual capabilities и ToolHive bridge. ToolHive управляет lifecycle and
proxy внешнего MCP runtime. Web даёт человеку компактную поверхность настройки,
авторизации, диагностики и trace.

## Структура реализации

Работа разделена на короткий обязательный foundation gate и три эпика:

```text
Этап 0. Contract and compatibility foundation
  фиксирует общий язык и снимает dependency risks

Эпик 1. Control Plane and Uprava MCP
  строит Core registry, policy and agent-facing interface

Эпик 2. Node Tool Runtime and ToolHive
  подключает data plane and first external MCP integration

Эпик 3. Web Management and Production Closure
  делает систему управляемой человеком и закрывает release quality
```

Эпики следуют архитектурным границам, а не только каталогам репозитория. Эпик 1
должен работать с deterministic mock backend. Эпик 2 подключает реальный
backend без изменения agent-facing contract. Эпик 3 использует API contracts,
зафиксированные на этапе 0, поэтому Web можно начинать параллельно после их
стабилизации.

## Зафиксированные решения

1. Agent-facing contract: Uprava MCP.
2. Discovery contract: `Search -> Inspect -> Execute`.
3. Постоянная model-visible surface: `search_tools`, `inspect_tool`,
   `execute_tool`.
4. Gateway split: Core принимает agent MCP calls, применяет policy and trace;
   Node bridge исполняет ToolHive-backed calls рядом с runtime.
5. Все managed calls проходят через Core. Direct agent-to-ToolHive exposure не
   входит в v1.
6. Первый внешний acceptance server: официальный remote Linear MCP
   `https://mcp.linear.app/mcp`, подключённый через ToolHive remote proxy.
7. Первый search engine: permission-filtered keyword/BM25; embeddings не нужны.
8. Native `bash`, files, `git`, `gh`, `glab` остаются direct tools и попадают
   только в observed capability inventory.
9. Отдельный Uprava CLI не создаётся.
10. Реализация ориентируется на стабильную MCP revision, доступную на момент
    начала работы. Release candidate protocol changes принимаются только после
    отдельного compatibility review.

## Общие инварианты

1. Полные tool schemas не передаются модели до Inspect.
2. Search фильтрует visibility before ranking and output.
3. Inspect не является authorization grant.
4. Execute всегда повторяет schema, permission, approval and availability
   checks.
5. Core domain/API/command contracts остаются authority; MCP transport не
   создаёт параллельную domain model.
6. ToolHive runtime status не подменяет Core product policy.
7. Definition живёт независимо от временной backend availability.
8. Secret values не попадают в model context, registry output, events or logs.
9. Tool call имеет стабильный `tool_call_id`, correlation and trace refs.
10. Observed capability не выдаётся за managed tool и не обещает execution
    trace, которым Uprava не владеет.
11. Core-to-Node execution использует durable typed command path, а не
    untracked HTTP tunnel.
12. Web не обращается к Node или ToolHive напрямую.

## Вне scope 0.2.11

- Plugin Registry;
- большой integration marketplace;
- direct ToolHive vMCP exposure to agents;
- separate Uprava CLI;
- programmatic tool calling/code mode;
- embeddings/vector database;
- remote installation of native CLI dependencies;
- rich renderers/artifacts for every tool result;
- team/enterprise RBAC;
- generic secret-management platform;
- automatic migration to an unreleased MCP revision.

## Этап 0. Contract and compatibility foundation

### Назначение

До параллельной реализации Core, Node and Web зафиксировать единый protocol and
domain contract и проверить внешние assumptions. Этот этап является gate:
эпики не должны независимо изобретать identifiers, schemas, states or errors.

### 0.1 MCP and ToolHive compatibility spike

- Выбрать Rust MCP SDK или небольшой protocol adapter по критериям:
  Streamable HTTP, auth hooks, JSON Schema fidelity, cancellation, progress,
  notifications and maintained protocol compatibility.
- Зафиксировать dependency versions, licenses and supported MCP revision.
- Проверить ToolHive local binary/API contract на supported platforms.
- Поднять через ToolHive proxy официальный Linear remote MCP.
- Получить `tools/list`, вызвать один read-only Linear tool на test workspace и
  проверить OAuth connect/disconnect.
- Проверить, что Node может получить safe health/schema state без передачи
  credential в Core logs.
- Зафиксировать exact ToolHive commands/API payloads в spike notes/tests, а не
  в долговечной architecture.

### 0.2 Shared domain vocabulary

Зафиксировать типы:

```text
ToolId
ToolSourceId
IntegrationId
McpDependencyInstanceId
ToolCallId

ToolDefinition
ToolAvailability
ObservedCapability
IntegrationConnectionSummary
McpDependencyStatus
ToolCallSummary
ToolCallDetail
```

Для каждого типа определить:

- stable identity and namespace;
- owner and source;
- version/schema hash rules;
- serialization shape;
- redaction rules;
- lifecycle and terminal states;
- backward/forward compatibility expectation.

### 0.3 Progressive discovery contract

Зафиксировать точные schemas:

```text
search_tools(scope, query, filters, cursor)
inspect_tool(scope, tool_id)
execute_tool(scope, tool_id, arguments)
```

Нужно определить:

- bounded result limits and pagination;
- search result detail level;
- Inspect output and unavailable reasons;
- Execute validation/error envelope;
- direct dynamic mounting versus stable `execute_tool` fallback;
- schema change and `tools/list_changed` behavior.

### 0.4 Core-to-Node and event contracts

Зафиксировать typed command/event model:

```text
external tool execute command
cancel command
dependency desired-state update
dependency actual-state report
tool definitions discovered/changed
tool call started/completed/failed/denied
availability changed
```

Extension payload не использовать как постоянный contract.

### 0.5 Authentication and policy contract

Определить:

- short-lived session-scoped MCP access lease;
- actor/session/project/placement binding;
- token audience, expiry, rotation and revocation;
- permission decision `allow | deny | require_approval`;
- integration connection and secret refs;
- audit policy and argument/result redaction;
- OAuth callback state and disconnect semantics.

### 0.6 API and Web contract

Зафиксировать read/mutation contracts, необходимые Web:

```text
tool definitions and Inspect detail
effective availability
observed Node capabilities
integration desired/actual/auth state
connect/reconnect/disconnect
recent tool calls and trace refs
```

Подготовить canonical Rust fixtures and generated/schema-checked Web types до
начала UI implementation.

### Результаты этапа 0

Локальная contract foundation реализована: shared Rust vocabulary,
progressive-discovery schemas, Core-to-Node payloads, lease/policy contracts и
schema-checked Web fixtures находятся в репозитории. Compatibility baseline и
частичный ToolHive/Linear spike описаны в
[`agent-tooling-contracts.md`](../development/agent-tooling-contracts.md) и
[`0.2.11-toolhive-linear-spike.md`](0.2.11-toolhive-linear-spike.md).

Transport и OAuth discovery через ToolHive доказаны. Завершение Linear OAuth,
`tools/list`, read-only call и disconnect не выполнены из-за действующей
пользовательской browser policy, запрещающей `linear.app`. Поэтому этап 0 пока
не прошёл exit criteria, а эпики остаются заблокированы.

- dependency decision record;
- pinned MCP/ToolHive compatibility baseline;
- successful Linear discovery and read-only spike;
- shared Rust protocol/domain types or approved schemas;
- Core-to-Node commands/events;
- API/Web fixtures;
- threat checklist for auth, metadata poisoning and redaction;
- explicit `toolhive_missing` fallback.

### Exit criteria

- Core, Node and Web могут реализовывать contracts без собственных локальных
  трактовок;
- ToolHive and Linear path технически доказан;
- identifiers, states, errors and redaction rules reviewable;
- unresolved spike failure блокирует эпики, а не переносится в production code.

## Эпик 1. Control Plane and Uprava MCP

### Цель

Построить Core-owned Tool Registry и основной machine interface агента. Эпик
завершается работающим end-to-end Uprava-native tool call без зависимости от
ToolHive.

### Результат эпика 1 — 2026-07-19

Эпик реализован поверх локальной contract foundation с deterministic external
mock; внешний ToolHive/Linear gate этапа 0 по-прежнему блокирует эпики 2–3 и
release closure, но не входит в runtime path эпика 1.

- migration `12` обновляет numbered baseline `0.2.10` и создаёт registry,
  availability-supporting state, tool-call journal, session snapshots и MCP
  leases;
- Core атомарно регистрирует и переиндексирует пять bounded Uprava-native
  definitions, сохраняя schema hash/version rules;
- permission-filtered BM25 Search использует bounded signed cursor, Inspect
  возвращает одну current definition, Execute заново проверяет scope, policy,
  schema и availability;
- Uprava MCP на `/mcp` использует `rmcp 2.2.0`, MCP `2025-11-25`, Streamable
  HTTP и публикует только `search_tools`, `inspect_tool`, `execute_tool`;
- session-scoped HMAC lease проверяется на каждом MCP request, ротируется при
  перевыпуске, отзывается при stop session и при ротации Core credential;
- native execution вызывает application services напрямую; deterministic mock
  фиксирует будущую external routing/error boundary;
- tool-call journal сохраняет actor/scope/schema/policy/route, lifecycle events,
  hashes, sizes, refs и только redacted bounded summaries;
- Core Web read API предоставляет definitions, effective availability и recent
  tool calls; integration/Node projections остаются работой эпиков 2–3.

### 1.1 Protocol and persistence

- Реализовать утверждённые shared contracts.
- Добавить versioned command/event payloads.
- Добавить SQLite migrations:

```text
tool_definitions
tool_sources
integration_connections
mcp_dependency_instances
observed_capabilities
tool_calls
session_tool_snapshots
```

- Добавить индексы для stable source identity, effective availability, search
  refresh, recent scoped calls and idempotent command correlation.
- Покрыть migration from current `0.2.10` state.

### 1.2 Registry and availability

- Зарегистрировать небольшой static set Uprava-native definitions.
- Реализовать schema hash and version rules.
- Разделить definitions and runtime availability.
- Реализовать effective availability by actor/Node/project/session.
- Добавить deterministic mock external backend для policy/routing tests.
- Сохранять session tool snapshot там, где это нужно для reproducibility.

### 1.3 Progressive search

- Построить bounded keyword/BM25 index.
- Применять permission visibility before retrieval output.
- Реализовать pagination and source/risk/availability filters.
- Поддержать atomic re-index on definition changes.
- Не раскрывать denied tools через names, counts or diagnostic details.

### 1.4 Uprava MCP endpoint

- Добавить MCP transport рядом с существующим Axum application boundary.
- Не вызывать Core HTTP API через loopback; использовать application services.
- Реализовать `search_tools`, `inspect_tool`, `execute_tool`.
- Добавить short-lived session-scoped MCP access lease.
- Lease связывает actor, session, placement/project, audience and expiry.
- Поддержать revocation on session stop and credential rotation.
- Добавить request/rate/size limits and transport security policy.

### 1.5 Uprava-native tools

Начальный bounded set:

```text
uprava.node.inspect
uprava.workspace.inspect
uprava.session.inspect
uprava.trace.resolve
uprava.ref.resolve
uprava.artifact.describe
uprava.capability.inspect
```

Точный набор можно сократить после contract review. Нельзя автоматически
экспортировать каждый HTTP endpoint или command как MCP tool.

### 1.6 Tool call, trace and audit foundation

- Создать `tool_call_id` до routing.
- Связать actor, scope, tool/schema version, policy decision and route.
- Добавить requested/authorized/started/completed/failed/denied events.
- Сохранять refs, hashes, sizes and bounded summaries before raw payload.
- Security audit писать для denial, approval and mutation policy decisions.

### Acceptance criteria эпика 1

1. [x] Model-facing fixture содержит только три discovery meta-tools.
2. [x] Search находит Uprava-native capability без schema.
3. [x] Inspect возвращает ровно один current definition.
4. [x] Execute повторно проверяет permission, schema and availability.
5. [x] Uprava-native call завершается через Core application service.
6. [x] Trace связывает actor, tool, command/events and result.
7. [x] Expired, foreign and revoked leases отклоняются.
8. [x] Mock backend доказывает routing/error contract для будущего external tool.

## Эпик 2. Node Tool Runtime and ToolHive

### Цель

Подключить реальный external MCP data plane, не меняя Search/Inspect/Execute
contract эпика 1. Для агента Uprava-native и Linear tools различаются source and
availability, но не способом discovery and invocation.

### 2.1 Node capability inventory

- Добавить ToolHive availability/version probe.
- Добавить informational probes для `git`, `gh`, `glab`.
- Репортить version, health and safe authentication status.
- Не проксировать native CLI execution.
- Не раскрывать credential paths, tokens or лишние account metadata.

### 2.2 Desired and actual state

- Core отправляет desired MCP dependency snapshot.
- Node reconciler сравнивает desired and actual state.
- Поддержать states:

```text
toolhive_missing
missing_auth
installing
starting
running
degraded
failed
stopped
```

- Node restart запускает reconciliation; desired state не считается running до
  actual report.
- Node offline немедленно делает effective availability false.

### 2.3 ToolHive adapter

- Реализовать pinned ToolHive CLI/API boundary.
- Создавать и удалять remote MCP proxy workload.
- Получать health, upstream identity, tools and schema hashes.
- Обрабатывать discovery refresh/list changes.
- Нормализовать upstream Linear tools под stable Uprava namespace.
- Возвращать explicit diagnostic state, если ToolHive отсутствует.

### 2.4 Core-to-Node execution

```text
Core policy and tool_call.started
-> typed durable Node command
-> Node ToolHive bridge
-> Linear remote MCP
-> bounded normalized result
-> Node event/result
-> Core tool_call.completed/failed
```

- Добавить timeout and cancellation.
- Ограничить request/result sizes.
- Сделать duplicate dispatch idempotent.
- Не давать Core Web доступ к Node-local ToolHive endpoint.
- Любой terminal failure должен оставлять terminal tool-call state and trace.

### 2.5 Linear connection and authorization backend

- Добавить Core connection record без token value.
- Реализовать backend connect/reconnect/disconnect lifecycle.
- Поддержать OAuth 2.1/DCR официального Linear remote MCP через выбранную trusted
  runtime boundary.
- Начать с read-only policy; mutations требуют explicit approval policy.
- Deauthorization немедленно меняет effective availability.
- Credentials не должны появляться в DB metadata, logs, events or MCP results.

### 2.6 External tool trace

- Связать `tool_call_id` с Node command, ToolHive workload and upstream tool.
- Сохранить tool/schema version, connection ref and policy version.
- Связать ToolHive runtime audit refs с Uprava trace без копирования secrets.
- Обработать schema drift во время persistent session.

### Acceptance criteria эпика 2

1. Node сообщает ToolHive and native CLI inventory.
2. Core различает configured, running and available-to-this-session.
3. ToolHive запускает/proxies официальный Linear MCP.
4. Linear tool появляется через тот же Search and Inspect path.
5. Execute проходит Core policy and durable Node dispatch.
6. Read-only Linear call завершается с bounded result and trace.
7. Missing ToolHive, missing auth, Node offline and revoked connection имеют
   explicit states.
8. Cancellation, timeout, restart and duplicate dispatch покрыты tests.
9. Direct agent-to-ToolHive exposure не требуется.

## Эпик 3. Web Management and Production Closure

### Цель

Дать человеку управляемую поверхность integrations and capabilities и закрыть
security, observability, E2E and release readiness всего среза.

### 3.1 Integration management UI

- Показать configured integrations and desired state.
- Добавить Linear connect, reconnect and disconnect.
- Показать auth state without secret-bearing detail.
- Показать Node and ToolHive runtime state.
- Объяснять missing auth, missing ToolHive, offline Node, degraded backend and
  policy denial.

### 3.2 Capability UI

- Effective availability by Node/project/session.
- Tool Inspect detail: source, schema version, permissions, risk and status.
- Observed `git`/`gh`/`glab` inventory.
- Явно различать managed tool and observed capability.
- Не превращать экран в marketplace и не копировать agent Search UX.

### 3.3 Tool call and trace UI

- Recent tool calls by current scope.
- Status, actor, source, route, started/completed timestamps.
- Approval/denial state where relevant.
- Links to trace, events and result/artifact refs.
- Bounded/redacted arguments and result summaries.

### 3.4 Agent provider delivery

- Подключить Uprava MCP к primary Codex session launch/attach path.
- Передавать endpoint and short-lived lease вне model prompt/transcript.
- Проверить reconnect and revoke behavior.
- Подтвердить, что постоянный prompt prefix содержит только discovery surface.
- Provider-specific dynamic mounting оставить optional optimization;
  `execute_tool` является обязательным fallback.

### 3.5 Product hardening

- Accessible loading/empty/error/degraded states.
- Request rate and size limits.
- OAuth callback state protection and disconnect verification.
- Metadata poisoning/schema validation tests.
- Secret and output redaction tests.
- Availability consistency across API, MCP and Web.
- Operational metrics for registry, search, dispatch, Node and ToolHive errors.
- Runbook for ToolHive installation, Linear connection and failure diagnosis.

### 3.6 Release closure

- Clean migration from `0.2.10`.
- Real opt-in ToolHive + Linear E2E.
- Offline deterministic tests remain required gate.
- Dependency license/advisory review.
- Threat-model review.
- Update package metadata, `docs/releases.md` and implementation ledger.
- Version bump to `0.2.11` only after all completion criteria pass.

### Acceptance criteria эпика 3

1. Пользователь подключает and disconnects Linear из Web.
2. UI честно показывает desired, actual, auth and effective availability.
3. Пользователь видит observed native capabilities без execution proxy.
4. Tool call открывается в trace and result refs.
5. Agent session получает только progressive discovery surface.
6. Expired/revoked MCP access and disconnected integration блокируют calls.
7. No secret values appear in prompts, logs, events, DB metadata or UI payloads.
8. Required offline gates and real opt-in E2E проходят.

## Зависимости и параллельность

```text
Этап 0 blocks all implementation epics

Эпик 1
  critical path: contracts -> persistence -> registry -> MCP -> native call

Эпик 2
  starts after stage 0
  ToolHive adapter may run in parallel with Epic 1
  final Core/Node integration depends on Epic 1 execution contract

Эпик 3
  Web shell may start after stage 0 API fixtures
  real data integration depends on Epic 1 and Epic 2 read models
  provider delivery and release closure depend on both backend epics
```

Практически возможны три параллельных workstreams после этапа 0:

- Core registry/MCP;
- Node/ToolHive;
- Web UI over fixtures.

Но объединение происходит только через утверждённые contracts, а не через
локальные ad hoc payloads.

- Node-specific availability;
