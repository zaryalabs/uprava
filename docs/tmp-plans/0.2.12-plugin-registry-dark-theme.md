Статус: `completed`

## Цель среза

Доказать, что Plugin Registry расширяет саму Uprava, а не дублирует Tool
Registry or integrations.

Первый working package — bundled data-only `uprava.theme-dark`. Он регистрирует
Dark Theme через versioned `ui.theme` contribution, проходит Core-owned
install/enable lifecycle и применяется Web Extension Host ко всем first-party
surfaces с безопасным `core.light` fallback.

## Product acceptance

Пользователь открывает Plugins/Appearance, видит установленный Dark Theme,
включает plugin, выбирает `uprava.dark` и без reload получает согласованную
темную тему App Shell, Agent, Workbench, Jobs, Inspector, Monaco and xterm.

После disable:

- Web немедленно возвращается к `core.light`;
- App Shell остается работоспособным;
- выбранный theme id не превращается в dangling UI state;
- после reload effective theme остается светлой;
- повторный enable позволяет снова выбрать или восстановить Dark Theme;
- lifecycle changes видны в Core audit без хранения лишнего client state.

## Scope 0.2.12

### Входит

- transport-neutral plugin manifest and registry contracts;
- bundled package discovery and idempotent registration;
- Core persistence для packages, installations, configuration and permission
  grants;
- compatibility and manifest validation;
- enable/disable lifecycle and effective contribution projection;
- Web Extension Host foundation;
- `ui.theme` contribution v1;
- `core.light` built-in fallback theme;
- bundled `uprava.theme-dark` package;
- versioned local appearance preference;
- pre-React theme bootstrap without visible flash;
- semantic token cleanup across first-party Web UI;
- Monaco and xterm theme adapters;
- Plugins/Appearance management UI;
- security audit, diagnostics, contract, component and E2E coverage;
- `0.2.12` release closure.

### Не входит

- Tool Registry changes except documentation boundaries;
- Linear conversion into a plugin;
- remote marketplace or catalog;
- local zip/package installation;
- arbitrary CSS, fonts, URLs or scripts from manifest;
- arbitrary JavaScript in the main React tree;
- Worker/iframe/Node plugin runtimes;
- plugin commands, Workbench tabs, Inspector aspects and renderers as active
  runtime contributions;
- Git Review extraction;
- per-user Core profile before team/user model exists;
- automatic OS theme switching beyond an explicit future preference.

Manifest reserves future contribution vocabulary, but v1 activates only
`ui.theme`. Unknown or unsupported contribution contracts stay inactive with a
diagnostic.

## Fixed decisions

1. Plugin Registry и Tool Registry — разные bounded contexts.
2. Core владеет installed/enabled/compatible state.
3. Theme selection является client/user preference, не plugin configuration.
4. До user profile preference хранится в versioned `localStorage`.
5. `core.light` встроена в shell и не может быть удалена or disabled.
6. Dark Theme не выбирается автоматически после install/enable.
7. Theme package является data-only и не содержит CSS or executable code.
8. Web применяет только allowlisted semantic tokens.
9. Missing, disabled, invalid or incompatible theme всегда деградирует к
   `core.light`.
10. Monaco and xterm являются theme consumers, а не владельцами независимых
    hardcoded palettes.
11. Numbered SQLite migrations immutable; новый schema slice добавляет только
    migration `13`.
12. Bundled package version независима от release version и начинает с
    `1.0.0`.

## Этап 0. Contract foundation

### 0.1 Protocol vocabulary

Добавить отдельный `uprava-protocol::plugins` module:

```text
PluginId
PluginVersion
PluginManifestVersion
PluginTrustLevel
PluginInstallSource
PluginDesiredState
PluginEffectiveState
PluginCompatibility
PluginPackageSummary
PluginInstallationSummary
PluginContribution
ThemeContributionV1
EffectivePluginSnapshot
```

Не помещать plugin types в tooling module. Optional связь с Tool Registry позже
будет typed reference contribution, а не общая aggregate model.

### 0.2 Manifest identity and hashing

- `plugin_id` и contribution ids имеют bounded namespaced form;
- `version` является SemVer;
- normalized manifest получает deterministic SHA-256 hash;
- package row immutable по `(plugin_id, version, manifest_hash)`;
- `manifest_version = 1` обязателен;
- unknown major manifest or contribution version дает compatibility diagnostic;
- `core.*` and reserved `uprava.*` ids доступны только coordinated bundled
  packages.

### 0.3 Theme contract

`ThemeContributionV1` содержит:

```text
theme_id
label
kind
color_scheme
semantic_tokens
monaco
terminal
```

Allowlist semantic tokens формируется из реального design system. Required
pairs покрывают background/foreground, muted content, borders, focus,
selection, risk and notice states.

Validation ограничивает:

- количество tokens;
- длину ids and labels;
- допустимые color formats;
- отсутствие URL, CSS functions outside a small allowlist, selectors and
  unknown variable references;
- terminal palette size;
- Monaco theme vocabulary;
- контраст critical text/background and focus pairs.

### 0.4 API contract

Минимальные endpoints:

```text
GET  /plugins
GET  /plugins/:plugin_id
POST /plugins/:plugin_id/enable
POST /plugins/:plugin_id/disable
GET  /plugin-contributions?kind=ui.theme
```

Configuration/permission DTO входят в shared contract, но Dark Theme не имеет
custom settings. Permission `ui.theme.contribute` выдается bundled data-only
package при idempotent bootstrap and фиксируется в audit.

### Exit criteria этапа 0

- Rust contracts serialize into Web fixtures;
- manifest and theme validation fixtures include valid, invalid, oversized,
  reserved-id and incompatible cases;
- Tool Registry contracts не импортируют plugin aggregate types;
- docs and fixtures use one vocabulary.

## Этап 1. Core Plugin Registry

### 1.1 Persistence migration 13

Добавить:

```text
plugin_packages
  plugin_id, version, manifest_hash, manifest_version, install_source,
  trust_level, manifest_json, discovered_at

plugin_installations
  plugin_id, active_version, desired_state, effective_state,
  compatibility_json, configuration_revision, installed_at, updated_at,
  last_error_code

plugin_configurations
  plugin_id, revision, values_json, values_hash, updated_at

plugin_permission_grants
  plugin_id, permission_id, decision, granted_at, updated_at
```

Raw manifest сохраняется вместе с normalized indexed fields. Effective state
пересчитывается, а не принимается от Web.

### 1.2 Bundled package bootstrap

Core composition root регистрирует immutable manifest
`uprava.theme-dark@1.0.0` idempotently.

Первый bootstrap:

- package существует в Registry;
- installation создается `disabled`;
- bundled permission grant создается явно;
- theme contribution не входит в effective snapshot до enable;
- повторный bootstrap не меняет user choice and audit history.

### 1.3 Lifecycle and projection

Enable выполняет fresh compatibility and permission check. Disable атомарно
закрывает effective contributions. Projection возвращает только contributions
active compatible plugins.

Нельзя физически удалять bundled package через public v1 API. Unknown plugin
id возвращает typed not-found; stale active version or manifest mismatch —
incompatible/error, а не silent fallback в Core state.

### 1.4 Audit and diagnostics

Audit events:

```text
plugin.discovered
plugin.enabled
plugin.disabled
plugin.compatibility_failed
plugin.activation_failed
```

Metadata содержит ids, versions, manifest hash, actor/origin and outcome. Theme
tokens целиком в audit не копируются.

### Exit criteria этапа 1

- clean DB применяет migrations 1–13;
- state 0.2.11 обновляется migration 13 без изменения migration 12 checksum;
- restart сохраняет desired state;
- enable/disable immediately changes effective projection;
- invalid bundled manifest fails startup явно и без partial rows;
- audit assertions pass.

## Этап 2. Web Extension Host and Theme Service

### 2.1 Host boundary

Добавить provider/service, который:

- bootstrap-ит core contributions;
- загружает effective plugin contribution snapshot;
- валидирует Web DTO через Zod;
- разрешает theme id;
- применяет semantic tokens к controlled root boundary;
- уведомляет Monaco/xterm consumers;
- изолирует invalid contribution и публикует bounded client diagnostic;
- никогда не исполняет code or CSS из manifest.

### 2.2 Appearance preference

Versioned preference:

```text
uprava.appearance.v1:
  version
  selected_theme_id
```

Resolution:

```text
selected theme exists in current effective snapshot
  -> selected theme
else
  -> core.light
```

Preference parse/storage failure не блокирует App Shell. Disabled theme id
можно сохранить как preference intent, но effective state и DOM всегда должны
использовать fallback.

### 2.3 Pre-React bootstrap

До `main.tsx` применить безопасный cached theme marker and `color-scheme`.
Bootstrap не доверяет cached arbitrary tokens: он выбирает только ранее
валидированный bundled theme id/palette version. После Core load Theme Service
подтверждает availability or возвращает `core.light`.

Это предотвращает flash и не превращает localStorage в source of executable
styles.

### 2.4 Theme-safe first-party UI

Заменить raw visual colors вроде `border-black/*`, `text-white` and direct hex
в application components на semantic tokens. Static design-system gate должен
разрешать literal colors только внутри canonical theme definitions and
необходимых third-party adapter boundaries.

Derived tokens вычисляются host/design system, а не каждым component отдельно.

### 2.5 Monaco and xterm adapters

- Monaco получает coordinated light/dark definitions and live theme switch;
- уже открытые editors/diffs обновляются без recreation and content loss;
- xterm palette приходит из Theme Service;
- уже открытый terminal обновляет theme without reconnect;
- terminal ANSI colors сохраняют различимость на обеих темах;
- third-party widget failure не откатывает весь App Shell.

### Exit criteria этапа 2

- theme switches without reload;
- no flash in supported bootstrap flow;
- open Monaco model/draft and terminal socket survive switch;
- invalid/localStorage state falls back safely;
- static raw-color gate passes.

## Этап 3. Plugins and Appearance UI

Добавить `/settings/plugins` с двумя уровнями:

- installed packages and lifecycle;
- Appearance selection among effective themes.

Для Dark Theme показать:

- name, publisher, version and bundled/data-only trust;
- enabled/effective/compatibility state;
- declared permission;
- `ui.theme` contribution;
- enable/disable action;
- selection control only when effective.

Disable выбранной темы требует понятного inline explanation, затем сразу
активирует `core.light`. Нельзя показывать plugin как active до ответа Core.

Management surface не является marketplace и не показывает фиктивные Install
or Update buttons.

### Exit criteria этапа 3

- loading, empty, disabled, active, incompatible and error states покрыты;
- controls keyboard and screen-reader accessible;
- narrow/mobile layout usable;
- no optimistic state contradicts Core projection.

### Web tests

- contribution snapshot validation;

### E2E and visual matrix

```text
desktop light
desktop dark
mobile light
mobile dark
Workbench with Monaco light/dark
Workbench with xterm light/dark
Agent timeline light/dark
Jobs and Inspector light/dark

```

Golden screenshots используют deterministic explicit theme, а не OS default.
Contrast checks покрывают primary/muted text, focus, links, risk, notice,
selected controls and terminal content.

## Completion criteria

Срез завершен, когда одновременно верно:

1. Plugin Registry существует отдельно от Tool Registry in protocol, storage
   and application boundaries.
2. Dark Theme зарегистрирована как package/contribution, а не как hardcoded
   boolean `darkMode`.
3. Core является authority для install/enable/effective state.
4. Web Theme Service применяет только validated semantic tokens.
5. Все основные first-party surfaces работают в light and dark.
6. Monaco and xterm переключаются без потери active state.
7. Disable selected plugin мгновенно дает safe light fallback.
8. Restart, malformed preferences and unavailable Core не создают broken or
   unstyled shell.
9. Contract, Core, Web, accessibility and visual gates проходят.
10. `make c` проходит без ослабления checks.
