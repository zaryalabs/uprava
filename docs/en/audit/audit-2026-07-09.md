# Uprava Unified Architecture and Code Quality Audit

Date: `2026-07-09`

Status: `active`

Audited implementation baseline: `0.1.8`

## Purpose

This document consolidates the current audit of Uprava architecture, code
quality, runtime logic, security boundaries, reliability, Web performance,
testing, CI/CD and deployment posture.

It supersedes no shipped implementation history. In particular, it does not
reopen the issues already fixed in the `0.1.6` unified audit hardening slice.
The remaining valid items from that work are still recorded in
[`../tmp-plans/audit-20260706-deferred-followups.md`](../tmp-plans/audit-20260706-deferred-followups.md).
This report incorporates those deferred items and adds findings confirmed
against the `0.1.8` implementation.

The report is a snapshot and prioritization document, not a release ledger or
a replacement for [`feature-queue.md`](feature-queue.md).

## Scope And Method

The audit covered:

- canonical product and architecture documentation;
- the Core Backend, Node Daemon, shared protocol and logging crates;
- the React Web Control Panel, workspace inspector, Monaco and xterm paths;
- SQLite persistence, commands, events, projections and enrollment;
- control-channel, SSE and terminal reconnect behavior;
- Codex provider execution and local workspace boundaries;
- Docker, systemd, release scripts and GitHub Actions;
- Rust, TypeScript, dependency and production-build quality gates.

The implementation was reviewed against the repository's own architectural
rules and against production Rust, Axum, Tokio and React practices. The audit
did not mutate source files or exercise the live self-hosted deployment.

## Actual Implementation Dependency Map

The intended `Web -> Core -> Node -> workspace/provider` system boundary is
present, but the implementation inside each process is considerably more
coupled:

```text
Web route/component
  -> generic Core API client and manually mirrored TypeScript contracts
  -> local command-availability rules and browser-side event projections

Core Axum handler / control-frame handler
  -> shared AppState
  -> raw SQL, domain-state changes, command dispatch and global broadcasts

Node heartbeat loop
  -> one NodeLocalState copy
  -> state JSON file
  -> control loop with a second NodeLocalState copy
  -> workspace, terminal and provider execution

uprava-protocol
  -> domain states and ids
  -> HTTP DTOs
  -> Core/Node control frames
  -> workspace and terminal DTOs
  -> artifact, agent-projection and UI-block contracts
```

The main dependency problem is therefore not the top-level distributed split.
It is the absence of internal ownership, application-service, persistence and
projection boundaries inside Core, Node and Web.

## Executive Assessment

The system architecture is coherent and worth preserving:

- Core is the control plane;
- Node Daemon is the data plane;
- Web communicates through Core rather than directly with nodes;
- commands and events are explicit, typed and traceable;
- workspace enforcement is implemented on Node;
- durable Core state is separated from local provider/process state.

The code architecture has not yet caught up with that design. Responsibility
boundaries are clear in documentation but remain concentrated in a few large
implementation files. Several high-risk invariants are procedural rather than
enforced by module boundaries, authenticated context, transactions, bounded
channels or supervisor ownership.

Current posture:

| Area | Assessment |
| --- | --- |
| System architecture | Strong direction; Core/Node/Web split is sound |
| Code architecture | Functional but overly concentrated and weakly modularized |
| Local code readability | Generally good naming and explicit error handling |
| Reliability | Good baseline tests, but important transaction/reconnect gaps remain |
| Security | Appropriate only for an explicitly trusted controlled deployment |
| Team/cloud readiness | Blocked by trust-boundary, isolation and authorization gaps |
| Web performance | Functional, but initial bundle size is unacceptable |
| Delivery/tooling | Strong local gate; CI, E2E and image hardening need follow-through |

Uprava should remain classified as a trusted single-user or controlled
development deployment until the P0 and P1 findings below are resolved.

## Existing Strengths

The audit confirmed several implementation strengths:

- Web does not bypass Core to access Node workspaces.
- Workspace roots are canonicalized and checked against
  `UPRAVA_NODE_WORKSPACES`.
- File writes use no-follow and post-open validation where supported.
- Workspace command output is bounded during execution.
- Node local state writes are atomic at file-replacement level and use private
  permissions on Unix.
- Commands and events carry stable identifiers, correlation metadata and
  addressable references.
- Node command replay persists terminal result payloads for current workspace
  operations.
- Session event streaming uses a session projection cursor across raw event
  scopes.
- Core has structured safe API errors, request timeout and request-body limits.
- Production Rust avoids panic-based request handling and has meaningful
  negative-path coverage.
- The local quality gate covers formatting, Clippy, TypeScript, Rust/Web tests,
  Rust dependency audit, license/source policy and production Web build.

These strengths should be preserved while refactoring. The recommended work is
primarily about making existing invariants explicit and independently testable.

## Consolidated Findings

### P0-1: Inbound Node Frames Are Authenticated But Not Object-Authorized

Core authenticates the control WebSocket using a Node id and bearer credential,
and validates the `Hello` Node id. After that, command acknowledgements,
command results, event envelopes and terminal frames are processed without
checking that the referenced command, runtime, session, placement or terminal
belongs to the authenticated Node.

Evidence:

- `crates/uprava-server/src/lib.rs:1595` - control-frame dispatch;
- `crates/uprava-server/src/lib.rs:1663` - command acknowledgement/result;
- `crates/uprava-server/src/lib.rs:1694` - event batches;
- `crates/uprava-server/src/lib.rs:1717` - terminal output/status;
- `crates/uprava-server/src/lib.rs:3739` - event acceptance has no authenticated
  Node context.

Impact:

- a compromised enrolled Node can mutate another Node's command or projection
  state when it knows an identifier;
- terminal output can be injected into a terminal owned by another placement;
- command state can be set without enforcing a legal transition;
- Core's control-plane ownership guarantee is not enforced at the transport
  boundary.

Recommendation:

1. Introduce an authenticated `NodeContext` for every control connection.
2. Validate command target ownership before accepting ACK/result frames.
3. Validate event Node, runtime, session, placement, command and actor
   relationships before persistence.
4. Validate terminal ownership through its placement and Node.
5. Reject invalid state transitions and non-terminal result states.
6. Add frame count, event count, string and payload byte limits.
7. Add cross-node forgery regression tests.

### P0-2: Event Persistence And Projection Are Not Atomic

Core inserts an event before applying its runtime, session, turn, approval and
message projections. These steps use separate database operations rather than
one transaction.

If event insertion succeeds and a later projection fails, Node does not receive
an ACK and retries the event. Core then sees the existing `event_id`, treats it
as an accepted duplicate and returns without repairing the missing projection.

Evidence:

- `crates/uprava-server/src/lib.rs:3739` - duplicate check and event workflow;
- `crates/uprava-server/src/lib.rs:3800` - insert before projection;
- `crates/uprava-server/src/lib.rs:3893` - multi-step projection application.

Impact:

- an event can be durable while the corresponding session/runtime/message
  state is permanently missing;
- transient database or serialization failures can create silent read-model
  corruption;
- Node retry semantics cannot repair the state.

Recommendation:

1. Allocate the session projection cursor, insert the event and apply every
   projection inside one SQLx transaction.
2. Publish to the in-memory SSE bus only after commit.
3. Make duplicate acceptance conditional on a fully projected/committed inbox
   record, or make projection replay explicitly idempotent.
4. Add a fault-injection test that fails after event insertion and proves the
   retry repairs or rolls back the operation.

### P0-3: Codex Runs With Sandbox And Approvals Disabled

Both new and resumed Codex executions pass
`--dangerously-bypass-approvals-and-sandbox`.

Evidence:

- `crates/uprava-node/src/main.rs:3883` - `codex exec` launch;
- `crates/uprava-node/src/main.rs:3926` - `codex exec resume` launch.

The workspace allow-list validates the process working directory, but it does
not restrict what the provider process can access using the Node Unix user's
permissions. The implementation also parses provider approval events while the
default launch mode disables the primary approval/sandbox boundary.

Recommendation:

1. Use workspace-write sandboxing as the default provider policy.
2. Put unrestricted execution behind an explicit development-only setting.
3. Persist the effective execution policy in command/event metadata.
4. Display unrestricted execution as a hard warning in Web.
5. Integrate real provider approvals with Core before describing the execution
   path as hardened.
6. Keep host Unix user, ACL and workspace-root controls as defense in depth,
   not as the only sandbox.

### P1-1: Slow Commands Block The Entire Node Control Plane

Node awaits command execution directly inside the sole WebSocket receive loop.
A Codex turn or a workspace command lasting up to 120 seconds blocks:

- pings and protocol liveness;
- event-batch acknowledgements;
- terminal input, resize and close frames;
- unrelated workspace commands;
- commands for other runtimes on the same Node.

Evidence: `crates/uprava-node/src/main.rs:702`.

Recommendation:

- keep socket read/write tasks transport-only;
- route commands through a bounded dispatcher;
- serialize operations per runtime or terminal where ordering matters;
- permit bounded concurrency across independent runtimes/workspaces;
- preserve one authoritative state owner for dedupe, sequences and outbox;
- use cancellation tokens and tracked task handles.

### P1-2: Control Reconnect Can Remove A Newer Core Channel

Core replaces `control_channels[node_id]` when a connection arrives. When an
older socket later closes, it removes the map entry unconditionally, which can
delete the newer channel.

Evidence: `crates/uprava-server/src/lib.rs:1549`.

Recommendation:

- identify connections with a generation/token;
- remove only when the stored generation matches the closing socket;
- explicitly close or supersede the old connection;
- test overlapping connect/disconnect ordering.

### P1-3: PTY Ownership Is Scoped To A Transient Control Connection

`WorkspaceTerminalManager` is created inside one control-channel connection.
On reconnect, Node loses the terminal registry while Core can continue to show
those terminals as running.

When the manager is dropped, terminal control senders close. The PTY task exits
its control loop and then waits for the child without first killing it; a shell
can remain alive and inaccessible.

Evidence:

- `crates/uprava-node/src/main.rs:702` - per-connection manager;
- `crates/uprava-node/src/main.rs:1026` - detached terminal task;
- `crates/uprava-node/src/main.rs:1105` - terminal lifecycle and child wait.

Recommendation:

- move terminal supervision to daemon scope;
- keep terminal metadata, replay and task handles in one supervisor;
- define detach, reconnect, reattach and daemon-shutdown semantics;
- kill and reap children on cancellation with a bounded shutdown timeout;
- reconcile Core terminal projections after reconnect.

### P1-4: Control Queues And Provider Output Are Unbounded

Core and Node use unbounded control channels. Codex execution accumulates raw
stdout, stderr, activity events and live outbox events without a global byte or
event limit during the process run. Outbox retention is applied only after the
provider returns.

Evidence:

- `crates/uprava-server/src/lib.rs:1551` - unbounded Core control channel;
- `crates/uprava-node/src/main.rs:671` - unbounded Node outbound channel;
- `crates/uprava-node/src/main.rs:1272` - live outbox append;
- `crates/uprava-node/src/main.rs:4008` - unbounded provider accumulation.

Recommendation:

- replace unbounded channels with bounded queues and explicit overflow policy;
- cap line length, total stdout/stderr bytes, activity-event count and raw JSON;
- apply outbox retention while streaming, not only at completion;
- batch events and propagate backpressure;
- emit explicit truncation/gap events rather than silently dropping evidence.

### P1-5: Enrollment And Aggregate Creation Lack Units Of Work

Enrollment claim reads state, creates a Node, replaces capabilities and updates
the enrollment in separate operations. Concurrent claims can both observe an
approved enrollment and create multiple valid Nodes.

Session creation and turn submission similarly persist several pieces of one
aggregate without a transaction. A partial failure can leave a session without
its runtime/command or a command without its turn/message.

Evidence:

- `crates/uprava-server/src/lib.rs:4287` - enrollment claim;
- `crates/uprava-server/src/lib.rs:2938` - session creation;
- `crates/uprava-server/src/lib.rs:3201` - turn submission.

Recommendation:

- create each aggregate in one transaction;
- claim enrollment with compare-and-set semantics such as
  `update ... where status = 'approved' returning ...`;
- allocate turn indexes and event projection cursors transactionally;
- add concurrent claim/session/turn regression tests.

### P1-6: Session SSE Reconnects After Every Applied Event

The session stream effect depends on `session.data.events`. Applying an SSE
event changes the array, tears down the current `EventSource` and opens a new
connection using the new cursor.

Evidence: `apps/web/src/features/sessions/SessionRoute.tsx:53`.

Impact:

- connection churn for every provider activity event;
- repeated Core catch-up queries;
- avoidable reconnect races and load during active turns.

Recommendation:

- keep one stream per session id;
- store the current cursor in a ref or dedicated stream store;
- reconnect only on explicit reload, terminal error or session change;
- test that multiple received events do not create multiple `EventSource`
  instances.

### P1-7: Initial Web Bundle Is Excessive

The production build emits an initial JavaScript chunk of approximately
`4,464.66 kB` minified and `1,166.58 kB` gzip, plus Monaco workers including a
TypeScript worker of approximately `7 MB`.

All routes are statically imported, and the placement/workspace route eagerly
loads Monaco, xterm and their styles.

Evidence:

- `apps/web/src/app/App.tsx:3` - eager route imports;
- `apps/web/src/features/workspace-inspector/WorkspaceInspector.tsx:1` - eager
  Monaco/xterm imports.

Recommendation:

- lazy-load routes;
- lazy-load editor, diff and terminal surfaces independently;
- load only the Monaco languages/workers required by the opened file;
- add a bundle budget to CI;
- consider a lighter read-only renderer when editing is not active.

### P1-8: Node State Has Two Concurrent Owners

The Node heartbeat loop loads and owns a `NodeLocalState`, then starts the
control loop with a clone of that state. The control task mutates and persists
its copy, while the heartbeat loop periodically reloads the JSON file to observe
those changes. On authentication failure the heartbeat loop can abort the
control task and write its own copy.

Evidence:

- `crates/uprava-node/src/main.rs:77` - heartbeat-loop state owner;
- `crates/uprava-node/src/main.rs:115` - disk reload used to observe control
  state;
- `crates/uprava-node/src/main.rs:132` - control loop receives a cloned state;
- `crates/uprava-node/src/main.rs:630` - second long-lived state owner.

Impact:

- the state file acts as inter-task communication rather than persistence only;
- stale heartbeat data and lost updates are possible;
- aborting the control task does not prove that a blocking save has stopped;
- runtime, credential and outbox ordering cannot be owned or tested in one
  place.

Recommendation:

1. Introduce one daemon-level `NodeSupervisor` or state-store actor.
2. Route heartbeat snapshots, command dedupe, sequences, outbox and runtime
   metadata through that owner.
3. Keep disk writes behind the owner and move blocking serialization/fsync off
   Tokio worker threads.
4. Add concurrent heartbeat/control/shutdown tests that assert monotonic state.

### P1-9: Reliable Request/Response Paths Use Lossy Global Broadcasts

Workspace HTTP operations subscribe to a global command-result broadcast,
dispatch a command and wait for a matching result. Terminal clients similarly
subscribe to one global terminal broadcast and filter frames by terminal id.

Evidence:

- `crates/uprava-server/src/lib.rs:2796` - workspace command dispatch and wait;
- `crates/uprava-server/src/lib.rs:2870` - lagged command results are ignored;
- `crates/uprava-server/src/lib.rs:1801` - global terminal broadcast relay;
- `crates/uprava-server/src/lib.rs:1859` - lagged terminal output is ignored.

Impact:

- a durable result can exist in SQLite while the waiting HTTP request times out;
- every waiter receives unrelated results or terminal frames;
- terminal output can be silently lost without a sequence-gap notification;
- synchronous HTTP requests remain coupled to a 5-120 second distributed
  command lifecycle.

Recommendation:

- use per-command `oneshot` waiters with a durable result lookup fallback;
- use per-terminal channels or a terminal supervisor registry;
- treat terminal sequence gaps as explicit replay/reload events;
- decide which workspace operations are synchronous RPC and which should expose
  an asynchronous command resource.

### P1-10: Project, Placement And Workspace Identity Rules Conflict

Heartbeat discovery derives both project and placement ids from the same
`(node_id, workspace_path)` tuple. Explicit placement validation instead creates
new random project and placement ids. The database has no uniqueness constraint
on `(node_id, workspace_path)`.

At the same time, `Workspace` and `Placement` references both carry a placement
id, and the Web routes `/workspaces/:id` and `/placements/:id` to the same
component.

Evidence:

- `crates/uprava-server/src/lib.rs:2085` - explicit placement creation;
- `crates/uprava-server/src/lib.rs:4617` - heartbeat-derived placement/project;
- `crates/uprava-server/src/lib.rs:6661` - project and placement ids use the
  same identity inputs;
- `crates/uprava-server/src/lib.rs:7263` - placement schema lacks node/path
  uniqueness;
- `apps/web/src/app/App.tsx:28` - placement/workspace route aliases.

Impact:

- one physical workspace can produce several logical placements and projects;
- the same repository on different Nodes cannot naturally be represented as
  one Project with several placements;
- Project is currently close to a one-to-one mirror of Placement;
- Workspace is a second name for Placement rather than an independently defined
  domain entity or an explicitly documented UI surface.

Recommendation:

1. Record an ADR defining Project, Workspace and Placement identity/lifecycle.
2. Decide whether `(node_id, canonical_workspace_path)` is unique or whether
   aliases are first-class.
3. Give Project an identity independent of Node placement when several copies
   of one repository should belong to the same project.
4. Treat Workspace as either a real entity or an explicitly named view over a
   Placement, not both.

### P2-1: Code Boundaries Exist In Documentation More Than In Modules

Main concentration points at the audited baseline:

| File | Approximate lines | Mixed responsibilities |
| --- | ---: | --- |
| `crates/uprava-server/src/lib.rs` | 11,850 | config, HTTP, auth, control, commands, events, projections, SQL, migrations, tests |
| `crates/uprava-node/src/main.rs` | 7,287 | config, enrollment, state, control, workspace, PTY, providers, outbox, tests |
| `crates/uprava-protocol/src/lib.rs` | 1,621 | all public protocol contracts and tests |
| `apps/web/src/features/workspace-inspector/WorkspaceInspector.tsx` | 1,364 | data, drafts, tree, Monaco, xterm, commands, diff, history |
| `apps/web/src/workbench/inspector/InspectorStack.tsx` | 933 | ref resolution, data shaping and many inspector presentations |

Local function naming and error handling are generally clear. The cleanliness
problem is structural: changes cross too many responsibilities, and important
invariants remain implicit in procedural code.

Recommendation: perform the mechanical module split defined later in this
document after the P0/P1 behavior fixes. Avoid broad framework-style traits
until concrete alternate implementations or test seams justify them.

### P2-2: Rust And Web Protocol Contracts Can Drift Silently

The Web client casts successful HTTP JSON directly to generic TypeScript types.
SSE validates JSON syntax but not the envelope shape. Rust and TypeScript
contracts are maintained manually.

Confirmed drift already exists in test fixtures: the E2E fixture uses
`security.mode = "controlled_dev"`, while Rust and TypeScript declare
`"hardened"`.

Evidence:

- `apps/web/src/shared/api/http-client.ts:62`;
- `apps/web/src/shared/api/sse-client.ts:5`;
- `apps/web/src/shared/protocol/types.ts:86`;
- `apps/web/e2e/golden.spec.ts:65`.

Recommendation:

1. Generate TypeScript and/or JSON Schema from the Rust source contract, or
   adopt OpenAPI/JSON Schema as the shared source.
2. Validate high-risk HTTP, SSE and terminal payloads at runtime.
3. Generate checked fixtures from Rust for Web contract tests.
4. Treat malformed payloads as typed observable errors.

### P2-3: Node Local State Is A Growing Multi-Concern Document

One JSON state file contains enrollment identity, credential, command status,
typed result replay, runtime sequences, outbox, provider/workspace metadata,
runtime states, transcripts, resume refs and placement sequences.

The outbox is bounded, but several maps grow without retention. The entire
state is synchronously serialized, flushed and fsynced after changes.

Evidence: `crates/uprava-node/src/main.rs:230`.

Recommendation:

- introduce one state-store actor/API;
- separate identity, command cache, outbox and runtime metadata logically;
- define retention/compaction for completed commands and stopped runtimes;
- keep atomic persistence, but avoid full-document rewrites on every event;
- consider SQLite or an append/checkpoint model when the state outgrows a small
  controlled deployment.

### P2-4: Public Endpoints And Local Auth Need Hardening

`/client/logs` and Node enrollment request/claim routes are public by design,
but lack robust peer-aware quotas. Client logs append to an unrotated file.
Local passwords use a salted single SHA-256 operation instead of a password
KDF. Authentication failure counters are in-memory and coarse-grained.

Evidence:

- `crates/uprava-server/src/lib.rs:376` - public routes;
- `crates/uprava-server/src/lib.rs:1090` - client log append;
- `crates/uprava-server/src/lib.rs:6680` - password hashing.

Recommendation:

- use Argon2id or scrypt with a versioned stored format;
- add peer-aware and global rate/concurrency limits;
- bound pending enrollment count and request frequency;
- authenticate client logs after login or issue a narrowly scoped pre-auth
  logging policy;
- rotate/cap local logs and audit-event retention;
- do not label the current profile simply as hardened.

### P2-5: Workspace Drafts Can Be Discarded Silently

Workspace file query updates replace editor state even when the buffer is
dirty. Refreshing or selecting another path can overwrite an unsaved edit.

Evidence: `apps/web/src/features/workspace-inspector/WorkspaceInspector.tsx:153`.

Recommendation:

- store drafts by placement/path;
- confirm before destructive refresh/navigation;
- update a buffer from server data only when it is clean;
- add draft preservation and conflict tests.

### P2-6: Deployment Images And CI Need Production Hardening

Current concerns:

- Core and Web containers run as root;
- the Web image copies all `node_modules` and serves through `vite preview`;
- Docker builds use mutable base-image tags;
- the Web image uses `npm install` instead of `npm ci`;
- Rust image builds do not pass `--locked`;
- Core and Node Dockerfiles each build both binaries;
- CI grants `packages: write` at workflow scope, including the check job;
- pull-request code runs on a persistent self-hosted runner;
- CI uses Rust `1.90` while the declared MSRV is `1.88`;
- Playwright E2E and npm dependency audit are outside the required CI gate.

Evidence:

- `Dockerfile.core`;
- `Dockerfile.node`;
- `apps/web/Dockerfile`;
- `.github/workflows/ci.yml`;
- `Cargo.toml:15`;
- `Makefile:59` and `Makefile:414`.

Recommendation:

- use non-root runtime users and read-only filesystems where practical;
- serve static Web assets with a production static server;
- use `npm ci`, `cargo build --locked` and digest-pinned base images;
- split binary builds and improve layer caching;
- scope workflow permissions per job;
- isolate/clean self-hosted runner workspaces and restrict untrusted PR code;
- add an explicit Rust `1.88` MSRV job plus a current stable job;
- run deterministic Playwright golden tests and npm audit policy in CI.

The production npm audit performed during this review reported three findings:
one moderate DOMPurify issue through `monaco-editor` and two low-severity
findings including `esbuild`. Reachability should be assessed before choosing a
compatible Monaco upgrade or pin.

### P2-7: Logging And Observability Are Still Synchronous/Incomplete

`uprava-logging` writes and flushes through a standard `Mutex<File>` from the
tracing writer path. High-volume provider events can synchronously block Tokio
worker threads. File rotation and the metrics/OTLP position documented in
[`deployment-observability.md`](deployment-observability.md) are not yet part of
the implementation.

Evidence: `crates/uprava-logging/src/lib.rs:23`.

Recommendation:

- prefer non-blocking tracing writers with bounded loss policy;
- rely on stdout/stderr and platform rotation in containers;
- add explicit file rotation/retention for host logs;
- implement low-cardinality command, event, queue, reconnect, outbox and
  terminal metrics;
- attach request/correlation ids through middleware spans and response headers.

### P2-8: The Shared Protocol Mixes Domain, Transport And Presentation

`uprava-protocol` contains domain states and identifiers, HTTP request/response
DTOs, control frames, workspace and terminal contracts, references, artifact
projections and the generic `UiBlock` presentation model. Node therefore
depends on Web-facing contracts that it does not consume.

Command and event envelopes also repeat identity through `scope_ref`, optional
Node/runtime/session/turn ids and reference lists without a type-level rule that
the values agree. Their payload remains untyped JSON for built-in commands and
events.

Evidence:

- `crates/uprava-protocol/src/lib.rs:1` - all contracts in one module;
- `crates/uprava-protocol/src/lib.rs:318` - generic command payload;
- `crates/uprava-protocol/src/lib.rs:336` - redundant event identity and generic
  payload;
- `crates/uprava-protocol/src/lib.rs:1160` - UI block in the shared Rust
  protocol.

Recommendation:

- split the crate into internal `domain`, `api`, `control`, `workspace`,
  `references` and `projections` modules before considering more crates;
- define typed payloads for built-in commands/events and retain an explicit
  extension JSON variant;
- validate envelope identity relationships at construction and ingress;
- move `UiBlock` to Web until Core or plugins actually publish it as a protocol
  object.

### P2-9: State And Command Policy Are Reimplemented Across Layers

The mapping from events to runtime/session states exists in Core event
projection, Node local projection and Web cache projection. Command
availability exists in Core request validation, Core Agent Projection and the
Web command registry.

Evidence:

- `crates/uprava-server/src/lib.rs:3893` - Core event projection;
- `crates/uprava-server/src/lib.rs:4803` - Core command validation;
- `crates/uprava-server/src/lib.rs:5756` - Core available-command projection;
- `crates/uprava-node/src/main.rs:1814` - Node runtime projection;
- `apps/web/src/workbench/projection/apply-session-event.ts:16` - Web event
  projection;
- `apps/web/src/workbench/commands/registry.ts:76` - Web command policy.

Impact:

- a new state/event/command requires synchronized edits in several files and
  languages;
- Web can enable an action rejected by Core or display a projection different
  from the durable read model;
- the Agent Projection publishes `available_commands`, but Web mostly
  recomputes the list and uses the server list only as display data.

Recommendation:

- make legal state transitions and command predicates pure Core domain rules;
- publish authoritative action capability/reason data for Web;
- keep Node projection limited to local operational ownership;
- use generated fixtures/contract tests to prove equivalent incremental and
  durable projections where a browser-side projection remains useful.

### P2-10: Persistence Stores Duplicate Representations Without Authority Rules

Capabilities are stored both in `nodes.capabilities_json` and normalized
`node_capabilities`. Commands and events store normalized columns, separate JSON
columns and a full serialized envelope. Session/runtime links and provider names
are stored on both sides of their relationship.

Unknown database state strings silently fall back to valid initial states such
as `Created`, `Starting` or `Recorded`. Inline migrations use `create if not
exists` plus optional `alter table` calls, while the externally reported schema
version remains `1`.

Evidence:

- `crates/uprava-server/src/lib.rs:4677` - normalized capability replacement;
- `crates/uprava-server/src/lib.rs:6911` - permissive state parsers;
- `crates/uprava-server/src/lib.rs:7171` - inline migrations;
- `crates/uprava-server/src/lib.rs:7285` - reciprocal session/runtime data;
- `crates/uprava-server/src/lib.rs:7356` - normalized plus full JSON
  command/event storage.

Recommendation:

1. Define the authoritative representation and rebuild policy for each
   duplicated snapshot/index.
2. Update duplicated representations in one transaction or remove the unused
   copy.
3. Return an observable corruption/compatibility error for unknown persisted
   states.
4. Move to numbered SQLx migrations with checksums and a meaningful schema
   version.

### P2-11: The Provider Boundary Leaks Node Storage And Transport

`RuntimeManager` currently dispatches only to `Codex` or `Unsupported`, while
the Codex adapter receives Node sequence maps, transcript maps, resume maps and
an optional live control-channel sink. Several methods suppress
`too_many_arguments`, and a live event send failure is converted into provider
execution failure.

Evidence:

- `crates/uprava-node/src/main.rs:3351` - two-variant runtime manager;
- `crates/uprava-node/src/main.rs:3443` - adapter receives Node storage maps;
- `crates/uprava-node/src/main.rs:3639` - argument-count suppression;
- `crates/uprava-node/src/main.rs:4123` - transport failure changes provider
  result.

Recommendation:

- keep a concrete Codex provider until a second real provider establishes a
  useful shared interface;
- pass a bounded `ProviderExecutionContext` and return/stream typed provider
  outcomes through a daemon-owned sink;
- make transport reconnect independent from local provider success;
- give a runtime supervisor ownership of process handles, cancellation,
  transcript and resume state.

### P2-12: Artifact And Agent Projections Have Misleading Identity Semantics

Artifact Tree is generated from messages and events on every request and assigns
new random artifact ids each time. It is therefore a trace/read-model tree, not
a stable artifact aggregate. Agent Projection repeats session/runtime/placement
data, and Web constructs a synthetic `SessionSummary` with a hardcoded title and
active state to reuse command-registry APIs.

Evidence:

- `crates/uprava-server/src/lib.rs:5336` - request-time artifact tree;
- `crates/uprava-server/src/lib.rs:5344` - random child artifact ids;
- `crates/uprava-server/src/lib.rs:5384` - request-time Agent Projection;
- `apps/web/src/features/agent-projection/AgentProjectionPanel.tsx:107` -
  synthetic session summary.

Impact:

- saved artifact references and React keys are unstable across refetches;
- the name `ArtifactTree` promises persistence and identity that do not exist;
- view models are converted back into fabricated domain models.

Recommendation:

- use deterministic projection ids and name the surface as a trace projection,
  or introduce a persisted Artifact aggregate;
- make Agent Projection actions consume their own typed context instead of a
  fabricated `SessionSummary`;
- avoid repeating full session data when stable references or focused action
  capabilities are sufficient.

### P2-13: Web Workbench Abstractions Combine Unrelated Responsibilities

The command registry combines remote domain actions, local clipboard commands,
navigation and cache-invalidation callbacks behind one optional property-bag
context. Timeline, Inspector, event projection and terminal code each contain
independent JSON/ref parsers. Inspector references are stored as full encoded
JSON in the URL and validated only by checking for a string `kind`.

Evidence:

- `apps/web/src/workbench/commands/registry.ts:28` - mixed command taxonomy;
- `apps/web/src/workbench/commands/registry.ts:48` - optional context bag;
- `apps/web/src/workbench/references/refs.ts:9` - full JSON URL encoding;
- `apps/web/src/workbench/references/refs.ts:246` - weak reference validation;
- `apps/web/src/features/sessions/timeline-blocks.ts:21` - repeated view-model
  reconstruction.

Recommendation:

- separate server actions, navigation commands and local UI operations;
- use discriminated per-command context types;
- centralize generated/runtime-validated payload and reference decoders;
- encode stable short reference keys in URLs rather than arbitrary full JSON;
- memoize timeline projection and avoid rebuilding all blocks on unrelated
  renders.

### P2-14: Speculative Contracts And Dependencies Add YAGNI Cost

Several terminal request DTOs and future reference variants have no production
consumer. `UiBlock` is shared with Rust before the server publishes blocks.
The Web package declares React Table, React Hook Form, Zod and the xterm search
addon without using them. Node and Server also declare direct logging-related
dependencies that are no longer needed after the logging helper was extracted.

Recommendation:

- add a protocol type only with a producer, consumer and contract test;
- remove unused dependencies and re-add them with the feature that needs them;
- do not expand plugin/tool/block taxonomies ahead of a vertical scenario;
- use Zod or generated validators if it remains installed, rather than keeping
  parallel handwritten guards.

### P3: Incremental Code Quality Improvements

- Move inline SQL migrations to numbered SQLx migration files with checksums.
- Add workspace lint policy for important Clippy and Rust lints.
- Add crate/module documentation to `uprava-protocol` and document public
  invariants, errors and compatibility expectations.
- Align the direct `tokio-tungstenite` dependency with Axum's resolved stack
  where compatible to reduce duplicated networking dependencies.
- Add documentation link/style validation; current docs checks verify presence
  only.
- Keep comments focused on invariants and design reasons; move durable behavior
  into tests and design documents.

Architecture rules for the split:

- Axum handlers extract/validate transport input and invoke one application
  use case.
- Application services own units of work and orchestration.
- Domain modules own legal state transitions and invariants without Axum/SQL.
- Persistence modules own SQL and transaction-scoped operations.
- Control transport never directly mutates projections without authorization.
- Node supervisors own long-lived mutable resources and cancellation.
- Use concrete store/service types first; introduce traits only for real
  alternate implementations or useful test seams.

## Remediation Sequence

### Slice A: Control-Plane Integrity

1. Bind inbound Node frames to authenticated ownership.
2. Add legal command transition validation.
3. Make event acceptance and projection transactional.
4. Make enrollment claim a transactional compare-and-set.
5. Add cross-node, partial-projection and concurrent-claim tests.

Completion gate: no authenticated Node can affect another Node's objects, and
an acknowledged event cannot exist without its required projections.

### Slice B: Node Execution And Isolation

1. Replace unrestricted Codex default with explicit execution policies.
2. Separate socket I/O, state ownership and command execution.
3. Add bounded queues and provider output caps.
4. Add runtime-scoped ordering and bounded cross-runtime concurrency.
5. Persist effective sandbox/approval posture in trace.

Completion gate: long provider work does not block terminal/control traffic and
cannot exhaust Node memory through unbounded output.

### Slice C: Terminal And Reconnect Reliability

1. Add connection generations in Core.
2. Move PTY supervision outside the control connection.
3. Add cancellation, child reaping and terminal reconciliation.
4. Add explicit terminal output gap/replay semantics.
5. Test control reconnect with active terminals and in-flight commands.

Completion gate: reconnect neither loses the active control channel nor leaves
inaccessible shell processes.

### Slice D: Protocol And Web Runtime

1. Fix the SSE effect lifetime.
2. Establish generated/shared protocol contracts.
3. Add runtime validation at HTTP/SSE/WebSocket boundaries.
4. Preserve workspace drafts across refresh/navigation.
5. Lazy-load routes and heavy workspace renderers.
6. Enforce a Web bundle budget.

Completion gate: protocol drift fails tests/build, one session uses one stable
stream, and the initial bundle excludes Monaco/xterm.

### Slice E: Mechanical Module Split

1. Split Core by transport, application, domain, control and persistence.
2. Split Node by supervisors, workspace, provider and state/outbox ownership.
3. Split Web workspace concerns into hooks and lazy feature components.
4. Move focused tests next to the module they protect.
5. Keep public interfaces narrow and behavior changes minimal.

Completion gate: command lifecycle, event ingestion, Node state, PTY lifecycle
and Web protocol handling can each be reviewed and tested without scanning a
multi-thousand-line root file.

### Slice F: Delivery And Operability

1. Harden images and static Web serving.
2. Align MSRV and CI toolchains.
3. Add deterministic E2E and npm audit policy to CI.
4. Add non-blocking logs, retention and minimum metrics.
5. Add backup/restore verification and automatic deploy rollback policy.

Completion gate: builds are reproducible, runtime containers are least
privilege, and release health failures produce a safe rollback path.

For every remediation slice:

1. Add a focused regression test for each corrected failure mode.
2. Preserve existing Core/Node responsibility boundaries.
3. Avoid unrelated refactors inside correctness/security fixes.
4. Run `make c` before handoff.
5. Run relevant Playwright or real-profile smoke tests when Web, terminal,
   provider or reconnect behavior changes.
6. Re-evaluate the SemVer implementation baseline after a completed large
   work block.

## Final Position

Uprava does not need a conceptual rewrite. Its Core/Node/Web architecture is a
good foundation. The next work should make the code enforce that architecture:
authenticated object ownership in Core, transactional durable state,
supervisor-owned resources on Node, bounded asynchronous channels, validated
shared protocol contracts and smaller independently testable modules.

Adding Tool Registry, Plugin Registry, more providers or multi-user/cloud
features before these foundations are repaired would multiply the current
coupling and trust-boundary risks. The P0/P1 slices should therefore precede
major feature expansion.
