# Uprava V01

Status: `draft`

This document defines the first usable product version of Uprava.

`V01` is not a phase in a long calendar roadmap. It is the first coherent
product cut: what a user can start, open, control, and return to in the first
version.

## Product Cut

V01 is **Distributed Agent Control Panel**.

The user can run Uprava as a web control panel for persistent agent sessions on
one or more registered nodes. The product is intentionally close to chat in its
first visible workflow, but it is not only chat: node placement, project binding,
session lifecycle, runtime state, and event history are first-class product
objects.

V01 validates the first product thesis:

```text
Persistent Runtime + Node Daemon + Core UI
turn agent chat into a distributed control surface.
```

The first version should feel like a small control panel for a distributed agent
system, not like a local chat wrapper and not yet like a full developer
workbench.

## Trust Boundary

V01 is a controlled development deployment, not a production security release.

V01 includes local web auth, CSRF checks, explicit node enrollment, hashed node
credentials, revoke/rotate basics and best-effort guardrails needed to avoid
accidental misuse. Formal permissions, stronger audit policy, secrets handling
and multi-user security remain post-V01 work.

## First-Run Experience

The first usable version should let a user:

1. Start Core Backend and Web Control Panel.
2. Pair one or more local or explicitly trusted remote Node Daemons.
3. See a sidebar tree shaped as `Nodes -> Projects/Workspaces -> Sessions`.
4. See node status, heartbeat age, basic capabilities, active sessions, and
   degraded/offline state.
5. Register or open a project/workspace binding on a selected node.
6. Start a Core-managed Codex-backed session for that project through an Agent
   Provider Adapter.
7. Attach to the session from the Web Control Panel.
8. Use chat as the first and primary session surface.
9. Receive assistant output and runtime status back into the session view.
10. Approve, interrupt, stop, detach from, reattach to, resume, or return to the
    same session later when provider support allows it.
11. Reload the UI and still see the nodes, project bindings, sessions, messages,
    lifecycle state, and recent event history.
12. Understand where the session is running and which node/project owns it.

## Scope

### Core Backend

- Core as the control plane.
- Web Control Panel served by or deployed with Core.
- SQLite-backed local/single-user persistence.
- Node registry and enrollment state.
- Node heartbeat state and basic capability snapshots.
- Project/workspace registry as metadata and placement context.
- Agent session/thread registry.
- Runtime instance registry.
- Message history for interactive sessions.
- Event log with ordered session/runtime lifecycle events.
- Basic status model for node, project/workspace, runtime, and session.
- Core-routed commands to Node Daemon.
- Idempotent command envelope for session/runtime commands.
- Provider identity and resume references when available.
- Versioned Core-to-Node API and event envelopes.

### Node Daemon

- Node enrollment and outbound registration with Core.
- Development node identity/enrollment metadata.
- Heartbeat with basic health and capability snapshot.
- Outbound control/stream channel for active work.
- Workspace root registration or explicit workspace binding validation.
- Runtime adapter lifecycle, including provider process ownership where the
  adapter supports it.
- Codex provider adapter as the first production adapter.
- Normalized provider input, output, lifecycle events, approvals, warnings, and
  errors.
- Start, attach, detach, interrupt, stop, and resume behavior where provider
  support allows it.
- Local resource warnings and runtime health reporting.
- Best-effort local runtime/workspace guardrails needed for trusted V01 use.

### Runtime And Agent Session

- Persistent Runtime as the default runtime strategy.
- Interactive developer session as the default work contract.
- Codex-backed runtime path on Node Daemon through the V01 exec/resume adapter.
- Minimal Agent Provider Adapter boundary.
- Start, resume, send turn, approve, interrupt, stop, attach, and detach.
- Provider resume reference when available.
- For V01, Codex continuity uses `codex exec` and `codex exec resume` where a
  provider session id is available; provider-native live process ownership,
  streaming output, and real interrupt escalation are post-V01 work.
- Managed runtime lifetime instead of an unbounded process promise.
- Normalized events for messages, lifecycle transitions, approvals, errors, and
  warnings.

### Distributed Runtime Coordination

- `Nodes -> Projects/Workspaces -> Sessions` inventory view.
- Explicit user choice of node/project placement.
- Runtime session as the coordination unit.
- Core-routed commands to Node Daemon.
- Ordered events per runtime/session stream.
- Stale/offline node behavior.
- Minimum resource, conflict, and disconnected-state warning badges.
- Explicit user override recorded as an event where needed.

### Web Control Panel

- Application shell with:
  - left navigation tree for nodes, projects/workspaces, and sessions;
  - main activity surface;
  - session header/breadcrumb with node, project, runtime, and status context;
  - reserved secondary/detail area for future workspace, artifact, trace, and
    configuration panels.
- Nodes surface with status, heartbeat age, basic capabilities, active sessions,
  pair/revoke flow, and diagnostics.
- Project/workspace binding view with selected node placement, refresh, and
  metadata deletion without deleting node-local files.
- Session/chat view as the first primary activity surface.
- Message stream with user messages, assistant output, system/runtime events, and
  approval cards where needed.
- Session lifecycle controls: start, attach, detach, interrupt, stop, resume.
- Stable entity ids and routes for nodes, projects/workspaces, sessions, turns,
  messages, runtime events, and provider references.
- Visible non-production/trusted-deployment warning where the app exposes node
  pairing or remote node connection flows.

## Extension Readiness Requirements

V01 should stay small, but it must not close the path to the known target model.

Implementation requirements:

- Do not model a session as just a chat transcript. It must be tied to node,
  project/workspace, runtime, provider, lifecycle state, and events.
- Keep Core as the only client-facing coordination point. The web client should
  not depend on direct access to every node.
- Keep Node Daemon as the owner of local runtime/process/workspace execution.
- Use typed command and event envelopes rather than ad hoc text protocols.
- Keep message rendering compatible with future structured blocks, even though
  V01 only ships basic text/status/approval rendering.
- Keep node capability snapshots separate from future Tool Registry and Plugin
  Registry concepts.
- Keep UI layout and routing open for future workspace inspector, editor,
  terminal, diff/review, artifact, trace, and plugin surfaces.
- Keep project/workspace identity stable enough that later file, terminal, diff,
  check, and artifact references can attach to it without reshaping the product.

## Not In V01

These are not rejected directions. They are outside the first usable product
cut and belong in the feature queue.

- Project Workspace Inspector.
- File tree, file viewer, and lightweight text editor.
- Workspace terminal/PTY or command runner.
- Command/output history beyond session/provider output needed for chat.
- Workspace diff view, check/test result view, and git/review workflow.
- Production security hardening.
- Hardened node authentication, credential rotation, keychain integration, mTLS,
  request signing, and full audit/security policy.
- Multi-user auth, permissions, secrets management, and team security model.
- Tool Registry implementation.
- Plugin Registry implementation.
- MCP or external integration platform.
- Visual artifact system.
- Dynamic UI from agents or plugin-rendered UI blocks.
- Task-based sandbox runs.
- Durable workflow engine.
- Background run queue.
- CI webhook wakeups.
- Full MR/PR flow.
- Multi-agent orchestration.
- Managed cloud node pool.
- Complex team RBAC.
- Production support for additional providers such as OpenCode or Claude Code.
- Full provider-neutral feature parity across CLI agents.
- Full browser IDE, language server integration, broad refactoring workflows,
  debugger, and VS Code extension compatibility.
- Next.js as the required app runtime.
- Tauri-specific domain logic.

## Acceptance Criteria

V01 is coherent when:

- a user can enroll at least one Node Daemon and the model supports multiple
  registered nodes;
- a user can bind a project/workspace to a node and remove that binding from
  Core without deleting node-local files;
- a user can start a Core-managed Codex-backed session on a chosen node/project
  with persisted state and resume continuity where provider support allows it;
- chat works through Core-routed session commands and ordered runtime events;
- the user can detach, reattach, stop, and resume when provider support allows
  it;
- the UI survives reload and reconstructs node/project/session state from Core;
- offline/stale/error states are visible instead of being hidden in chat text;
- V01 is clearly marked as trusted/local/single-user or controlled development
  deployment, with no production-security claim;
- the implementation leaves explicit extension points for workspace, tools,
  plugins, artifacts, trace, and task-run surfaces without implementing those
  surfaces in V01.

## Open Decisions

- When should Uprava adopt a provider-native Codex app-server/live process
  protocol for process ownership, output streaming, approvals, and interrupt
  escalation?
- Should V01 use WebSocket for both UI and Node streams, or HTTP plus
  SSE/WebSocket by channel?
- How much provider raw event data should Core persist for debugging?
- How should project and workspace be represented when V01 does not yet expose a
  file/workspace inspector?
- What exact warning copy should the UI show for the non-production
  `controlled_dev` profile?
