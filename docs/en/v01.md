# Cortex V01

Status: `draft`

This document defines the first usable product version of Cortex.

`V01` is not a phase in a long calendar roadmap. It is the first coherent
product cut: what a user can start, open, inspect, control, and review in the
first version.

## Product Cut

V01 is **Developer Node Workbench**.

The user can run Cortex as a web control panel for live agent work on one or
more registered nodes. Cortex is not only chat: the user can see the project,
workspace, agent session, files, terminal, changes, basic trace, and node state.

V01 validates the first product thesis:

```text
Persistent Runtime + Node Daemon + Core UI
give more control, continuity, and reviewability
than a regular local agent chat.
```

## First-Run Experience

The first usable version should let a user:

1. Start Core Backend and Web Control Panel.
2. Pair a local or remote Node Daemon.
3. See node status, heartbeat, basic capabilities, and active sessions.
4. Register or open a project workspace on that node.
5. Start a persistent Codex-backed interactive session through an Agent Provider
   Adapter.
6. Attach to the session from the Web Control Panel.
7. Use chat as one surface of the session.
8. Inspect workspace files through the Project Workspace Inspector.
9. Open a text file, make a small explicit edit, and see that edit reflected in
   diff/trace.
10. Open a workspace-scoped terminal/PTY session.
11. Inspect command/output history, changed files, basic checks, and trace.
12. Stop, detach from, resume, or return to the same session later.

## Scope

### Core Backend

- Core as the control plane.
- Web Control Panel served by or deployed with Core.
- SQLite-backed local/single-user persistence.
- Project registry.
- Node registry and discovery.
- Node heartbeat state.
- Agent session/thread registry.
- Workspace binding metadata.
- Event log and basic trace metadata.
- Basic status model for node, workspace, runtime, and session.
- Minimal Tool Registry shape.
- Minimal Plugin Registry shape.
- Minimal visual block/artifact contract.

### Node Daemon

- Node enrollment and outbound registration with Core.
- Registered node identity/credential.
- Heartbeat with basic health and capability snapshot.
- Outbound control channel for active work.
- Workspace root registration or explicit workspace binding.
- Path normalization and workspace boundary enforcement.
- File metadata and content reads.
- Controlled text writes or patch applies for small edits.
- Workspace terminal/PTY lifecycle.
- Process lifecycle for provider runtimes.
- Codex provider adapter as the first production adapter.
- Local resource and permission enforcement for privileged workspace actions.

### Runtime And Agent Session

- Persistent Runtime as the default runtime strategy.
- Interactive developer session as the default work contract.
- Codex-backed runtime on Node Daemon.
- Minimal Agent Provider Adapter boundary.
- Start, resume, send turn, approve, interrupt, and stop.
- Attach/detach semantics.
- Provider resume reference when available.
- Managed runtime lifetime instead of an unbounded process promise.
- Runtime/session events normalized enough for UI, trace, and review.

### Distributed Runtime Coordination

- `Nodes -> Projects/Workspaces` inventory view.
- Explicit user choice of placement.
- Runtime session as the coordination unit.
- Core-routed commands to Node Daemon.
- Idempotent command envelope.
- Ordered events per runtime/session stream.
- Stale/offline node behavior.
- Minimum resource and conflict warning badges.
- Explicit user override recorded as an event where needed.

### Web Control Panel

- Nodes surface: list/table, status, heartbeat age, capabilities, active
  sessions, pair/revoke flows, and diagnostics.
- Project/workspace view.
- Session/chat view.
- Project Workspace Inspector with:
  - file tree;
  - file viewer;
  - lightweight text editor;
  - explicit save/apply;
  - terminal/PTY panel;
  - command/output history;
  - basic diff/check entry points;
  - trace links to workspace evidence.
- Stable references for files, ranges, edits, terminal sessions, commands,
  output ranges, diff hunks, checks, artifacts, turns, and trace events.

### Tech Stack

The V01 technical baseline is:

```text
Rust workspace
Axum Core Backend
Rust Node Daemon
SQLite
HTTP + WebSocket/SSE
React 19 + TypeScript + Vite
Tailwind CSS v4
shadcn/ui conventions
lucide-react
TanStack Query
TanStack Table
React Hook Form + Zod
Vitest
Rust tooling: cargo, rust-analyzer, rustfmt, clippy, bacon, nextest, audit, deny, taplo
```

## Not In V01

These are not rejected directions. They are outside the first usable product
cut unless a smaller slice becomes necessary to make V01 coherent.

- Task-based sandbox runs.
- Durable workflow engine.
- Background run queue.
- CI webhook wakeups.
- Full MR/PR flow.
- Multi-agent orchestration.
- Full plugin marketplace.
- External plugin JavaScript in the main React tree.
- Managed cloud node pool.
- Complex team RBAC.
- Production support for additional providers such as OpenCode or Claude Code.
- Full provider-neutral feature parity across CLI agents.
- Full browser IDE, language server integration, broad refactoring workflows,
  debugger, and VS Code extension compatibility.
- Next.js as the required app runtime.
- Tauri-specific domain logic.

## Open Decisions

- Should the first terminal surface be a fully interactive PTY, a command
  runner, or both?
- Should workspace editing save whole files first, apply patches first, or
  support both through one command shape?
- Should V01 discover workspaces automatically, or should users explicitly
  register workspace roots?
- What is the minimum stable reference schema for files, ranges, commands,
  output ranges, diffs, checks, artifacts, and trace events?
- Is session-level diff enough for the first version, or does V01 need per-turn
  diff attribution?
- Should minimal Tool Registry and Plugin Registry be internal schemas only, or
  also visible in UI?
- How much provider raw event data should Core persist for debugging?
