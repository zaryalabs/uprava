# Единый аудит архитектуры и качества кода Uprava

Дата: `2026-07-09`

Статус: `active`

Проверенный implementation baseline: `0.1.8`

## Назначение

Этот документ объединяет текущий аудит архитектуры Uprava, качества кода,
runtime-логики, security boundaries, надежности, Web performance, тестирования,
CI/CD и deployment posture.

Документ не отменяет историю shipped implementation и не переоткрывает
проблемы, уже исправленные в unified audit hardening slice `0.1.6`. Оставшиеся
актуальные задачи того архивного аудита включены ниже.
Этот отчет включает их и добавляет findings, подтвержденные на реализации
`0.1.8`.

Отчет является snapshot и документом приоритизации. Он не заменяет release
ledger или [`feature-queue.md`](../product/feature-queue.md).

## Scope И Метод

Аудит охватывал:

- каноническую продуктовую и архитектурную документацию;
- Core Backend, Node Daemon, shared protocol и logging crates;
- React Web Control Panel, workspace inspector, Monaco и xterm paths;
- SQLite persistence, commands, events, projections и enrollment;
- control-channel, SSE и terminal reconnect behavior;
- Codex provider execution и локальные workspace boundaries;
- Docker, systemd, release scripts и GitHub Actions;
- Rust, TypeScript, dependency и production-build quality gates.

Реализация проверялась относительно собственных архитектурных правил
репозитория и production-практик Rust, Axum, Tokio и React. Аудит не изменял
source files и не запускал live self-hosted deployment.

## Фактическая Карта Implementation Dependencies

Целевая системная граница `Web -> Core -> Node -> workspace/provider`
реализована, но внутри каждого процесса implementation связана значительно
сильнее:

```text
Web route/component
  -> generic Core API client и вручную mirrored TypeScript contracts
  -> локальные command-availability rules и browser-side event projections

Core Axum handler / control-frame handler
  -> shared AppState
  -> raw SQL, domain-state changes, command dispatch и global broadcasts

Node heartbeat loop
  -> одна копия NodeLocalState
  -> state JSON file
  -> control loop со второй копией NodeLocalState
  -> workspace, terminal и provider execution

uprava-protocol
  -> domain states и ids
  -> HTTP DTOs
  -> Core/Node control frames
  -> workspace и terminal DTOs
  -> artifact, agent-projection и UI-block contracts
```

Основная dependency problem находится не в верхнеуровневом distributed split,
а в отсутствии внутренних ownership, application-service, persistence и
projection boundaries внутри Core, Node и Web.

## Итоговая Оценка

Системная архитектура логична и должна быть сохранена:

- Core является control plane;
- Node Daemon является data plane;
- Web работает через Core, а не напрямую с nodes;
- commands и events явные, typed и traceable;
- workspace enforcement выполняется на Node;
- durable Core state отделен от локального provider/process state.

Code architecture пока отстает от этого design. Responsibility boundaries
хорошо описаны в документации, но implementation сосредоточена в нескольких
крупных файлах. Ряд критичных invariants поддерживается процедурным кодом, а не
module boundaries, authenticated context, transactions, bounded channels или
supervisor ownership.

Текущее состояние:

| Область | Оценка |
| --- | --- |
| System architecture | Сильное направление; разделение Core/Node/Web корректно |
| Code architecture | Работоспособна, но слишком концентрирована и слабо модульна |
| Локальная читаемость | В целом хорошие naming и explicit error handling |
| Надежность | Хороший test baseline, но остаются transaction/reconnect gaps |
| Security | Допустима только для явно trusted controlled deployment |
| Team/cloud readiness | Заблокирована trust-boundary, isolation и authorization gaps |
| Web performance | Функциональна, но initial bundle неприемлемо велик |
| Delivery/tooling | Сильный local gate; CI, E2E и image hardening требуют продолжения |

Uprava должна оставаться trusted single-user или controlled development
deployment до исправления P0 и P1 findings ниже.

## Подтвержденные Сильные Стороны

- Web не обходит Core для доступа к Node workspaces.
- Workspace roots canonicalized и проверяются через
  `UPRAVA_NODE_WORKSPACES`.
- File writes используют no-follow и post-open validation там, где это
  поддерживает ОС.
- Workspace command output ограничивается во время выполнения.
- Node local state записывается атомарной заменой файла и получает private
  permissions на Unix.
- Commands и events несут stable identifiers, correlation metadata и
  addressable references.
- Node command replay сохраняет terminal result payloads текущих workspace
  operations.
- Session event streaming использует session projection cursor поверх raw
  event scopes.
- Core имеет structured safe API errors, request timeout и request-body limits.
- Production Rust не использует panic-based request handling и имеет хорошее
  negative-path coverage.
- Local quality gate проверяет formatting, Clippy, TypeScript, Rust/Web tests,
  Rust dependency audit, license/source policy и production Web build.

Эти сильные стороны нужно сохранить во время refactoring. Основная цель
рекомендаций — сделать существующие invariants явными и независимо тестируемыми.

## Модель Severity

| Приоритет | Значение |
| --- | --- |
| P0 | Блокер deployment за пределами trusted controlled use; риск integrity или isolation |
| P1 | Высокий риск reliability, availability или correctness в нормальных flows |
| P2 | Важная maintainability, performance, security-hardening или operability работа |
| P3 | Cleanup и постепенное повышение качества |

## Объединенные Findings

### P0-1: Inbound Node Frames Аутентифицированы, Но Не Авторизованы По Объектам

Core аутентифицирует control WebSocket через Node id и bearer credential и
проверяет Node id в `Hello`. После этого command acknowledgements, command
results, event envelopes и terminal frames обрабатываются без проверки, что
указанный command, runtime, session, placement или terminal принадлежит
аутентифицированной Node.

Evidence:

- `crates/uprava-server/src/lib.rs:1595` - dispatch control frames;
- `crates/uprava-server/src/lib.rs:1663` - command acknowledgement/result;
- `crates/uprava-server/src/lib.rs:1694` - event batches;
- `crates/uprava-server/src/lib.rs:1717` - terminal output/status;
- `crates/uprava-server/src/lib.rs:3739` - event acceptance без authenticated
  Node context.

Impact:

- compromised enrolled Node может изменить command или projection state другой
  Node, если знает identifier;
- terminal output может быть injected в terminal другого placement;
- command state меняется без проверки legal transition;
- ownership guarantee control plane не enforced на transport boundary.

Recommendation:

1. Добавить authenticated `NodeContext` для каждого control connection.
2. Проверять target ownership до приема ACK/result frames.
3. Проверять связи event Node, runtime, session, placement, command и actor до
   persistence.
4. Проверять terminal ownership через placement и Node.
5. Отклонять invalid state transitions и non-terminal result states.
6. Добавить limits на frame count, event count, strings и payload bytes.
7. Добавить cross-node forgery regression tests.

### P0-2: Event Persistence И Projection Не Атомарны

Core вставляет event до применения runtime, session, turn, approval и message
projections. Шаги выполняются отдельными database operations, а не одной
transaction.

Если event insert успешен, а projection падает, Node не получает ACK и повторяет
event. Core видит существующий `event_id`, считает duplicate принятым и не
восстанавливает отсутствующую projection.

Evidence:

- `crates/uprava-server/src/lib.rs:3739` - duplicate check и event workflow;
- `crates/uprava-server/src/lib.rs:3800` - insert before projection;
- `crates/uprava-server/src/lib.rs:3893` - multi-step projection application.

Impact:

- event может быть durable, а соответствующий session/runtime/message state —
  навсегда отсутствовать;
- transient database или serialization failure создает silent read-model
  corruption;
- Node retry semantics не восстанавливает state.

Recommendation:

1. Выделять session projection cursor, вставлять event и применять все
   projections внутри одной SQLx transaction.
2. Publish в in-memory SSE bus только после commit.
3. Считать duplicate принятым только при fully projected/committed inbox record
   или сделать projection replay явно idempotent.
4. Добавить fault-injection test с падением после event insertion.

### P0-3: Codex Запускается С Отключенными Sandbox И Approvals

Новый и resumed Codex запускаются с
`--dangerously-bypass-approvals-and-sandbox`.

Evidence:

- `crates/uprava-node/src/main.rs:3883` - `codex exec`;
- `crates/uprava-node/src/main.rs:3926` - `codex exec resume`.

Workspace allow-list проверяет working directory, но не ограничивает ресурсы,
доступные provider process через permissions Unix user Node. При этом
implementation парсит provider approval events, хотя default launch mode
отключает основной approval/sandbox boundary.

Recommendation:

1. Использовать workspace-write sandboxing как default provider policy.
2. Оставить unrestricted execution только за explicit development setting.
3. Сохранять effective execution policy в command/event metadata.
4. Показывать unrestricted execution как hard warning в Web.
5. Интегрировать реальные provider approvals с Core до использования термина
   hardened для execution path.
6. Считать host Unix user, ACL и workspace-root controls defense in depth, а не
   единственным sandbox.

### P1-1: Медленные Commands Блокируют Весь Node Control Plane

Node ожидает command execution прямо в единственном WebSocket receive loop.
Codex turn или workspace command длительностью до 120 секунд блокирует:

- pings и protocol liveness;
- event-batch acknowledgements;
- terminal input, resize и close frames;
- unrelated workspace commands;
- commands других runtimes на той же Node.

Evidence: `crates/uprava-node/src/main.rs:702`.

Recommendation:

- оставить socket read/write tasks transport-only;
- направлять commands через bounded dispatcher;
- сериализовать operations per runtime/terminal там, где важен order;
- разрешить bounded concurrency для independent runtimes/workspaces;
- оставить одного authoritative state owner для dedupe, sequences и outbox;
- использовать cancellation tokens и tracked task handles.

### P1-2: Control Reconnect Может Удалить Более Новый Core Channel

Core заменяет `control_channels[node_id]` при новом connection. Когда старый
socket закрывается позднее, он безусловно удаляет map entry и может удалить уже
новый channel.

Evidence: `crates/uprava-server/src/lib.rs:1549`.

Recommendation:

- идентифицировать connections generation/token;
- удалять entry только при совпадении generation;
- явно закрывать или supersede старый connection;
- тестировать overlapping connect/disconnect ordering.

### P1-3: PTY Ownership Привязан К Временному Control Connection

`WorkspaceTerminalManager` создается внутри одного control-channel connection.
После reconnect Node теряет terminal registry, хотя Core может продолжать
показывать terminals как running.

При drop manager закрываются terminal control senders. PTY task выходит из
control loop и ждет child, не остановив его; shell может остаться живым и
недоступным.

Evidence:

- `crates/uprava-node/src/main.rs:702` - per-connection manager;
- `crates/uprava-node/src/main.rs:1026` - terminal task;
- `crates/uprava-node/src/main.rs:1105` - lifecycle и child wait.

Recommendation:

- перенести terminal supervision на daemon scope;
- хранить metadata, replay и task handles в одном supervisor;
- определить detach, reconnect, reattach и daemon-shutdown semantics;
- kill/reap children при cancellation с bounded shutdown timeout;
- reconcile Core terminal projections после reconnect.

### P1-4: Control Queues И Provider Output Не Ограничены

Core и Node используют unbounded control channels. Codex накапливает raw
stdout, stderr, activity events и live outbox events без глобального byte/event
limit во время process run. Outbox retention применяется только после
завершения provider.

Evidence:

- `crates/uprava-server/src/lib.rs:1551`;
- `crates/uprava-node/src/main.rs:671`;
- `crates/uprava-node/src/main.rs:1272`;
- `crates/uprava-node/src/main.rs:4008`.

Recommendation:

- заменить unbounded channels на bounded queues с explicit overflow policy;
- ограничить line length, total stdout/stderr bytes, activity-event count и raw
  JSON;
- применять outbox retention во время streaming;
- batch events и распространять backpressure;
- emit explicit truncation/gap events вместо silent drop.

### P1-5: Enrollment И Aggregate Creation Не Имеют Units Of Work

Enrollment claim читает state, создает Node, заменяет capabilities и обновляет
enrollment отдельными operations. Concurrent claims могут оба увидеть approved
enrollment и создать несколько valid Nodes.

Session creation и turn submission также сохраняют несколько частей aggregate
без transaction. Partial failure может оставить session без runtime/command или
command без turn/message.

Evidence:

- `crates/uprava-server/src/lib.rs:4287`;
- `crates/uprava-server/src/lib.rs:2938`;
- `crates/uprava-server/src/lib.rs:3201`.

Recommendation:

- создавать каждый aggregate в одной transaction;
- claim enrollment через compare-and-set, например
  `update ... where status = 'approved' returning ...`;
- выделять turn indexes и event projection cursors transactionally;
- добавить concurrent claim/session/turn regression tests.

### P1-6: Session SSE Переподключается После Каждого Event

Session stream effect зависит от `session.data.events`. Применение SSE event
меняет array, закрывает текущий `EventSource` и открывает новый с обновленным
cursor.

Evidence: `apps/web/src/features/sessions/SessionRoute.tsx:53`.

Impact:

- connection churn на каждый provider activity event;
- повторные Core catch-up queries;
- reconnect races и лишняя нагрузка во время active turns.

Recommendation:

- держать один stream per session id;
- хранить current cursor в ref или dedicated stream store;
- reconnect только при explicit reload, terminal error или session change;
- тестировать, что несколько received events не создают несколько
  `EventSource` instances.

### P1-7: Initial Web Bundle Чрезмерно Велик

Production build создает initial JavaScript chunk примерно `4,464.66 kB`
minified и `1,166.58 kB` gzip, а также Monaco workers, включая TypeScript worker
примерно `7 MB`.

Все routes импортируются статически, а placement/workspace route eager-load-ит
Monaco, xterm и styles.

Evidence:

- `apps/web/src/app/App.tsx:3`;
- `apps/web/src/features/workspace-inspector/WorkspaceInspector.tsx:1`.

Recommendation:

- lazy-load routes;
- отдельно lazy-load editor, diff и terminal surfaces;
- загружать только нужные Monaco languages/workers;
- добавить bundle budget в CI;
- использовать более легкий read-only renderer, когда editing не активен.

### P1-8: Node State Имеет Двух Конкурирующих Владельцев

Node heartbeat loop загружает и хранит `NodeLocalState`, а затем запускает
control loop с клоном этого state. Control task изменяет и сохраняет свою копию,
а heartbeat loop периодически перечитывает JSON file, чтобы увидеть эти
изменения. При authentication failure heartbeat loop может abort-ить control
task и записать собственную копию.

Evidence:

- `crates/uprava-node/src/main.rs:77` - владелец state в heartbeat loop;
- `crates/uprava-node/src/main.rs:115` - disk reload для получения control
  state;
- `crates/uprava-node/src/main.rs:132` - control loop получает cloned state;
- `crates/uprava-node/src/main.rs:630` - второй long-lived state owner.

Impact:

- state file используется как inter-task communication, а не только
  persistence;
- возможны stale heartbeat data и lost updates;
- abort control task не доказывает завершение blocking save;
- ordering runtime, credential и outbox невозможно хранить и тестировать в
  одном месте.

Recommendation:

1. Ввести один daemon-level `NodeSupervisor` или state-store actor.
2. Проводить heartbeat snapshots, command dedupe, sequences, outbox и runtime
   metadata через этого владельца.
3. Скрыть disk writes за владельцем и вынести blocking serialization/fsync с
   Tokio worker threads.
4. Добавить concurrent heartbeat/control/shutdown tests, проверяющие monotonic
   state.

### P1-9: Reliable Request/Response Paths Используют Lossy Global Broadcasts

Workspace HTTP operations подписываются на global command-result broadcast,
отправляют command и ждут подходящий result. Terminal clients аналогично
подписываются на один global terminal broadcast и фильтруют frames по terminal
id.

Evidence:

- `crates/uprava-server/src/lib.rs:2796` - dispatch и ожидание workspace
  command;
- `crates/uprava-server/src/lib.rs:2870` - lagged command results игнорируются;
- `crates/uprava-server/src/lib.rs:1801` - global terminal broadcast relay;
- `crates/uprava-server/src/lib.rs:1859` - lagged terminal output игнорируется.

Impact:

- durable result может существовать в SQLite, пока ожидающий HTTP request
  завершится timeout;
- каждый waiter получает unrelated results или terminal frames;
- terminal output может быть молча потерян без sequence-gap notification;
- synchronous HTTP requests остаются связаны с 5-120 секундным distributed
  command lifecycle.

Recommendation:

- использовать per-command `oneshot` waiters с fallback lookup durable result;
- использовать per-terminal channels или terminal supervisor registry;
- превращать terminal sequence gaps в явные replay/reload events;
- определить, какие workspace operations являются synchronous RPC, а какие
  должны возвращать asynchronous command resource.

### P1-10: Identity Rules Project, Placement И Workspace Конфликтуют

Heartbeat discovery выводит project и placement ids из одной пары
`(node_id, workspace_path)`. Explicit placement validation вместо этого создает
новые случайные project и placement ids. В database отсутствует uniqueness
constraint на `(node_id, workspace_path)`.

Одновременно `Workspace` и `Placement` references содержат placement id, а Web
routes `/workspaces/:id` и `/placements/:id` ведут в один component.

Evidence:

- `crates/uprava-server/src/lib.rs:2085` - explicit placement creation;
- `crates/uprava-server/src/lib.rs:4617` - heartbeat-derived placement/project;
- `crates/uprava-server/src/lib.rs:6661` - project и placement ids используют
  одинаковые identity inputs;
- `crates/uprava-server/src/lib.rs:7263` - в placement schema нет node/path
  uniqueness;
- `apps/web/src/app/App.tsx:28` - placement/workspace route aliases.

Impact:

- один physical workspace может породить несколько logical placements и
  projects;
- один repository на разных Nodes нельзя естественно представить как один
  Project с несколькими placements;
- Project сейчас близок к one-to-one mirror Placement;
- Workspace является вторым именем Placement, а не отдельно определенной domain
  entity или явно документированной UI surface.

Recommendation:

1. Зафиксировать ADR с identity/lifecycle Project, Workspace и Placement.
2. Решить, уникальна ли `(node_id, canonical_workspace_path)` или aliases
   являются first-class.
3. Дать Project независимую от Node placement identity, если несколько копий
   одного repository должны входить в один project.
4. Считать Workspace либо реальной entity, либо явно названным view над
   Placement, но не обоими одновременно.

### P2-1: Code Boundaries Существуют В Документации Больше, Чем В Modules

Основные точки концентрации:

| Файл | Примерно строк | Смешанные responsibilities |
| --- | ---: | --- |
| `crates/uprava-server/src/lib.rs` | 11,850 | config, HTTP, auth, control, commands, events, projections, SQL, migrations, tests |
| `crates/uprava-node/src/main.rs` | 7,287 | config, enrollment, state, control, workspace, PTY, providers, outbox, tests |
| `crates/uprava-protocol/src/lib.rs` | 1,621 | все public protocol contracts и tests |
| `apps/web/src/features/workspace-inspector/WorkspaceInspector.tsx` | 1,364 | data, drafts, tree, Monaco, xterm, commands, diff, history |
| `apps/web/src/workbench/inspector/InspectorStack.tsx` | 933 | ref resolution, data shaping и inspector presentations |

Локальные naming и error handling в целом хорошие. Cleanliness problem является
структурной: изменения пересекают слишком много responsibilities, а invariants
остаются implicit в procedural code.

Recommendation: после P0/P1 behavior fixes выполнить mechanical module split,
описанный ниже. Не вводить broad framework-style traits до появления реальных
alternate implementations или полезных test seams.

### P2-2: Rust И Web Protocol Contracts Могут Незаметно Разойтись

Web client напрямую cast-ит successful HTTP JSON к generic TypeScript types.
SSE проверяет JSON syntax, но не envelope shape. Rust и TypeScript contracts
поддерживаются вручную.

В test fixtures уже есть подтвержденный drift: E2E fixture использует
`security.mode = "controlled_dev"`, а Rust и TypeScript объявляют
`"hardened"`.

Evidence:

- `apps/web/src/shared/api/http-client.ts:62`;
- `apps/web/src/shared/api/sse-client.ts:5`;
- `apps/web/src/shared/protocol/types.ts:86`;
- `apps/web/e2e/golden.spec.ts:65`.

Recommendation:

1. Генерировать TypeScript/JSON Schema из Rust contract или принять
   OpenAPI/JSON Schema как shared source.
2. Runtime-validate high-risk HTTP, SSE и terminal payloads.
3. Генерировать checked fixtures из Rust для Web contract tests.
4. Превращать malformed payloads в typed observable errors.

### P2-3: Node Local State Является Растущим Multi-Concern Document

Один JSON state file содержит enrollment identity, credential, command status,
typed result replay, runtime sequences, outbox, provider/workspace metadata,
runtime states, transcripts, resume refs и placement sequences.

Outbox bounded, но несколько maps растут без retention. Весь state синхронно
serializes, flushes и fsyncs после изменений.

Evidence: `crates/uprava-node/src/main.rs:230`.

Recommendation:

- ввести единый state-store actor/API;
- логически разделить identity, command cache, outbox и runtime metadata;
- определить retention/compaction completed commands и stopped runtimes;
- сохранить atomic persistence, но избежать full-document rewrite на каждый
  event;
- рассмотреть SQLite или append/checkpoint model при росте state.

### P2-4: Public Endpoints И Local Auth Требуют Hardening

`/client/logs` и Node enrollment request/claim routes публичны по design, но не
имеют надежных peer-aware quotas. Client logs append-ятся в unrotated file.
Local passwords используют salted single SHA-256 вместо password KDF.
Authentication failure counters in-memory и coarse-grained.

Evidence:

- `crates/uprava-server/src/lib.rs:376`;
- `crates/uprava-server/src/lib.rs:1090`;
- `crates/uprava-server/src/lib.rs:6680`.

Recommendation:

- использовать Argon2id или scrypt с versioned stored format;
- добавить peer-aware и global rate/concurrency limits;
- ограничить pending enrollment count и frequency;
- аутентифицировать client logs после login или определить narrow pre-auth
  logging policy;
- rotate/cap local logs и audit-event retention;
- не называть current profile просто hardened.

### P2-5: Workspace Drafts Могут Быть Потеряны Без Предупреждения

Workspace file query updates заменяют editor state даже при dirty buffer.
Refresh или выбор другого path может затереть unsaved edit.

Evidence: `apps/web/src/features/workspace-inspector/WorkspaceInspector.tsx:153`.

Recommendation:

- хранить drafts per placement/path;
- подтверждать destructive refresh/navigation;
- обновлять buffer из server data только когда он clean;
- добавить draft preservation и conflict tests.

### P2-6: Deployment Images И CI Требуют Production Hardening

Текущие concerns:

- Core и Web containers работают как root;
- Web image копирует все `node_modules` и использует `vite preview`;
- Docker builds используют mutable base-image tags;
- Web image использует `npm install`, а не `npm ci`;
- Rust image builds не передают `--locked`;
- Core и Node Dockerfiles каждый собирают оба binaries;
- CI дает `packages: write` на workflow scope, включая check job;
- pull-request code выполняется на persistent self-hosted runner;
- CI использует Rust `1.90`, хотя declared MSRV — `1.88`;
- Playwright E2E и npm dependency audit не входят в required CI gate.

Evidence:

- `Dockerfile.core`;
- `Dockerfile.node`;
- `apps/web/Dockerfile`;
- `.github/workflows/ci.yml`;
- `Cargo.toml:15`;
- `Makefile:59` и `Makefile:414`.

Recommendation:

- использовать non-root runtime users и read-only filesystems где возможно;
- serving static Web assets через production static server;
- использовать `npm ci`, `cargo build --locked` и digest-pinned base images;
- разделить binary builds и улучшить layer caching;
- scope workflow permissions per job;
- isolate/clean self-hosted runner workspaces и ограничить untrusted PR code;
- добавить explicit Rust `1.88` MSRV job и current stable job;
- запускать deterministic Playwright golden tests и npm audit policy в CI.

Production npm audit во время проверки сообщил три findings: один moderate
DOMPurify issue через `monaco-editor` и два low-severity findings, включая
`esbuild`. До выбора совместимого Monaco upgrade/pin нужно оценить reachability.

### P2-7: Logging И Observability Пока Синхронны/Неполны

`uprava-logging` пишет и flush-ит через standard `Mutex<File>` из tracing writer
path. High-volume provider events могут синхронно блокировать Tokio worker
threads. File rotation и metrics/OTLP position из
[`deployment-observability.md`](../deploy/deployment-observability.md) еще не
реализованы.

Evidence: `crates/uprava-logging/src/lib.rs:23`.

Recommendation:

- использовать non-blocking tracing writers с bounded loss policy;
- полагаться на stdout/stderr и platform rotation в containers;
- добавить explicit file rotation/retention для host logs;
- реализовать low-cardinality command, event, queue, reconnect, outbox и
  terminal metrics;
- связывать request/correlation ids через middleware spans и response headers.

### P2-8: Shared Protocol Смешивает Domain, Transport И Presentation

`uprava-protocol` содержит domain states и identifiers, HTTP request/response
DTOs, control frames, workspace и terminal contracts, references, artifact
projections и generic presentation model `UiBlock`. Поэтому Node зависит от
Web-facing contracts, которые не использует.

Command и event envelopes также повторяют identity через `scope_ref`, optional
Node/runtime/session/turn ids и reference lists без type-level правила их
согласованности. Payload встроенных commands и events остается untyped JSON.

Evidence:

- `crates/uprava-protocol/src/lib.rs:1` - все contracts в одном module;
- `crates/uprava-protocol/src/lib.rs:318` - generic command payload;
- `crates/uprava-protocol/src/lib.rs:336` - redundant event identity и generic
  payload;
- `crates/uprava-protocol/src/lib.rs:1160` - UI block в shared Rust protocol.

Recommendation:

- разделить crate на внутренние modules `domain`, `api`, `control`, `workspace`,
  `references` и `projections` до рассмотрения новых crates;
- определить typed payloads для встроенных commands/events, сохранив explicit
  extension JSON variant;
- проверять envelope identity relationships при construction и ingress;
- перенести `UiBlock` в Web, пока Core или plugins не публикуют его как protocol
  object.

### P2-9: State И Command Policy Повторно Реализованы В Разных Layers

Mapping events в runtime/session states существует в Core event projection,
Node local projection и Web cache projection. Command availability существует
в Core request validation, Core Agent Projection и Web command registry.

Evidence:

- `crates/uprava-server/src/lib.rs:3893` - Core event projection;
- `crates/uprava-server/src/lib.rs:4803` - Core command validation;
- `crates/uprava-server/src/lib.rs:5756` - Core available-command projection;
- `crates/uprava-node/src/main.rs:1814` - Node runtime projection;
- `apps/web/src/workbench/projection/apply-session-event.ts:16` - Web event
  projection;
- `apps/web/src/workbench/commands/registry.ts:76` - Web command policy.

Impact:

- новый state/event/command требует synchronized edits в нескольких files и
  languages;
- Web может включить action, который Core отклонит, или показать projection,
  отличающуюся от durable read model;
- Agent Projection публикует `available_commands`, но Web в основном вычисляет
  список заново и использует server list только как display data.

Recommendation:

- сделать legal state transitions и command predicates pure Core domain rules;
- публиковать authoritative action capability/reason data для Web;
- ограничить Node projection локальным operational ownership;
- использовать generated fixtures/contract tests для доказательства
  эквивалентности incremental и durable projections, если browser-side
  projection остается необходимой.

### P2-10: Persistence Хранит Duplicate Representations Без Authority Rules

Capabilities хранятся одновременно в `nodes.capabilities_json` и normalized
`node_capabilities`. Commands и events содержат normalized columns, отдельные
JSON columns и полный serialized envelope. Session/runtime links и provider
names хранятся с обеих сторон отношения.

Неизвестные database state strings молча превращаются в валидные initial states
вроде `Created`, `Starting` или `Recorded`. Inline migrations используют
`create if not exists` и optional `alter table`, тогда как внешний schema
version остается `1`.

Evidence:

- `crates/uprava-server/src/lib.rs:4677` - normalized capability replacement;
- `crates/uprava-server/src/lib.rs:6911` - permissive state parsers;
- `crates/uprava-server/src/lib.rs:7171` - inline migrations;
- `crates/uprava-server/src/lib.rs:7285` - reciprocal session/runtime data;
- `crates/uprava-server/src/lib.rs:7356` - normalized и full JSON storage
  commands/events.

Recommendation:

1. Определить authoritative representation и rebuild policy для каждого
   duplicated snapshot/index.
2. Обновлять duplicated representations в одной transaction или удалить
   неиспользуемую копию.
3. Возвращать observable corruption/compatibility error для неизвестных
   persisted states.
4. Перейти на numbered SQLx migrations с checksums и meaningful schema version.

### P2-11: Provider Boundary Протекает Node Storage И Transport

`RuntimeManager` сейчас dispatch-ит только в `Codex` или `Unsupported`, а Codex
adapter получает Node sequence maps, transcript maps, resume maps и optional
live control-channel sink. Несколько методов suppress-ят `too_many_arguments`,
а failure отправки live event превращается в provider execution failure.

Evidence:

- `crates/uprava-node/src/main.rs:3351` - two-variant runtime manager;
- `crates/uprava-node/src/main.rs:3443` - adapter получает Node storage maps;
- `crates/uprava-node/src/main.rs:3639` - suppression argument count;
- `crates/uprava-node/src/main.rs:4123` - transport failure меняет provider
  result.

Recommendation:

- оставить concrete Codex provider до появления второго реального provider,
  формирующего полезный shared interface;
- передавать bounded `ProviderExecutionContext` и возвращать/stream-ить typed
  provider outcomes через daemon-owned sink;
- отделить transport reconnect от локального provider success;
- отдать runtime supervisor ownership process handles, cancellation, transcript
  и resume state.

### P2-12: Artifact И Agent Projections Имеют Ошибочные Identity Semantics

Artifact Tree генерируется из messages и events на каждый request и каждый раз
получает новые случайные artifact ids. Поэтому это trace/read-model tree, а не
stable artifact aggregate. Agent Projection повторяет session/runtime/placement
data, а Web создает synthetic `SessionSummary` с hardcoded title и active state
для переиспользования command-registry API.

Evidence:

- `crates/uprava-server/src/lib.rs:5336` - request-time artifact tree;
- `crates/uprava-server/src/lib.rs:5344` - random child artifact ids;
- `crates/uprava-server/src/lib.rs:5384` - request-time Agent Projection;
- `apps/web/src/features/agent-projection/AgentProjectionPanel.tsx:107` -
  synthetic session summary.

Impact:

- сохраненные artifact references и React keys нестабильны между refetches;
- имя `ArtifactTree` обещает persistence и identity, которых нет;
- view models обратно преобразуются в fabricated domain models.

Recommendation:

- использовать deterministic projection ids и назвать surface trace projection
  либо ввести persisted Artifact aggregate;
- передавать actions Agent Projection собственный typed context вместо
  fabricated `SessionSummary`;
- не повторять полные session data, когда достаточно stable references или
  focused action capabilities.

### P2-13: Web Workbench Abstractions Объединяют Несвязанные Responsibilities

Command registry объединяет remote domain actions, local clipboard commands,
navigation и cache-invalidation callbacks за одним optional property-bag
context. Timeline, Inspector, event projection и terminal code имеют
независимые JSON/ref parsers. Inspector references сохраняются как полный
encoded JSON в URL и проверяются только по наличию string `kind`.

Evidence:

- `apps/web/src/workbench/commands/registry.ts:28` - mixed command taxonomy;
- `apps/web/src/workbench/commands/registry.ts:48` - optional context bag;
- `apps/web/src/workbench/references/refs.ts:9` - full JSON URL encoding;
- `apps/web/src/workbench/references/refs.ts:246` - weak reference validation;
- `apps/web/src/features/sessions/timeline-blocks.ts:21` - repeated view-model
  reconstruction.

Recommendation:

- разделить server actions, navigation commands и local UI operations;
- использовать discriminated per-command context types;
- централизовать generated/runtime-validated payload и reference decoders;
- кодировать в URL stable short reference keys вместо arbitrary full JSON;
- memoize timeline projection и не перестраивать все blocks при unrelated
  renders.

### P2-14: Speculative Contracts И Dependencies Добавляют YAGNI Cost

Некоторые terminal request DTOs и future reference variants не имеют production
consumer. `UiBlock` shared с Rust до того, как server начал публиковать blocks.
Web package объявляет React Table, React Hook Form, Zod и xterm search addon, но
не использует их. Node и Server также объявляют direct logging-related
dependencies, которые больше не нужны после выделения logging helper.

Recommendation:

- добавлять protocol type только вместе с producer, consumer и contract test;
- удалять unused dependencies и возвращать их вместе с использующей feature;
- не расширять plugin/tool/block taxonomy до появления vertical scenario;
- использовать Zod или generated validators, если dependency остается, вместо
  параллельных handwritten guards.

### P3: Постепенные Улучшения Качества

- Перенести inline SQL migrations в numbered SQLx migration files с checksums.
- Добавить workspace lint policy для важных Clippy и Rust lints.
- Добавить crate/module documentation в `uprava-protocol` и описать public
  invariants, errors и compatibility expectations.
- При совместимости выровнять direct `tokio-tungstenite` dependency со stack,
  resolved через Axum.
- Добавить documentation link/style validation; текущая проверка подтверждает
  только наличие files.
- Оставлять comments для invariants и design reasons; durable behavior
  фиксировать tests и design documents.

## Последовательность Исправлений

### Slice A: Control-Plane Integrity

1. Связать inbound Node frames с authenticated ownership.
2. Добавить legal command transition validation.
3. Сделать event acceptance и projection transactional.
4. Сделать enrollment claim transactional compare-and-set.
5. Добавить cross-node, partial-projection и concurrent-claim tests.

### Slice B: Node Execution And Isolation

1. Заменить unrestricted Codex default на explicit execution policies.
2. Разделить socket I/O, state ownership и command execution.
3. Добавить bounded queues и provider output caps.
4. Добавить runtime-scoped ordering и bounded cross-runtime concurrency.

не может исчерпать Node memory через unbounded output.

### Slice C: Terminal And Reconnect Reliability

1. Добавить connection generations в Core.
2. Вынести PTY supervision за пределы control connection.
3. Добавить cancellation, child reaping и terminal reconciliation.
4. Добавить explicit terminal output gap/replay semantics.
5. Тестировать reconnect с active terminals и in-flight commands.

Completion gate: reconnect не теряет active control channel и не оставляет
inaccessible shell processes.

### Slice D: Protocol And Web Runtime

1. Исправить lifetime SSE effect.
2. Установить generated/shared protocol contracts.

1. Разделить Core на transport, application, domain, control и persistence.
2. Разделить Node по supervisors, workspace, provider и state/outbox ownership.

Completion gate: command lifecycle, event ingestion, Node state, PTY lifecycle

## Финальная Позиция

Uprava не требует conceptual rewrite. Архитектура Core/Node/Web является
хорошим foundation. Следующая работа должна заставить code enforce эту
архитектуру: authenticated object ownership в Core, transactional durable
state, supervisor-owned resources на Node, bounded asynchronous channels,
validated shared protocol contracts и меньшие independently testable modules.

Добавление Tool Registry, Plugin Registry, новых providers или
multi-user/cloud features до этого ремонта умножит текущие coupling и
trust-boundary risks. Поэтому P0/P1 slices должны предшествовать крупному
feature expansion.
