# Cortex V01

Статус: `draft`

Этот документ определяет первую пригодную к использованию версию продукта Cortex.

`V01` - это не фаза длинного календарного roadmap. Это первый цельный срез
продукта: что пользователь может запустить, открыть, посмотреть, контролировать
и проверить в первой версии.

## Продуктовый срез

V01 - это **Developer Node Workbench**.

Пользователь может запускать Cortex как web control panel для живой агентской
работы на одной или нескольких зарегистрированных нодах. Cortex - это не только
chat: пользователь видит project, workspace, agent session, files, terminal,
changes, basic trace and node state.

V01 проверяет первую продуктовую гипотезу:

```text
Persistent Runtime + Node Daemon + Core UI
дают больше контроля, непрерывности и reviewability,
чем обычный локальный agent chat.
```

## Первый пользовательский запуск

Первая пригодная к использованию версия должна позволять пользователю:

1. Запустить Core Backend и Web Control Panel.
2. Подключить локальный или удаленный Node Daemon.
3. Увидеть node status, heartbeat, базовые capabilities and active sessions.
4. Зарегистрировать или открыть project workspace на этой node.
5. Запустить persistent interactive session на базе Codex через Agent Provider
   Adapter.
6. Подключиться к session из Web Control Panel.
7. Использовать chat как одну из поверхностей session.
8. Смотреть workspace files через Project Workspace Inspector.
9. Открыть text file, внести небольшое явное изменение и увидеть это изменение
   в diff/trace.
10. Открыть workspace-scoped terminal/PTY session.
11. Смотреть command/output history, changed files, basic checks and trace.
12. Остановить session, отсоединиться, возобновить или вернуться к той же
    session позже.

## Scope

### Core Backend

- Core как control plane.
- Web Control Panel, который обслуживается Core или деплоится вместе с ним.
- SQLite-backed local/single-user persistence.
- Project registry.
- Node registry and discovery.
- Node heartbeat state.
- Agent session/thread registry.
- Workspace binding metadata.
- Event log and basic trace metadata.
- Basic status model для node, workspace, runtime and session.
- Минимальная форма Tool Registry.
- Минимальная форма Plugin Registry.
- Минимальный visual block/artifact contract.

### Node Daemon

- Node enrollment and outbound registration with Core.
- Registered node identity/credential.
- Heartbeat with basic health and capability snapshot.
- Outbound control channel for active work.
- Workspace root registration или explicit workspace binding.
- Path normalization and workspace boundary enforcement.
- File metadata and content reads.
- Controlled text writes or patch applies для небольших edits.
- Workspace terminal/PTY lifecycle.
- Process lifecycle for provider runtimes.
- Codex provider adapter как первый production adapter.
- Local resource and permission enforcement for privileged workspace actions.

### Runtime and Agent Session

- Persistent Runtime как default runtime strategy.
- Interactive developer session как default work contract.
- Codex-backed runtime на Node Daemon.
- Минимальная граница Agent Provider Adapter.
- Start, resume, send turn, approve, interrupt and stop.
- Attach/detach semantics.
- Provider resume reference, если он доступен.
- Managed runtime lifetime вместо обещания бесконечного процесса.
- Runtime/session events, нормализованные достаточно для UI, trace and review.

### Distributed Runtime Coordination

- `Nodes -> Projects/Workspaces` inventory view.
- Явный выбор placement пользователем.
- Runtime session как coordination unit.
- Core-routed commands to Node Daemon.
- Idempotent command envelope.
- Ordered events per runtime/session stream.
- Stale/offline node behavior.
- Минимальные resource and conflict warning badges.
- Explicit user override recorded as an event where needed.

### Web Control Panel

- Nodes surface: list/table, status, heartbeat age, capabilities, active
  sessions, pair/revoke flows and diagnostics.
- Project/workspace view.
- Session/chat view.
- Project Workspace Inspector with:
  - file tree;
  - file viewer;
  - lightweight text editor;
  - explicit save/apply;
  - terminal/PTY panel;
  - command/output history;
  - basic diff/check entry points;
  - trace links to workspace evidence.
- Stable references for files, ranges, edits, terminal sessions, commands,
  output ranges, diff hunks, checks, artifacts, turns and trace events.

### Tech Stack

Технический baseline для V01:

```text
Rust workspace
Axum Core Backend
Rust Node Daemon
SQLite
HTTP + WebSocket/SSE
React 19 + TypeScript + Vite
Tailwind CSS v4
shadcn/ui conventions
lucide-react
TanStack Query
TanStack Table
React Hook Form + Zod
Vitest
Rust tooling: cargo, rust-analyzer, rustfmt, clippy, bacon, nextest, audit, deny, taplo
```

## Не входит в V01

Эти направления не отвергаются. Они находятся за пределами первого пригодного
продуктового среза, если только меньший кусок не станет необходимым для
целостности V01.

- Task-based sandbox runs.
- Durable workflow engine.
- Background run queue.
- CI webhook wakeups.
- Full MR/PR flow.
- Multi-agent orchestration.
- Full plugin marketplace.
- External plugin JavaScript in the main React tree.
- Managed cloud node pool.
- Complex team RBAC.
- Production support for additional providers such as OpenCode or Claude Code.
- Full provider-neutral feature parity across CLI agents.
- Full browser IDE, language server integration, broad refactoring workflows,
  debugger and VS Code extension compatibility.
- Next.js as the required app runtime.
- Tauri-specific domain logic.

## Открытые решения

- Должна ли первая terminal surface быть fully interactive PTY, command runner
  или поддерживать оба режима?
- Должна ли workspace editing сначала сохранять whole files, сначала применять
  patches или поддерживать оба пути через одну форму command?
- Должна ли V01 автоматически находить workspaces, или пользователь должен явно
  регистрировать workspace roots?
- Какой минимальный stable reference schema нужен для files, ranges, commands,
  output ranges, diffs, checks, artifacts and trace events?
- Достаточно ли session-level diff для первой версии, или V01 нужен per-turn
  diff attribution?
- Должны ли минимальные Tool Registry and Plugin Registry быть только internal
  schemas, или их также нужно показывать в UI?
- Сколько provider raw event data Core должен сохранять для debugging?
