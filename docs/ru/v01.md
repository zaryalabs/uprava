# Cortex V01

Статус: `draft`

Этот документ определяет первую пригодную к использованию версию продукта
Cortex.

`V01` - это не фаза длинного календарного roadmap. Это первый цельный срез
продукта: что пользователь может запустить, открыть, контролировать и к чему
может вернуться в первой версии.

## Продуктовый срез

V01 - это **Distributed Agent Control Panel**.

Пользователь может запускать Cortex как web control panel для persistent agent
sessions на одной или нескольких зарегистрированных нодах. Первый видимый
workflow намеренно близок к chat, но это не просто chat: node placement, project
binding, session lifecycle, runtime state and event history являются
полноценными product objects.

V01 проверяет первую продуктовую гипотезу:

```text
Persistent Runtime + Node Daemon + Core UI
превращают agent chat в distributed control surface.
```

Первая версия должна ощущаться как небольшая панель управления распределенной
системой агентов, а не как локальная chat-обертка и еще не как полноценный
developer workbench.

## Trust Boundary

V01 - это trusted local/single-user или controlled development deployment. Это
не production security release.

V01 может включать simple development enrollment, local-only defaults and
best-effort guardrails, нужные чтобы избежать случайного misuse, но hardened
auth, credential lifecycle, permissions, audit policy, secrets handling and
multi-user security относятся к первому post-V01 security baseline.

## Первый пользовательский запуск

Первая пригодная к использованию версия должна позволять пользователю:

1. Запустить Core Backend и Web Control Panel.
2. Подключить один или несколько local or explicitly trusted remote Node Daemons.
3. Увидеть sidebar tree формата `Nodes -> Projects/Workspaces -> Sessions`.
4. Увидеть node status, heartbeat age, basic capabilities, active sessions and
   degraded/offline state.
5. Зарегистрировать или открыть project/workspace binding на выбранной node.
6. Запустить Core-managed Codex-backed session для этого project через Agent
   Provider Adapter.
7. Подключиться к session из Web Control Panel.
8. Использовать chat как первую и основную поверхность session.
9. Получать assistant output and runtime status в session view.
10. Approve, interrupt, stop, detach, reattach, resume или вернуться к той же
    session позже, если provider это поддерживает.
11. Перезагрузить UI и все еще видеть nodes, project bindings, sessions,
    messages, lifecycle state and recent event history.
12. Понимать, где именно запущена session и какая node/project ей владеет.

## Scope

### Core Backend

- Core как control plane.
- Web Control Panel, который обслуживается Core или деплоится вместе с ним.
- SQLite-backed local/single-user persistence.
- Node registry and enrollment state.
- Node heartbeat state and basic capability snapshots.
- Project/workspace registry как metadata and placement context.
- Agent session/thread registry.
- Runtime instance registry.
- Message history for interactive sessions.
- Event log with ordered session/runtime lifecycle events.
- Basic status model для node, project/workspace, runtime and session.
- Core-routed commands to Node Daemon.
- Idempotent command envelope for session/runtime commands.
- Provider identity and resume references, если они доступны.
- Versioned Core-to-Node API and event envelopes.

### Node Daemon

- Node enrollment and outbound registration with Core.
- Development node identity/enrollment metadata.
- Heartbeat with basic health and capability snapshot.
- Outbound control/stream channel for active work.
- Workspace root registration or explicit workspace binding validation.
- Runtime adapter lifecycle, including provider process ownership where adapter
  support exists.
- Codex provider adapter как первый production adapter.
- Normalized provider input, output, lifecycle events, approvals, warnings and
  errors.
- Start, attach, detach, interrupt, stop and resume behavior, где provider это
  поддерживает.
- Local resource warnings and runtime health reporting.
- Best-effort local runtime/workspace guardrails needed for trusted V01 use.

### Runtime and Agent Session

- Persistent Runtime как default runtime strategy.
- Interactive developer session как default work contract.
- Codex-backed runtime path на Node Daemon through the V01 exec/resume adapter.
- Минимальная граница Agent Provider Adapter.
- Start, resume, send turn, approve, interrupt, stop, attach and detach.
- Provider resume reference, если он доступен.
- Для V01 Codex continuity использует `codex exec` and `codex exec resume`,
  когда доступен provider session id; provider-native live process ownership,
  streaming output and real interrupt escalation остаются post-V01 work.
- Managed runtime lifetime вместо обещания бесконечного процесса.
- Normalized events for messages, lifecycle transitions, approvals, errors and
  warnings.

### Distributed Runtime Coordination

- `Nodes -> Projects/Workspaces -> Sessions` inventory view.
- Explicit user choice of node/project placement.
- Runtime session как coordination unit.
- Core-routed commands to Node Daemon.
- Ordered events per runtime/session stream.
- Stale/offline node behavior.
- Minimum resource, conflict and disconnected-state warning badges.
- Explicit user override recorded as an event where needed.

### Web Control Panel

- Application shell with:
  - left navigation tree for nodes, projects/workspaces and sessions;
  - main activity surface;
  - session header/breadcrumb with node, project, runtime and status context;
  - reserved secondary/detail area for future workspace, artifact, trace and
    configuration panels.
- Nodes surface with status, heartbeat age, basic capabilities, active sessions,
  pair/revoke flow and diagnostics.
- Project/workspace binding view with selected node placement.
- Session/chat view как первая primary activity surface.
- Message stream with user messages, assistant output, system/runtime events and
  approval cards where needed.
- Session lifecycle controls: start, attach, detach, interrupt, stop, resume.
- Stable entity ids and routes for nodes, projects/workspaces, sessions, turns,
  messages, runtime events and provider references.
- Visible non-production/trusted-deployment warning where the app exposes node
  pairing or remote node connection flows.

## Extension Readiness Requirements

V01 должна оставаться маленькой, но не закрывать путь к уже известной target
model.

Требования к реализации:

- Не моделировать session как простой chat transcript. Она должна быть связана с
  node, project/workspace, runtime, provider, lifecycle state and events.
- Оставить Core единственной client-facing coordination point. Web client не
  должен зависеть от direct access to every node.
- Оставить Node Daemon владельцем local runtime/process/workspace execution.
- Использовать typed command and event envelopes вместо ad hoc text protocols.
- Сохранить message rendering совместимым с future structured blocks, хотя V01
  поставляет только basic text/status/approval rendering.
- Держать node capability snapshots отдельно от будущих Tool Registry and Plugin
  Registry concepts.
- Оставить UI layout and routing открытыми для будущих workspace inspector,
  editor, terminal, diff/review, artifact, trace and plugin surfaces.
- Сохранить project/workspace identity достаточно стабильной, чтобы будущие
  file, terminal, diff, check and artifact references могли прикрепиться к ней
  без переделки product model.

## Не входит в V01

Эти направления не отвергаются. Они находятся за пределами первого пригодного
продуктового среза и относятся к feature queue.

- Project Workspace Inspector.
- File tree, file viewer and lightweight text editor.
- Workspace terminal/PTY or command runner.
- Command/output history beyond session/provider output needed for chat.
- Workspace diff view, check/test result view and git/review workflow.
- Production security hardening.
- Hardened node authentication, credential rotation, keychain integration, mTLS,
  request signing and full audit/security policy.
- Multi-user auth, permissions, secrets management and team security model.
- Tool Registry implementation.
- Plugin Registry implementation.
- MCP or external integration platform.
- Visual artifact system.
- Dynamic UI from agents or plugin-rendered UI blocks.
- Task-based sandbox runs.
- Durable workflow engine.
- Background run queue.
- CI webhook wakeups.
- Full MR/PR flow.
- Multi-agent orchestration.
- Managed cloud node pool.
- Complex team RBAC.
- Production support for additional providers such as OpenCode or Claude Code.
- Full provider-neutral feature parity across CLI agents.
- Full browser IDE, language server integration, broad refactoring workflows,
  debugger and VS Code extension compatibility.
- Next.js as the required app runtime.
- Tauri-specific domain logic.

## Acceptance Criteria

V01 является цельной, когда:

- пользователь может enroll at least one Node Daemon, а модель поддерживает
  несколько registered nodes;
- пользователь может bind project/workspace to a node;
- пользователь может start Core-managed Codex-backed session on chosen
  node/project with persisted state and resume continuity where provider
  support allows it;
- chat работает через Core-routed session commands and ordered runtime events;
- пользователь может detach, reattach, stop and resume, когда provider это
  поддерживает;
- UI переживает reload and reconstructs node/project/session state from Core;
- offline/stale/error states видны, а не спрятаны внутри chat text;
- V01 clearly marked as trusted/local/single-user or controlled development
  deployment, with no production-security claim;
- implementation leaves explicit extension points for workspace, tools, plugins,
  artifacts, trace and task-run surfaces without implementing those surfaces in
  V01.

## Открытые решения

- Когда Cortex должен перейти на provider-native Codex app-server/live process
  protocol для process ownership, output streaming, approvals and interrupt
  escalation?
- Должна ли V01 использовать WebSocket для UI and Node streams, или HTTP plus
  SSE/WebSocket by channel?
- Сколько provider raw event data Core должен сохранять for debugging?
- Как представлять project and workspace, если V01 еще не показывает
  file/workspace inspector?
- Какой explicit warning/deployment profile UI должен показывать до появления
  post-V01 security baseline?
