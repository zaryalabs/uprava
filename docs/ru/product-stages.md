# Uprava Product Stages

Статус: `superseded`

Этот документ сохранен для исторического контекста. Активная модель продукта
перенесена в [product-evolution.md](product-evolution.md), [v01.md](v01.md) и
[feature-queue.md](feature-queue.md).

Этот документ описывает стадии развития продукта. Главная идея: каждая стадия
должна давать качественно новый продукт, а не просто набор дополнительных фич.

## Принципы стадий

- Каждая стадия должна быть usable end-to-end.
- Каждая стадия должна проверять один главный продуктовый тезис.
- Архитектура первой стадии должна оставлять место для registry, plugins,
  visual blocks, integrations and task-based mode.
- Первый продуктовый фокус - developer workflow.
- Первый execution mode - persistent agent session на нодах.
- Task-based sandbox mode важен, но не нужен в первой версии.
- Лучше получить маленькую, но цельную рабочую систему, чем сразу строить
  полный cloud-agent runtime.

## Stage 1. Developer Node Workbench

**Качественно новый продукт:** можно использовать Uprava как web control panel
для работы с живым агентом разработки на одной или нескольких нодах.

Это не просто чат. Пользователь видит проект, агентскую сессию, терминал,
файлы, изменения, базовый trace и состояние ноды.

Техническая база Stage 1 описана в [tech-stack.md](tech-stack.md): Rust
Core/Node, React 19 + TypeScript + Vite, Tailwind CSS v4, shadcn/ui conventions,
lucide-react, TanStack Query/Table, React Hook Form + Zod and Vitest.

### Главный тезис

Persistent agent session + Node Daemon + Core Backend уже дают больше контроля,
прозрачности and mobility, чем обычный локальный agent chat.

### Scope

- Core Backend как control plane.
- Web Control Panel как первый клиент.
- Node Daemon как data plane.
- Регистрация нод в Core.
- Базовый node discovery and heartbeat.
- Project registry.
- Project/workspace binding на ноде.
- Persistent agent session mode.
- Codex как default AI agent.
- Attach/detach к agent session.
- Chat/session view.
- Terminal/output view.
- File browser.
- Basic diff view.
- Basic session trace.
- Basic event log.
- Minimal status model для session/workspace/node.
- Minimal Tool Registry shape в Core.
- Minimal Plugin Registry shape в Core.
- Minimal visual block/artifact contract.

### Не входит

- Task-based sandbox runs.
- Durable workflow engine.
- Multi-agent orchestration.
- Full plugin marketplace.
- Complex RBAC/team model.
- Managed cloud nodes.

### Почему так

Persistent mode проще и ближе к первому developer use case: запустить агента на
своей ноде, подключиться к нему, видеть окружение и управлять работой. Это
быстрее проверит базовую ценность Uprava: control plane + node daemon +
прозрачный UI.

Task-based mode требует больше инфраструктуры: sandbox lifecycle, workflow
state, queues, review contracts, artifact packaging, retries, webhook wakeups.
Его лучше добавлять после того, как базовая модель нод, сессий, файлов,
терминала, diff and trace уже работает.

### Критерий готовности

Пользователь может:

1. Поднять Core and Web Control Panel.
2. Подключить локальную или удаленную Node.
3. Открыть проект на Node.
4. Запустить persistent Codex session.
5. Смотреть чат, terminal output, файлы, diff and trace.
6. Остановить/продолжить работу и вернуться к сессии позже.

## Stage 2. Modular Developer Workbench

**Качественно новый продукт:** Uprava становится не hardcoded workbench, а
расширяемой платформой для developer workflows.

В этой стадии появляются реальные registry contracts and первые интеграции.

### Главный тезис

Модульность Uprava должна быть системной возможностью, а не будущей декларацией.

### Scope

- Tool Registry v1.
- Plugin Registry v1.
- Integration adapter model.
- First-class MCP adapter support.
- Native adapter path for integrations where MCP is not enough.
- Git provider integration.
- Linear integration as first task tracker.
- Basic plugin configuration UI.
- Basic permissions for tools.
- Tool call trace.
- Artifact metadata tied to tool/plugin output.
- Minimal visual plugin/block API.
- CLI access to registered tools.

### Product shift

Stage 1 - это workbench для живых agent sessions.

Stage 2 - это workbench, который можно расширять: tools, integrations, visual
blocks and workflows начинают подключаться через понятную модель.

## Stage 3. Visual Agent Work Surface

**Качественно новый продукт:** Uprava становится визуальной рабочей
поверхностью для agent work, а не набором панелей вокруг чата.

### Главный тезис

Главное преимущество Uprava над agent chat - не только control plane, но и
визуальное представление результата, состояния and review.

### Scope

- Rich diff/review view.
- Test/check report artifacts.
- Trace timeline.
- Terminal replay or structured command history.
- UML visualization.
- Basic UML editor or diagram artifact editing.
- Dashboard artifact support.
- Forms generated by agents/plugins.
- Embedded external views, e.g. Grafana.
- Artifact gallery per project/session.
- Visual block layout in chat/session pages.

### Product shift

Stage 2 подключает tools and plugins.

Stage 3 делает результаты этих tools полноценными UI artifacts, которые можно
смотреть, проверять and использовать без чтения длинного текста агента.

## Stage 4. Task-Based Agent Runtime

**Качественно новый продукт:** Uprava становится runtime для background agent
tasks, а не только interactive sessions.

Здесь появляется второй execution mode: task-based sandbox run.

### Главный тезис

После того как Core, Node Daemon, registry, trace and visual artifacts уже
существуют, Uprava может безопасно добавить cloud-agent-like task flow.

### Scope

- Task-based agent run mode.
- Context package.
- Sandbox/workspace lifecycle.
- Isolated branch or workspace per run.
- Explicit stop condition.
- Expected evidence contract.
- Run event log.
- Review-ready output.
- Durable workflow state.
- Basic run queue.
- Git webhook wakeups.
- CI follow-up loop.
- MR/PR flow.
- Review queue.

### Product shift

Stage 3 - пользователь работает с живыми агентами and visual artifacts.

Stage 4 - пользователь может ставить bounded tasks в фон и получать review-ready
output.

## Stage 5. Hybrid and Orchestrated Workflows

**Качественно новый продукт:** Uprava соединяет interactive and background
modes.

### Главный тезис

Сильный агентский workflow не обязан быть только live session или только task
run. Живая сессия может порождать bounded runs, а task-based workflow может
возвращаться в interactive review.

### Scope

- Hybrid managed session.
- Orchestration agent inside Uprava.
- Session spawns task runs.
- Task run results merge back into session/workflow state.
- Multi-step workflow templates.
- Semi-deterministic pipelines: implement -> self-check -> review -> fix.
- Skills/guides/guidelines library.
- Agent self-review and check steps.
- WIP limits and review debt visibility.

### Product shift

Stage 4 добавляет background tasks.

Stage 5 превращает их в управляемые agentic workflows.

## Stage 6. Multi-Node, Team and Cloud

**Качественно новый продукт:** Uprava становится distributed/team platform.

### Главный тезис

После проверки single-user and personal distributed сценариев Uprava можно
расширять до team/cloud модели.

### Scope

- Multi-user projects.
- Roles and permissions.
- Team audit trail.
- Shared review queues.
- Managed Core deployment.
- Managed or registered cloud nodes.
- Node pools.
- Multi-node scheduling.
- Organization-level plugin/integration management.
- Stronger secrets model.
- Billing/account/project model if commercial path is chosen.

### Product shift

Stage 5 - advanced personal/small-team agent OS.

Stage 6 - managed distributed Agent OS for teams.

## Stage 7. Beyond Software Development

**Качественно новый продукт:** Uprava становится general WorkOS for agents.

### Главный тезис

Developer workflow - первый vertical. Но базовая модель шире: nodes, agents,
tools, visual artifacts, trace and workflows применимы к research, analytics,
documents, finance and knowledge work.

### Scope

- Research workflows.
- Analytics workflows.
- Document and presentation workflows.
- Finance/monitoring workflows.
- Knowledge base workflows.
- Domain-specific visual artifacts.
- Domain-specific plugins and templates.

### Product shift

Stage 6 делает Uprava зрелой distributed платформой.

Stage 7 расширяет ее за пределы software development.

## Предварительная последовательность

```text
1. Developer Node Workbench
   persistent sessions, node daemon, core, web control panel

2. Modular Developer Workbench
   tool registry, plugin registry, integrations

3. Visual Agent Work Surface
   first-class artifacts, visual review, dashboards/forms/UML

4. Task-Based Agent Runtime
   sandboxed runs, durable workflow state, MR/PR flow

5. Hybrid and Orchestrated Workflows
   sessions spawn runs, pipelines, skills/guides

6. Multi-Node, Team and Cloud
   collaboration, RBAC, managed core/nodes

7. Beyond Software Development
   research, analytics, docs, finance, knowledge workflows
```

## Открытые вопросы

- Должен ли Stage 1 включать базовую git integration или достаточно diff view
  над workspace?
- Нужен ли Stage 1 мобильный UI, или достаточно responsive web для
  monitoring/attach?
- Какая минимальная форма trace нужна для persistent session?
- Должен ли Tool Registry в Stage 1 быть только internal schema, или уже иметь
  UI?
- Где провести границу между Stage 2 visual block API and Stage 3 rich visual
  artifacts?
- Когда именно появляется MR/PR flow: в Stage 2 как git integration или в Stage
  4 как task-based output?
- Нужно ли Stage 4 строить на готовом workflow engine или сначала сделать
  минимальный внутренний durable state?
